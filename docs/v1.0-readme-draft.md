## Hero / one-liner

> Mnemos is a narrative-first AI memory system for Claude Code, built on
> plain markdown you own. Your past sessions become a connected graph
> your AI can read, traverse, and brief you with — automatically, every
> time you start working.

(Earlier framing called this "Obsidian-native AI memory palace." v1.0
drops "memory palace" — the metaphor came from a paradigm we tested and
left behind. See "Why Mnemos exists" below.)

## Why Mnemos exists

Mnemos started inspired by [MemPalace](https://github.com/MemPalace/mempalace),
the 42K-star project that proved memory palaces work for AI. I built on
its foundation for four months — Obsidian-native storage, multilingual
mining, an LLM-quality pipeline that turned Claude Code conversation
history into ~600 atomic memory fragments ("drawers") classified by hall
(decisions, problems, preferences, events).

Then I measured it.

**Three things didn't survive contact with real data:**

1. **Vector recall on small fragments converges.** The Reciprocal Rank
   Fusion score distribution clusters in a narrow 0.014–0.017 band on
   ~600 drawers. The supposed precision boost from atomization doesn't
   materialize on conversational data — the embeddings of two unrelated
   decisions are barely distinguishable from the embeddings of two
   related ones once you've cut them small enough.
2. **LLM synthesis prefers narrative.** When asked to brief a project,
   the model produces tighter, more accurate answers from full session
   notes (one conversation = one rich markdown file) than from
   fragmented decision atoms it has to glue back together. The
   chronology and cause-and-effect chains live in the prose, not in the
   atoms.
3. **A 600-node graph is fancy, not navigable.** The user doesn't
   traverse it manually; the AI doesn't traverse it programmatically.
   The "palace" became a poster — beautiful, unreadable.

**Mnemos v1.0 is a different paradigm: narrative-first.**

- **Sessions stay whole.** Each Claude Code conversation becomes one
  rich markdown note in your vault, with structured sections (decisions,
  problems, next steps, summary). The session is the unit of memory.
- **Wikilinks connect Sessions through shared entities.** When two
  sessions both mention "Project X" or "Customer Y", a `[[Project X]]`
  link bridges them — producing a sparse, readable graph instead of a
  dense, atomic one.
- **Skills traverse the graph.** The briefing and recall skills don't
  just keyword-search; they follow wikilinks, read backlinks, and
  resolve entity neighborhoods to assemble context the way a human
  would scroll through related notes.

This isn't MemPalace done better. It's the path I found by testing
MemPalace's path and meeting its wall. Credit for the wall belongs to
the original project — without it, there's no map.

It also isn't unique to MemPalace. The broader **atomic-fragmentation
paradigm** (mem0, several LangChain memory backends, most "vector
database as memory" approaches) shares the same hypothesis: smaller is
sharper. Mnemos v1.0 is a counter-bet — bigger units, denser links,
smarter traversal.

## What "narrative-first" means in practice

**You write nothing.** Mnemos refines your Claude Code transcripts in
the background and produces one Session note per closed conversation,
in your existing Obsidian vault, in plain markdown.

**Each Session is a rich note**, not a transcript. It has:
- A title and date
- A `cwd:` frontmatter field (which project this session was about)
- An "Özet" / "Summary" section
- "Alınan Kararlar" / "Decisions" — what was decided
- "Sorunlar" / "Problems" — what went wrong, what's still open
- "Sonraki Adımlar" / "Next steps"
- Inline `[[wikilinks]]` to entities (projects, people, tools, files)

**Sessions form a graph.** Two sessions linking to `[[Project X]]` are
neighbors. Obsidian shows this in Local Graph View. Mnemos's skills
walk it programmatically.

**Two skills consume the graph:**

- **`mnemos-briefing`** runs automatically when you start Claude Code in
  a directory. It finds Sessions with matching `cwd:`, traverses
  wikilinks one hop out for relevant entity context, and synthesizes a
  200–400 word briefing covering active decisions, revised decisions,
  open problems, and what's likely next. Injected as additional context
  before your first turn.
- **`/mnemos-recall <query>`** is for cross-context questions: working
  in directory A but asking about work done in directory B. It does
  vector search over Sessions, follows wikilinks to gather neighborhood
  context, and synthesizes a 150–300 word narrative answer with
  clickable wikilink citations.

Both skills run inside your existing Claude Code session via the
subscription quota. **Zero per-query API cost.**

## What v1.0 keeps from earlier versions

- **Obsidian-native markdown.** Every memory is still a `.md` file you
  can read, edit, delete, and search with Obsidian's native tools.
- **MCP server.** AI clients (Claude Code, Cursor, others) connect via
  MCP and query the same files Obsidian shows you.
- **Multilingual.** Turkish + English work equally; new languages drop
  in as marker sets.
- **Two backends.** ChromaDB (default) or sqlite-vec for the vector
  index. Migration in either direction is one command.
- **Auto-refine hook.** SessionStart hook turns closed Claude Code
  transcripts into Session notes in the background.
- **`mnemos init` onboarding.** Five-phase guided setup, unchanged.

## What v1.0 removes

- **Drawer mining pipeline.** Skill-mine subprocess, processing log
  Excel audit, pilot orchestrator, accept command, ledger reconcile,
  hall classification — all gone. Mining was 50% of the project's
  complexity for ~5% of the recall quality.
- **Hall-based memory atomization.** Sessions replace the
  `Mnemos/<wing>/<hall>/<drawer>.md` hierarchy. The flat `Sessions/`
  directory plus a wikilink graph carries the weight.
- **Knowledge graph indirection.** The SQLite triple store
  (`mnemos_graph` / `mnemos_timeline` MCP tools) is deprecated.
  Wikilinks in Sessions are the new graph; backlinks are the new
  reverse query.
- **The "memory palace" framing.** No more wings, rooms, halls,
  drawers. The vocabulary was elegant but it was also a leash to a
  paradigm we no longer share.

## Migration: v0.x → v1.0

If you've been running Mnemos v0.3 or v0.4, the v1.0 upgrade migrates
you automatically:

1. **Your Sessions are preserved as-is.** No content loss.
2. **Drawers are archived, not deleted.** They move to
   `<vault>/_recycled/drawers-v0.x-archive/`. If you want to come back
   to them, they're untouched.
3. **Entity mentions are recovered.** A one-time migration script reads
   the entities your old drawers extracted and injects them into your
   Sessions as `[[wikilinks]]`. Your wikilink graph starts populated.
4. **The vector index rebuilds on Sessions only.** First boot after
   upgrade re-indexes once; subsequent boots are incremental.
5. **MCP tool surface narrows.** `mnemos_graph` and `mnemos_timeline`
   tools are still available but route through the wikilink graph
   instead of the SQLite triple store. Existing AI clients that called
   them keep working with no code change.

If you preferred the v0.x atomic model, the last v0.4.x release on PyPI
is preserved. `pip install mnemos-dev==0.4.x` pins you there.

## Notes on tone

- **Don't apologize for the v0.x architecture.** It was an honest test;
  the data answered. Apologetic framing weakens the discovery story.
- **Don't sneer at MemPalace or atomic-memory projects.** Saygılı pivot,
  not sataşma. Their work was the map; we walked it and found the wall.
- **Lead with measurement, not opinion.** "We measured X" beats "We
  felt Y" every time. The 0.014–0.017 score band is concrete; it gives
  the reader something to weigh.
- **Be explicit about what we keep.** Readers fear a v1.0 will throw
  away what they liked. The "What v1.0 keeps" section is load-bearing.
