**Status:** Design — implementation pending
**Date:** 2026-04-24
**Owner:** Tugra / Claude Code
**ROADMAP reference:** v0.4.0 Phase 1 — skill-first, task 4.3.1 (sub-ship 2/2 of task 4.3)

## 1. Problem

The 4.3 first ship (2026-04-23) put **cwd-aware automatic briefing** live: as
soon as the user opens a Claude Code session in a folder, a narrative
synthesis of past sessions for that cwd is injected via `additionalContext`.
This is enough for **90% of the usage** of the "where are we in this
project" question.

The remaining 10%: **cross-context query**. The user is working in one cwd
but wants to recall something from a different context. Canonical example
(2026-04-24 brainstorm):

> The user is working under `C:\Projeler\teklif-mukayese` on the comparison
> skill. Something occurs to them: "when we ran the po skill, what format
> did we use?" — that information is in drawers produced in a different
> wing (GYP), under a different cwd.

Existing mechanisms try to close this gap but introduce friction:

1. **MCP `mnemos_search` (AI auto-call, `recall_mode: script`)** — Claude
   calls it automatically, but under `recall_mode: skill` (briefing-first)
   the MCP instructions say _"don't auto-call"_. The user has to ask for the
   MCP tool call explicitly ("search in mnemos").
2. **Current briefing (4.3)** — cwd-scoped only. Cannot answer a cross-wing
   query.
3. **Obsidian manual search** — open the vault by hand + search bar, leave
   the session; Claude can't synthesize the answer.

The user request is clear: let me type a `/mnemos-recall` slash command and
have Claude find drawers relevant to my question and **tell me what it
remembers** — without opening a separate session, without breaking my
current workflow.

## 2. Goal

When the user types `/mnemos-recall <free-text-query>` in the current
session:

1. The skill loads into the current Claude (no subprocess).
2. Current Claude calls the `mnemos_search` MCP tool with the query.
3. Reads the full markdown body of the top 5 drawers via `Read`.
4. Produces a 150-300 word **narrative answer**, with a
   `[[drawer-slug]]` wikilink for every claim.
5. On weak/empty match, instead of fabricating, gives a soft fallback in the
   "no exact record, the closest matches are these" format.

The whole flow is in-session, typical duration 3-5s, no new Python module.

## 3. Non-goals

- **Subprocess orchestration** — the briefing skill spawns a subprocess
  because it's called by a hook. Explicit recall is user-interactive → runs
  in the current session. Fork-bomb guard, env scrub, and cache layer are
  none of them necessary.
- **Query parsing / filter flags** — no `--wing X` arguments. Natural-
  language semantic search is sufficient; RRF + embedding score handle
  poorly-phrased queries reasonably well (kasamd R@5=90%).
- **Multi-turn agentic retrieval** — the skill is a single pass
  (search → read → answer). On a weak match, there's no "search again,
  broaden, narrow" loop run by Claude (Approach 3 rejected, YAGNI).
- **Cwd filter or bias** — explicit recall is _by definition_ cross-context.
  Applying boost/filter for the current cwd defeats the purpose.
- **Cache** — queries are unique; cache hit rate is low. Every call is fresh.
- **Rerank / LLM-judge filter** — mnemos_search's RRF+embedding score is
  accepted as the baseline. Phase 1 rerank is deferred to v0.5 hygiene.
- **Junction automation** — having `mnemos init` automatically lay down
  skill junctions is shared work for all skills, not in scope for 4.3.1.

→ Skill loads (current session context)

→ Top 3 drawer .md files read in full

```

→ mnemos_search(...) top score 0.31 (weak)

```

### Scenario 3 — No arguments
```

→ Skill:

```

```

→ Skill detects that the mnemos_search tool does not exist:
"Mnemos MCP is not connected. You can set it up with `mnemos init`."
```

## 5. Architecture

**No subprocess, no new Python code, no CLI command.**

```
User types /mnemos-recall <query>
    ↓
Claude Code Skill tool loads skills/mnemos-recall/SKILL.md
    into current session context
    ↓
Current Claude (already connected to mnemos MCP) executes SKILL.md:
    mnemos_search(query, limit=8, collection="both")
        ↓
    (backend: sqlite-vec or ChromaDB; vector similarity; ~10-50ms)
        ↓
    Read top 5 drawer .md files (full body)
        ↓
    Synthesize narrative answer with [[wikilinks]]
    ↓
Output inline in current session
```

**Interaction with recall_mode:** none. `recall_mode: script` does AI
auto-call by default; `recall_mode: skill` is briefing-first; in both modes
the **user's explicit `/mnemos-recall` invocation is an override** — the
skill always runs.

**Parallels with the briefing skill:**

| Aspect | `mnemos-briefing` (4.3) | `mnemos-recall` (4.3.1) |
|---|---|---|
| Trigger | SessionStart hook (automatic) | User slash command (explicit) |
| Execution | Subprocess (`claude --print`) | Inline in current session |
| Scope | Cwd filter | Query-based semantic |
| Output | Briefing cache file + stdout | Current session output |
| Caching | `.mnemos-briefings/<slug>.md` | None |
| Token budget | ≤60K input | ≤20K input (5 drawers) |

## 6. Components

### 6.1 `skills/mnemos-recall/SKILL.md`

A single file. Frontmatter + 6 numbered steps + edge cases.

**Frontmatter (for Claude Code slash command discovery):**
```yaml
---
name: mnemos-recall
description: Cross-context recall from the Mnemos memory palace.
  User invokes /mnemos-recall <query> to ask a free-text question;
  skill searches the palace via MCP, reads top drawers, and synthesizes
  a narrative answer with wikilink citations. In-session only
  (no subprocess). Use when cwd-based auto-briefing doesn't cover
  the query (e.g. working in cwd A but asking about work done in cwd B).
---
```

**Body content (as prose):**
1. User query = argument (all text after `/mnemos-recall `).
2. Call `mnemos_search(query=<arg>, limit=8, collection="both")`. No
   wing/room/hall/cwd filter.
3. Inspect top results. If top score < 0.5 OR no results → jump to step 6.
4. Iterate candidates in score order (highest first). For each, `Read`
   the file at `source_path` frontmatter. If `Read` fails (file missing /
   renamed), skip and move to the next candidate. Stop after **5
   successfully read drawers** or the candidate pool is exhausted. Thus
   the 8-result search gives a 3-drawer buffer for broken links.
5. Synthesize 150-300 word narrative answering the user's question:
   - Every claim cites a drawer as `[[drawer-slug]]` (slug = filename
     without `.md` extension).
   - Language matches query language (TR query → TR answer).
     The drawer body language may differ (TR question + EN drawer
     contents) — during synthesis, Claude summarizes the contents into
     the question's language; the wikilink slug does not change.
   - Prose paragraphs, no headers unless answer spans multiple halls.
   - No hedging ("I think...", "I hope..."). If data is thin, say so.
6. Soft fallback (weak/no match): emit "I couldn't find a clear record of
   this" + top 3 drawers with score + 1-line snippet, suggest they may be
   related but the skill won't synthesize.
7. Edge handlers:
   - No args → usage hint + example
   - MCP not connected → "Mnemos MCP not connected, set it up with `mnemos init`"
   - Empty palace → "Palace is empty, start with `mnemos mine`"
   - Broken drawer links → skip that drawer, continue with the rest

Size: ~120 lines of markdown.

### 6.2 Junction

Windows:
```
mklink /J "C:\Users\<user>\.claude\skills\mnemos-recall" "C:\Projeler\mnemos\skills\mnemos-recall"
```

Unix:
```
ln -s "$REPO/skills/mnemos-recall" "$HOME/.claude/skills/mnemos-recall"
```

Zero-drift: repo is canonical, the user's `~/.claude/skills/` side is the
shortcut. Manual install (same pattern as the briefing skill) — automatic
install is deferred to v0.6 community polish.

## 7. Data flow

```
[User types /mnemos-recall "query"]
         │
         ▼
[Claude Code Skill tool loads SKILL.md]
         │
         ▼
[Current Claude parses args + checks mnemos MCP connected]
         │
         ├─ not connected → "mnemos init" hint → exit
         ├─ no args       → usage hint → exit
         │
         ▼
[mnemos_search(query, limit=8, collection="both")]
         │
         ▼
[Backend: sqlite-vec or ChromaDB vector search]
         │
         │   Returns: [{drawer_id, score, wing, room, hall,
         │              source_path, snippet}, ...]
         ▼
[Inspect top 3 scores]
         │
         ├─ all < 0.5 → Soft fallback (list top 3 with snippets) → exit
         │
         ▼
[For top 5 drawers: Read <source_path>]
         │
         │   ~5-15 KB total on disk
         ▼
[Synthesize narrative with [[wikilinks]]]
         │
         ▼
[Output inline in current session]
```

## 8. Error handling

| Situation | Behavior |
|---|---|
| `mnemos_search` tool not present | "Mnemos MCP not connected" message |
| `mnemos_search` empty array | "Palace is empty, start with mnemos mine" |
| Top score ≥ 0.5, single drawer | Synthesize from one drawer, note "no additional records" |
| Top 3 < 0.5 | Soft fallback (list + snippet) |
| `Read` error (drawer file missing) | Skip that drawer, continue with the rest |
| All read drawers broken | "Index-filesystem mismatch, `mnemos mine --rebuild`" |
| Invocation without arguments | Usage hint + example |
| Very long query (>500 tokens) | Embedding clips it; no special handling |

## 9. Testing

Since the skill is just a `.md` file, there's no code that traditional
pytest assertions can run against. Verification is in two layers:

### 9.1 Manual smoke (after every change)

Current Claude Code session in the author's kasamd vault:

1. **Happy path:** `/mnemos-recall "what format did the po skill run in"`
   → Narrative answer, with wikilinks, 150-300 words.
2. **Weak match:** `/mnemos-recall "xyzabc query that never exists"`
   → Soft fallback list.
3. **No args:** `/mnemos-recall`
   → Usage hint.
4. **Cross-language (1):** `/mnemos-recall "what backend do we use"`
   → English answer (query EN).
5. **Cross-language (2):** `/mnemos-recall "how was the pytest setup done"`
   (TR-style query, but drawer bodies may be EN) → TR answer; drawer
   contents are summarized into TR during synthesis.
6. **Broken-drawer simulation:** temporarily rename a drawer; the skill
   should skip that drawer and answer from the rest.
7. **Backend parity:** also run scenarios 1 and 2 against the sqlite-vec
   backend (if necessary, plan via `mnemos migrate --backend sqlite-vec
   --dry-run`, then revert). The 0.5 threshold should produce the "weak
   match" label for the same drawers under both backends. If there's
   divergence, calibrate the threshold in SKILL.md (e.g. 0.4 for
   sqlite-vec, 0.5 for ChromaDB) or leave a backend-aware helper. This
   calibration does not require a spec revision (§11).

### 9.2 Automated unit tests (`tests/test_mnemos_recall_skill.py`)

Two tests:

- `test_mnemos_recall_skill_frontmatter_valid` —
  `skills/mnemos-recall/SKILL.md` exists, the YAML frontmatter parses, and
  the `name` + `description` fields are non-empty. Critical for Claude Code
  slash command discovery.
- `test_mnemos_recall_skill_junction_zero_drift` — the repo's SKILL.md and
  `~/.claude/skills/mnemos-recall/SKILL.md` have the same mtime/size
  (junction intact). If a similar pattern already exists for
  `skills/mnemos-briefing`, share it (factor out into a helper fn).

Test count: 641 → 643. No new Python code → coverage unchanged.

### 9.3 (Optional) Prompt regression

If desired later: an E2E test under `pytest.mark.slow` that calls
`claude --print "/mnemos-recall 'known-good query'"` and asserts the
output contains a wikilink. YAGNI for 4.3.1; can be revisited in v0.4.1
polish.

## 11. Open questions

None. All open ends were closed during the brainstorm:

- ✅ Output: narrative synthesis with wikilinks
- ✅ Execution: in-session (no subprocess)
- ✅ Scope: full-vault search, no cwd bias, no filter flags
- ✅ Threshold: 0.5 RRF score (can be calibrated against kasamd during
  implementation; write the constant into SKILL.md — empirical tuning
  doesn't require a spec revision). There may be backend-specific
  divergence (sqlite-vec brute-force cosine vs ChromaDB HNSW: same
  embedding model but the score distribution may differ) — verified in
  §9.1 smoke 7; if needed, write a single fixed value into SKILL.md
  (not a yaml-driven backend-aware threshold), choosing during
  calibration the value with the smallest divergence.
- ✅ Fallback: soft (list top 3 + snippets)
- ✅ Testing: 2 unit tests + manual smoke
- ✅ `recall_mode` interaction: none, explicit always overrides
