# v0.4 Task 4.3 — First Ship (Path-Checker + Auto-Briefing + MCP `recall_mode`) — Design

**Status:** Design — implementation pending
**Date:** 2026-04-23
**Owner:** Tugra / Claude Code
**ROADMAP reference:** v0.4.0 Phase 1 — skill-first, task 4.3 (sub-ship 1/2)

---

## 1. Problem

v0.4.2-alpha shipped skill-mine to production + set up the 4.3.A hook route.
There are now 599 skill-mined drawers under `Mnemos/` in the vault. But on
the **recall side** there are currently two mechanisms, both inadequate:

1. **MCP `mnemos_search` (AI auto-call)** — Claude calls it when needed,
   but it depends on keyword match and the user has to explicitly mention
   the topic every time. It's "searching" the context, not "remembering"
   it.
2. **MCP `mnemos_wake_up` (SessionStart instruction)** — only loads identity
   + L1 wing summaries. It is not aware of **which cwd the user is working
   in or what was discussed there**.

The user expressed this gap clearly in the 2026-04-23 brainstorm: *"I had
a session in the farcry folder; 3 days later when I open it again, our
mnemos should remember the previous session at this path on its own,
without us having to say anything"*. The recall skill was pushed to a
cross-context edge case — the real job is **cwd-aware automatic briefing**.

## 2. Goal

As soon as a Claude Code session opens, `<vault>/.mnemos-briefings/<cwd-slug>.md`
or a fresh briefing that needs to be generated should be injected as
`additionalContext`. The user will not type any command; from the very
first turn Claude will arrive with a **narrative synthesis** of past
sessions in that cwd — not just "the last thing we talked about" but
evolution-aware at the level of "first you said X, then you turned to Y,
right now Y is active".

Additionally: the MCP server `instructions` field should switch to
skill-driven behavior under `recall_mode: skill` yaml — the AI should not
auto-call `mnemos_search`; the briefing is enough.

## 3. Non-goals (out of scope for this ship)

- **`/mnemos-recall <query>` explicit user skill** — cross-context edge
  case, separate ship (4.3.1). Get the automatic briefing right first.
- **Vault-wide contradiction detection scanner** — deferred to v0.5. In
  this ship contradiction detection only happens within the briefing
  skill's prompt-level synthesis (cwd-scoped); no separate tool.
- **Briefing UI (Obsidian plugin, timeline view)** — v0.6.
- **Briefing for non-Claude-Code MCP clients** — the briefing hook is
  specific to Claude Code SessionStart; no generalization for now.
- **Briefing token cap optimization** — start with the assumption of ≤10
  sessions per cwd; if recursive summary-of-summaries is needed, it's a
  Phase 2 polish item.

## 4. User experience (ELI5)

### Scenario 1 — First use of a cwd
```
User opens Claude under C:\...\farcry for the first time
  ↓
Path checker: "this cwd is being seen for the first time"
  ↓
Record in state.json, spawn bg briefing (but no memory yet, will produce a placeholder)
  ↓
Session starts immediately, no waiting
  ↓
Claude starts without any previous farcry context (there is none anyway)
```

### Scenario 2 — Return to the same cwd (memory in pocket)
```
User worked in farcry 2 weeks ago, opens it again
  ↓
Path checker: "farcry return-visit, previous sessions have been refined+mined"
  ↓
Fast path: briefing cache file exists → inject as additionalContext
  ↓
Bg regen spawn (keep cache up to date — new drawer polishes may have
arrived in 2 weeks), session is immediate
  ↓
Claude: "In farcry we last decided on X, we left question Y open"
```

### Scenario 3 — Back-to-back opens (we talked yesterday, not yet processed)
```
1st session in farcry ended, 2nd session opened 5 min later
  ↓
Path checker: "farcry 2nd time, the 1st session's JSONL has not been refined yet"
  ↓
BLOCKING sequence (hook timeout 600s):
  1. refine(jsonl-1) — /mnemos-refine-transcripts sync
  2. mine(session-1.md) — /mnemos-mine-llm sync
  3. brief(cwd=farcry) — /mnemos-briefing sync (sees session 1 + synthesis)
  4. inject briefing as additionalContext
  ↓
~70-90s wait; when Claude arrives it knows session 1
```

### Scenario 4 — 3rd session, conflicting decisions
```
Session 1: decision "let's do X"
Session 2: "we abandoned X, went back to Y"
Session 3 opens
  ↓
Path checker: if Session 2 is pending, BLOCKING refine+mine(2) (like Scenario 3)
  ↓
Briefing skill reads Session 1 + 2 together via the cwd=farcry filter
  ↓
LLM narrative synthesis:
  **Active state:** Working on Y.
  **Current decisions:** Y, 2026-04-20.
  **Revised/cancelled:** X (2026-04-15) → Y (2026-04-20) — rationale: ...
  **Open issues:** ...
  ↓
Session 3 starts with this context; Claude knows "we said X earlier but
abandoned it" — the AI does not relive the contradiction.
```

## 6. Path Checker — core logic

### 6.1 State file

**Cwd slug normalization** — exactly the same as the Claude Code project
dir convention: `re.sub(r'[^\w-]', '-', cwd)` + repeated-dash collapse.
Note: `\w` covers underscore, letters, and digits; hyphens are preserved.
Different paths produce different slugs:

- `C:\Projects\my-app` → `C--Projects-my-app`
- `C:\Projects\my_app` → `C--Projects-my_app` (underscore preserved)
- `C:\Projects\my-app\` (trailing slash) → `C--Projects-my-app-` (trailing dash)

To normalize trailing slash/whitespace, the wrapper first runs `cwd =
cwd.rstrip('/\\').strip()`. The collision risk is largely closed off by
`\w` preservation; regression tests (§13) verify the edge cases.

`<vault>/.mnemos-cwd-state.json`:

```json
{
  "version": 1,
  "cwds": {
    "C--Users-tugrademirors-OneDrive-Masaustu-farcry": {
      "cwd": "C:\\Users\\tugrademirors\\OneDrive\\Masaüstü\\farcry",
      "first_seen": 1745500800.0,
      "last_seen": 1745587200.0,
      "visit_count": 3,
      "last_session_id": "abc-123",
      "last_briefing_at": 1745587200.0
    }
  }
}
```

- **Key:** the slug of the cwd (same as the Claude Code project dir naming
  convention — `re.sub(r'[^\w-]', '-', cwd)`; collapse repeated `-`).
- **Corruption:** JSON parse fail → file is deleted, state starts from
  scratch. Visit count loss is acceptable (it's only used for briefing
  pathing and observation metrics).
- **Lifecycle:** never deleted in the normal flow. The user can clear it
  via `mnemos recall-reset` (polish item, out of scope for this ship).
- **Atomicity:** write-then-rename pattern (`<vault>/.mnemos-cwd-state.json.tmp`
  → `os.replace`). Filelock for concurrent sessions
  (`.mnemos-cwd-state.json.flock`).

### 6.2 Decision tree

```
SessionStart → recall_briefing.py wrapper runs (Claude Code hook input on stdin)
  │
  ├─ 1. Input parse: cwd, session_id, transcript_path
  │
  ├─ 2. Skip filter (same as refine-hook):
  │      - source ∈ {"startup", "resume", "clear", ""} → continue
  │      - source == "compact" → exit 0 (no briefing during compaction)
  │      - transcript_path /subagents/ → exit 0
  │
  ├─ 3. mine_mode + recall_mode yaml check:
  │      - recall_mode != "skill" → exit 0 (default script mode, hook idle)
  │      - mine_mode != "skill" → exit 0 (briefing depends on skill-mined
  │        drawers; regex palace incompatible, exit with stranded log)
  │
  ├─ 4. Load state.json (missing or corrupt → empty dict)
  │
  ├─ 5. Produce cwd slug, load state[slug] (if any)
  │
  ├─ 6. DECISION:
  │
  │   CASE A — Cwd never seen (no state[slug]):
  │     → record into state (first_seen=now, visit_count=1)
  │     → exit 0 (no briefing spawn — no refined sessions yet, generating
  │        an empty briefing is a wasted LLM call)
  │
  │   CASE B — Cwd seen before (visit_count >= 1):
  │     → look for unrefined JSONL: ~/.claude/projects/<cwd-slug>/*.jsonl
  │       (no OK in refine ledger AND not alive per PID marker)
  │
  │     SUB-B1 — No pending:
  │       → does the cache file exist (<vault>/.mnemos-briefings/<slug>.md)?
  │           NO  → no injection; bg spawn briefing (first generation)
  │           YES → staleness check (session-count threshold):
  │             - read cache frontmatter `session_count_used: N`
  │             - compute current cwd refined session count M
  │               (Sessions/*.md where cwd frontmatter == current match)
  │             - M - N >= STALE_THRESHOLD (default 3)?
  │                YES → SYNC regen: claude --print "/mnemos-briefing <cwd>"
  │                      (no refine/mine — all in pocket — briefing only);
  │                      inject the new cache
  │                NO  → inject body (strip frontmatter); spawn bg regen
  │       → state update (last_seen, visit_count++)
  │       → exit 0
  │
  │     SUB-B2 — Pending exists (1+ JSONL):
  │       → filelock .mnemos-catch-up.flock (race with other session)
  │       → BLOCKING sequence (hook timeout = 600s):
  │         for jsonl in pending:
  │           result = sync_refine(jsonl)  # claude --print /mnemos-refine
  │           if result.ok:
  │             sync_mine(result.session_md)  # claude --print /mnemos-mine-llm
  │         sync_brief(cwd)  # claude --print /mnemos-briefing
  │         # briefing skill reads ALL refined sessions via cwd filter
  │         # (we just refined the pending ones — now "in pocket")
  │       → read briefing cache, inject as additionalContext
  │       → state update
  │       → exit 0
  │
  ├─ 7. Exception → log, exit 0 (don't fail the hook; Claude Code silently
  │     starts Claude, user does not lose the session)
```

### 6.3 Coordination with the auto-refine hook

There is currently a `mnemos/auto_refine.py` SessionStart hook (calling
the skill-pipeline via the `4.3.A` route). Recall-briefing will be added
as a new SessionStart entry. Coordination of the two hooks:

**Strategy:** Recall-briefing runs **first** (entry order matters in
settings.json). Recall-briefing does the sync work (pending of the same
cwd); after it writes OK to the refine ledger for those JSONLs, the
auto-refine hook runs afterwards — its picker will skip the JSONLs
recall-briefing already processed (OK is in the ledger).

Alternative: merge into a single hook (`mnemos.recall_briefing` calls
auto_refine, then does its own briefing work). It complicates things — for
now two separate entries with sequential ordering.

**Filelock sharing:** Recall-briefing holds `.catch-up.flock`; auto-refine
holds `.auto-refine.flock`. There is no collision on the same JSONL
because in sync mode recall releases after writing to the ledger; when
auto-refine takes its turn and sees the ledger, it skips.

### 6.4 Statusline progress during blocking sequence

The blocking catch-up (SUB-B2) can take 70-90s; the user should not think
the terminal has frozen. The existing auto-refine statusline infrastructure
(`.mnemos-hook-status.json` + `install-statusline` snippet) will be extended:

New fields in **`<vault>/.mnemos-hook-status.json`**:

```json
{
  "phase": "refining" | "mining" | "briefing" | "idle",
  "current": 1,
  "total": 1,
  "cwd_slug": "C--...-farcry",
  "sub_phase": "catch-up" | "bg-regen" | null,
  "started_at": 1745587200.0,
  "last_outcome": "ok",
  "last_finished_at": 1745587270.0
}
```

During the blocking sequence the wrapper writes status at each phase
entry:
1. `refining 1/1 (catch-up for farcry)` — when sync refine starts
2. `mining 1/1 (catch-up for farcry)` — when skill-mine starts
3. `briefing (catch-up for farcry)` — when the briefing skill is invoked
4. `idle` + `last_outcome: ok` — after injection

The statusline snippet (existing bash/cmd script) reads these values and
prints them at the bottom of the terminal:

```
mnemos: preparing briefing for farcry — mining 1/1 (45s)
```

For SUB-B1 stale-regen, `sub_phase: "stale-regen"` is used; shorter
(briefing only).

If the statusline isn't installed (user did not run `install-statusline`),
status writing continues (the file is updated silently) but the user
won't see it. Not bad UX — the blocking is 60-90s anyway and the user
waits up to Ctrl+C.

### 6.5 Hook timeout

`mnemos install-recall-hook` writes `"timeout": 600000` (10 min) into
this hook entry. The default 60s is not enough — even a short prior
session takes ~70-90s for refine+mine+brief. 10 min upper bound:

- Short session (~30 turns): ~70s
- Medium (~100 turns): ~150s
- Long (~3h marathon): ~4-5 min
- Pathological (thousands of turns): safety net, user can Ctrl+C

If the timeout is exceeded, the Claude Code hook fails but the session
still starts; no briefing arrives. `last_outcome=timeout` is written to
state, and the next session is a fresh attempt.

## 7. Briefing Skill — narrative synthesis

### 7.1 SKILL.md (summary)

```markdown
---
name: mnemos-briefing
description: Cwd-aware project memory briefing. Invoked by SessionStart
  hook (bg or sync depending on cwd state). Reads refined Sessions/.md with
  cwd frontmatter match, synthesizes evolution-aware narrative.
---
```

### 7.2 Input format

The cwd is passed to the skill subprocess as an argument (`/mnemos-briefing
"C:\\...\\farcry"`). The skill reads the vault in these steps:

1. Scan `<vault>/Sessions/*.md` → filter by frontmatter `cwd == input_cwd`
2. Sort chronologically (by frontmatter `date`)
3. Extract each session's "Summary", "Decisions Made", "Problems", "Next
   Steps" sections
4. For each session, gather the related drawers:
   `<vault>/Mnemos/*/drawers/*.md` WHERE frontmatter `source == session.md`
5. Sort drawers by hall priority: decisions > problems > events >
   preferences > emotional
6. Token budget: total ≤60K input tokens. If there are too many sessions,
   take the last 10 + first 2.

### 7.3 Prompt instructions (summary)

```
You are a project historian. You are given all past sessions for a cwd
in chronological order. Your task:

1. Track decisions with their evolution: if an old decision was revised
   or cancelled, call this out separately under "revised".
2. Collect open issues (unresolved problems, TODOs).
3. Derive the active state as of today.
4. Write a 200-400 word briefing in the following structure:

**Active state:** <1-2 sentences, what is being focused on>
**Current decisions:** <chronological order>
**Revised/cancelled decisions:** <old → new, date, rationale>
**Open issues:** <unresolved>
**Up next:** <estimate for the next session>

Point to drawers via wikilink (`[[drawer-slug]]`). When you see a
contradiction, call it out clearly in the "revised" section. For an empty
cwd (no sessions):

> No prior sessions recorded for this cwd yet. Mnemos will brief from the
> next session onwards.

Output: briefing markdown body to stdout. No frontmatter.
```

### 7.4 Output

The briefing is written to stdout → wrapper captures → writes to
`<vault>/.mnemos-briefings/<cwd-slug>.md`:

```markdown
---
cwd: C:\Users\tugrademirors\OneDrive\Masaüstü\farcry
generated_at: 2026-04-23T14:32:00
session_count_used: 3
drawer_count_used: 24
---

**Active state:** ...
**Current decisions:** ...
...
```

The wrapper reads this file as `additionalContext`, strips the frontmatter,
and injects the body. The user can also open it in the vault and read it
(in Obsidian the `.mnemos-briefings/` folder shows up because there is no
`_` prefix — optional polish: hide as `_mnemos-briefings/`).

**Decision:** `.mnemos-briefings/` (no underscore prefix, hidden on the
Obsidian side via the dot-prefix) — appears normal in Windows Explorer,
hidden in Obsidian (`.` prefix).

## 8. MCP `recall_mode` Instructions

### 8.1 `mnemos/config.py`

New yaml fields:

```yaml
recall_mode: script        # default; script | skill
briefing_projects: []      # optional filter list; empty = all cwds automatic
```

`Config.recall_mode` property, default `"script"` (backwards-compatible —
existing users' behavior does not change).

### 8.2 `mnemos/server.py` — `build_instructions`

```python
def build_instructions(cfg: Config) -> str:
    base = (
        "Obsidian-native AI memory palace.\n"
    )
    if cfg.recall_mode == "skill":
        return base + (
            "SessionStart briefing is already injected as `additionalContext` "
            "for the current cwd — rely on it as primary project memory. Do "
            "NOT auto-call `mnemos_search` on every user turn; it wastes "
            "tokens when briefing already covers the context. However, IF the "
            "user explicitly asks to recall something not in the briefing "
            "(cross-project reference, older history, keyword lookup), CALL "
            "`mnemos_search` — user-explicit requests override the default-off "
            "rule. `/mnemos-recall <query>` skill (when available) is the "
            "preferred entry point for cross-context queries."
        )
    # default script mode
    return base + (
        "At the START of every session, call `mnemos_wake_up` to load identity "
        "and project context (~200 tokens). When the user mentions a project "
        "or topic, call `mnemos_search` to retrieve relevant memories before "
        "responding. Use `mnemos_recall` with level=L2 and a wing name to get "
        "deeper room-level details when needed."
    )
```

At MCP server startup `cfg = load_config(vault)` → `instructions =
build_instructions(cfg)` → pass to the FastMCP constructor.

### 8.3 Behavior impact

- `recall_mode: skill` (briefing hook active scenario):
  - Main Claude does not auto-call `mnemos_search`
  - The briefing is already in `additionalContext` → Claude arrives with
    cwd context from the first turn
  - Unless the user says explicit `/mnemos-recall`, no cross-context lookup
- `recall_mode: script` (default):
  - Current behavior — AI auto-calls `mnemos_search` when relevant
  - Briefing hook not running (unless installed)

## 9. Backfill + Refine Prompt Update

### 9.1 Refine prompt template (prompt.md + SKILL.md mirror)

The frontmatter template in `docs/prompts/refine-transcripts.md`:

**Before:**
```markdown
---
date: YYYY-MM-DD
project: <Wing>
tags: [session-log, ...]
duration: ~Xm
---
```

**After:**
```markdown
---
date: YYYY-MM-DD
project: <Wing>
cwd: <absolute path>       # ← new, from the "cwd" field in the JSONL's first message
tags: [session-log, ...]
duration: ~Xm
---
```

Instruction to add to the skill:

> The `cwd` field is read from the `"cwd"` field of the first
> non-metadata message in the JSONL (Claude Code records it every turn,
> the first turn is enough). Write it in absolute Windows path format
> (e.g. `C:\Users\...\farcry`). Don't use forward slashes. If the JSONL
> has no cwd (rare), omit the field entirely.

`skills/mnemos-refine-transcripts/SKILL.md` is junction'd, same repo —
the prompt.md change is reflected automatically. SKILL.md also contains
a summary reference of the same instruction; update that too.

### 9.2 Backfill script

`scripts/backfill_cwd_frontmatter.py`:

**Input:**
- `--vault <path>` (default: from yaml)
- `--ledger <path>` (default: `~/.claude/skills/mnemos-refine-transcripts/state/processed.tsv`)
- `--dry-run` (show what it would do, no writing)
- `--apply` (write)

**Logic:**
1. Read ledger lines: `<ts>\tOK\t<jsonl>\t<session_md>`
2. For each OK line:
   - Open the JSONL, match the first `"cwd":"..."` pattern (grep-like)
   - Open session_md, parse YAML frontmatter
   - If `cwd` is already present, skip (idempotent)
   - Add: `cwd: <absolute path>` — under the project field
   - Write back
3. Summary: N sessions updated, M skipped (cwd already present), K errors.

**Edge cases:**
- JSONL deleted → skip, log "jsonl missing"
- session_md deleted → skip
- No cwd in JSONL (old Claude Code version) → skip
- session_md yaml frontmatter corrupt → skip + log
- Duplicate ledger line → idempotent behavior

**Output report** (stdout, at the end):

```
Backfill complete:
  Updated:      N sessions (cwd added)
  Skipped:      K sessions (cwd already present)
  JSONL missing: L sessions (cannot derive cwd; briefing will ignore these)
  Errors:       M sessions (frontmatter parse / IO failure; see log)
  Total:        (N+K+L+M) sessions
```

When L > 0, the user sees which sessions remain without cwd (silently
skipped in the briefing filter). L should typically be small (archived
JSONLs or older sessions written by hand).

**Test:** `tests/test_backfill_cwd.py` — 3-session fixture (2 valid, 1
broken), apply + dry-run parity, idempotent re-run, the report producing
the right counts.

### 11.2 `mnemos install-recall-hook`

New subcommand parallel to `install-hook`. Adds to settings.json:

```json
{
  "hooks": {
    "SessionStart": [
      { ...existing auto-refine entry... },
      {
        "type": "command",
        "command": "python -m mnemos.recall_briefing",
        "timeout": 600000
      }
    ]
  }
}
```

- **Idempotent:** if entry already exists, no-op (command match)
- `--uninstall`: removes only the recall-briefing entry (auto-refine
  untouched)
- Pre-check: `mine_mode: skill` is required (in yaml); otherwise warning
  "enable skill-mine first"
- Pre-check: the auto-refine hook must be installed (recall hook on its
  own makes no sense, because without refine the cwd filter cannot find
  any sessions)
- Backup: `.bak-YYYY-MM-DD` per the existing pattern

### 11.3 `mnemos init` integration

In the `mnemos init` flow, after "install auto-refine hook?" a new prompt:
**"Install recall-briefing hook? (Y/n, default Y)"** — i18n TR + EN.

`recall_mode` defaults to `script` in yaml. If the user installs the
briefing, it switches to `skill` mode automatically.

## 12. Error handling matrix

| Situation | Behavior |
|---|---|
| state.json corrupt | Back up, delete, start over. Log `state reset`. |
| No cwd in JSONL | Path checker continues for the current session (cwd is still in the hook input); only backfill skips that session |
| No cwd frontmatter on Sessions/ session.md | Won't appear in the briefing skill filter; will be added on the next refine |
| Pending JSONL refine fail | Log, don't proceed to mine, briefing is generated without that session |
| Pending JSONL mine fail | Log, briefing uses the refined body but no drawers |
| Briefing LLM fail | Keep the previous cache (if any); otherwise no `additionalContext` |
| Hook timeout (600s exceeded) | Claude Code hook fails, session still starts, no briefing. State `last_outcome=timeout`. |
| recall_mode=skill but no install-hook | `install-recall-hook` issues a warning, recommends manual install |
| Concurrent sessions (multi-window) | filelock: the next session waits with a 10s timeout; if exceeded it falls back to bg on its own turn |
