# v0.4 Task 4.3.A — Skill-Mine Route in the Hook + Catch-up — Design

**Status:** Design — implementation pending
**Date:** 2026-04-22
**Owner:** Tugra / Claude Code
**ROADMAP reference:** v0.4.0 Phase 1 — skill-first (PRE-4.3 mini-task)

---

## 1. Problem

Skill-mine went live with v0.4.2-alpha (pilot orchestrator + accept). But
the auto-refine hook currently only triggers **regex mine**
(`mnemos/auto_refine.py:538`). The 2026-04-21 commit `a5042a2` added an
emergency guard that skips regex mine for `mine_mode: skill` — meaning the
hook today "refines incoming sessions but does not produce drawers". The
permanent fix: have the hook call **skill-mine**. After 4.3.A the regex-mine
path is fully idle (it stays live only for new vaults whose default is
`mine_mode: script`, and that flow is already legacy).

There is also a side issue: in the 2026-04-21 cleanup (cleaning up 535 hybrid
drawers), when two post-accept sessions (v042-alpha-skill-mine-accept +
gyp-deadline-anchor) were processed via the pilot orchestrator, **it started
re-mining all 122 sessions from scratch**. Diagnosis: pilot ledger path
normalization did not match. The orchestrator is in the "pilot palace +
accept" paradigm. It was not designed for atomic single-session mine on the
production path.

## 2. Goal

In `mine_mode: skill` vaults, after the auto-refine hook refines new JSONL
transcripts, it should **directly invoke the skill-mine-llm skill** and write
drawers to the **production `Mnemos/`** (pilot palace bypass, accept bypass).
With a two-phase pick/process and a persistent audit trail, visibility is
complete.

Additionally: a manual `mnemos catch-up [--limit N] [--parallel N]` command
for users — alongside the hook, this gives bulk processing for fresh-install
or large-backlog scenarios.

## 3. Non-goals

- Changes to the skill-mine-llm skill itself. The skill already takes a
  `<palace-root>` parameter (`skills/mnemos-mine-llm/SKILL.md:49`); the hook
  just passes the production path.
- Changes to the pilot orchestrator / accept flow. The user can still run
  `mnemos mine --pilot-llm` for batch compare; 4.3.A does not touch this path.
- Yaml knobs (`auto_refine_parallel`, `hook_cap`). YAGNI for now; we start
  with fixed values. If demand arises, added in v0.4.1.
- LLM rerank / recall. That is the topic of 4.3 Skill-Recall, not 4.3.A.
- Removing the regex-mine flow. `mine_mode: script` is the default and is
  backward-compatible; only in `skill` mode is regex skipped.

## 4. Scope constraints (agreed with the user)

### 4.1 Pick strategy — "all unprocessed (excluding active)"

- The hook calls two separate pickers (see 4.3 two-phase queue):
  - Phase A picker: `Sessions/*.md` files that are **not OK in the mine ledger** but are OK in the refine ledger (refine done, mine left half-finished).
  - Phase B picker: files under `~/.claude/projects/**/*.jsonl` that are **not OK/SKIP in the refine ledger** + have **no live PID marker** under `.mnemos-active-sessions/` + are not under a subagent path.
- The existing `compute_backlog()` function is reused as a predicate (DRY); the picker returns the same filter as a `list`.
- The existing "last 3 by mtime" pick logic (`pick_recent_jsonls`) is removed **only on the skill-mine path**; preserved for `mine_mode: script` (no regression).

### 4.2 Fixed cap: 10 / fire (hook, total A+B)

- The cap is the **total work count per fire**, Phase A + Phase B combined. E.g., 7 unmined Sessions + 15 unrefined JSONLs → 7 Phase A + 3 Phase B = 10. The remaining 12 JSONLs go to the next fire.
- Phase A fills slots first (priority); remaining slots go to Phase B.
- In routine flow the backlog is 1-2; the cap is invisible.
- Fresh-install / long-gap edge case: the cap kicks in; the remaining backlog is picked up at the next SessionStart (slow burn).
- No adaptive logic. Fixed 10 = predictable behavior, single code path.

### 4.3 Two-phase queue

- **Phase A (priority):** `Sessions/*.md` not OK in the mine-skill ledger → skill-mine. Already refined; only mine remains.
- **Phase B (main):** unrefined JSONLs → chained per worker (refine → same worker mines the same session).

Phase B starts after Phase A finishes. If Phase A has 0 iterations, jump straight to B.

### 4.4 Parallelism

- **Hook:** fixed `parallel=1`. The bg subprocess uses the subscription quota responsibly in a place the user does not see.
- **Catch-up:** `--parallel N` flag, default 1. A conscious user opt-in.

### 4.5 Ledger coordination

- Refine ledger: `~/.claude/skills/mnemos-refine-transcripts/state/processed.tsv` (existing).
- Mine ledger: `~/.claude/skills/mnemos-mine-llm/state/mined.tsv` (existing).
- Both ledgers are written by the skills themselves. The hook/catch-up reads them for coordination (the question: is this source processed?).
- The backlog function takes the **intersection** of the two ledgers: "refine OK AND mine OK" = processed. Refine OK alone does not count (mid-chain crash).

### 4.6 Failure mode

- Subprocess crash / timeout / token-quota: that source is not marked OK in the ledger → natural retry on the next fire.
- The hook does not stop; it moves on to the next source. The log file (`<vault>/.mnemos-hook.log`) holds the error detail.
- In `_processing.xlsx`, the row's `refine_outcome` or `mined_outcome` column becomes `ERROR` + `notes`.

## 5. UX channels

### 5.1 Statusline (live, ephemeral)

Short, compact format — detail in `_processing.xlsx`.

```
mnemos: refining 3/10 · backlog 42       # Phase B, refine sub-step
mnemos: mining 5/10 · backlog 42         # Phase A or Phase B mine sub-step
mnemos: idle · last run 2m ago · backlog 0
```

The existing `write_status()` helper extends the `phase` field: `refining | mining | idle`. `current / total` is the position within picked.

### 5.2 `_processing.xlsx` (persistent audit trail — new)

**Location (definitive):** `<vault>/Mnemos/_processing.xlsx`

Not at the root (`<vault>/_processing.xlsx`) — the vault root is the user's main
working area, clutter risk. Under the palace (`Mnemos/`) for logical proximity
and matching the `_recycled/` underscore-ignored convention.

- `_` prefix → invisible in the Obsidian file list.
- Double-click in Windows Explorer → opens Excel.
- Native `.xlsx` (not CSV), written via openpyxl.
- Used jointly by hook + catch-up. Concurrent-safe via `filelock` (`.mnemos-processing-xlsx.lock`).

**Schema — one row per source (JSONL or Session md):**

| Column | Type | Example | Meaning |
|---|---|---|---|
| `source_type` | enum | `jsonl` / `md` | Raw transcript or refined Session? |
| `path` | abs-path | `C:/.../session-abc.jsonl` | Source file. Upsert key. |
| `refined_at` | ISO-ts / `-` | `2026-04-22T10:05:30Z` | `-` for md (already refined). |
| `refine_outcome` | enum | `OK` / `SKIP` / `ERROR` / `PENDING` / `-` | `-` for md. |
| `mined_at` | ISO-ts / `PENDING` | `2026-04-22T10:14:12Z` | Skill-mine success ts. |
| `mined_outcome` | enum | `OK` / `SKIP` / `ERROR` / `PENDING` | Final state. |
| `drawer_count` | int / `-` | `7` | Drawers produced by this source. |
| `tokens` | int / `-` | `480000` | Sum of skill-mine subprocess `usage`. |
| `notes` | str | `timeout` / `ledger-skipped; recovered` | Error/warning. |

**Upsert rule:**
- The `path` column is the primary key.
- If the row does not exist: a new row is added with `PENDING`s.
- If the row exists: the relevant columns are updated.
- `_processing.xlsx` is **not the canonical record** — the skill ledgers are canonical. The xlsx is a human-readable aggregate view; if it desyncs from the skill ledgers, the skills are taken as truth and the xlsx can be regenerated.

**Rebuild command (YAGNI but cheap):** `mnemos processing-log --rebuild` → produces the xlsx from scratch from the two ledgers. Not in the first version; small to add later.

### 5.3 Hook log (existing + phase disambiguation)

`<vault>/.mnemos-hook.log` — append-only text log. Subprocess exit code, timestamp, raw stderr. For debugging.

**4.3.A addition:** every subprocess line carries a `phase=A` or `phase=B` prefix:

```
[2026-04-22T10:05:30Z] phase=B refine C:/.../session-abc.jsonl
  exit=0
[2026-04-22T10:09:12Z] phase=B mine C:/.../Sessions/2026-04-22-abc.md
  exit=0  drawers=7  tokens=480000
[2026-04-22T10:13:45Z] phase=A mine C:/.../Sessions/2026-04-21-gyp.md
  exit=0  drawers=8  tokens=520000
```

Because the statusline `refining/mining` stays compact, it does not show the Phase A/B distinction;
hook.log carries the Phase distinction (for post-hoc debug).

## 6. Components

### 6.1 `mnemos/auto_refine.py` (change)

New flow:

```python
def _run_locked(..., runner, ...):
    mine_mode = _read_mine_mode(vault)  # existing helper

    if mine_mode == "skill":
        _run_skill_pipeline(vault, projects_dir, ledger_path, runner, cap=10)
    else:
        _run_legacy_regex_pipeline(...)  # existing code, not rewritten
```

New helpers:

- `_pick_unprocessed_jsonls(projects_dir, refine_ledger, active_sessions_dir, limit)` → list[Path]. The first `limit` JSONLs (mtime desc) that are not OK/SKIP in the refine ledger + have no live PID + are not subagent.
- `_pick_unmined_sessions(vault, mine_ledger, limit)` → list[Path]. `Sessions/*.md` not OK/SKIP in the mine ledger.
- `_run_skill_pipeline(vault, ..., cap=10)`:
  1. Phase A: `unmined = _pick_unmined_sessions(vault, mine_ledger, limit=cap)`. For each one: `claude --print "/mnemos-mine-llm <path> <palace>"`. Statusline + xlsx update.
  2. Phase B: `remaining = cap - len(unmined)`. If `remaining > 0`: `unrefined = _pick_unprocessed_jsonls(..., limit=remaining)`. For each:
     - Statusline: `refining current/total`
     - `claude --print "/mnemos-refine-transcripts <jsonl>"`
     - If refine OK: derive the relevant Sessions/xx.md path from the skill ledger query
     - Statusline: `mining current/total`
     - `claude --print "/mnemos-mine-llm <sessions/xx.md> <palace>"`
  3. Xlsx upsert before/after each subprocess.

The `_read_mine_mode()` helper already exists (commit `a5042a2`). The regex skip path now changes to "mine_mode=skill → route to _run_skill_pipeline"; the old skip behavior is removed.

### 6.3 `mnemos/catch_up.py` (new)

Foreground implementation of `mnemos catch-up`.

```python
def run_catch_up(
    vault: Path,
    limit: int | None = None,
    parallel: int = 1,
    dry_run: bool = False,
) -> CatchUpResult:
    """Foreground two-phase queue, progress prints."""
```

- Uses the same helpers as `_run_skill_pipeline` (DRY). Differences: no cap (or user-provided `--limit`), parallel N, foreground prints.
- One-line progress per source with an `[i/N]` prefix.
- Clean exit on Ctrl+C: the active subprocess finishes, remaining sources are left for the next `catch-up` or hook fire.
- `--dry-run`: show plan, do not invoke subprocesses.
- When `--parallel N` (N > 1) is passed, **a warning in the plan output**: "N concurrent Claude Code subprocesses will be opened. If your subscription quota does not allow this, the subprocesses may queue / fail. Continue? [y/N]" — bypass with `--yes`.
- Subprocess pattern: every claude call uses `--dangerously-skip-permissions` (same as the hook; non-interactive foreground command).

## 7. Data flow

```
Hook fire (SessionStart)
  ↓
auto_refine_hook.py (wrapper, <1s)
  ↓ detached subprocess
auto_refine_background → run() → _run_locked()
  ↓ (read mine_mode yaml)
_run_skill_pipeline(cap=10):
  Phase A: unmined = _pick_unmined_sessions(limit=cap)
    for each session_md:
      processing_log.upsert_row(source_type=md, path=session_md, mined_outcome=PENDING)
      write_status(phase=mining, ...)
      claude --print --dangerously-skip-permissions /mnemos-mine-llm <session_md> <palace>
      read mine ledger → outcome, drawer_count, tokens
      processing_log.upsert_row(path=session_md, mined_at, mined_outcome, drawer_count, tokens)
  Phase B: remaining = cap - len(unmined)
    if remaining > 0:
      unrefined = _pick_unprocessed_jsonls(limit=remaining)
      for each jsonl:
        processing_log.upsert_row(source_type=jsonl, path=jsonl, refine_outcome=PENDING)
        write_status(phase=refining, current=i, total=N)
        claude --print --dangerously-skip-permissions /mnemos-refine-transcripts <jsonl>
        read refine ledger → outcome, session_md_path (if OK)
        processing_log.upsert_row(path=jsonl, refined_at, refine_outcome)
        if refine OK:
          processing_log.upsert_row(path=jsonl, mined_outcome=PENDING)
          write_status(phase=mining, current=i, total=N)
          claude --print --dangerously-skip-permissions /mnemos-mine-llm <session_md_path> <palace>
          read mine ledger → outcome, drawer_count, tokens
          processing_log.upsert_row(path=jsonl, mined_at, mined_outcome, drawer_count, tokens)
          # NOTE: the JSONL row also keeps the mine info (one-row-per-JSONL schema).
          # The session_md path is not written to the notes column or to a separate audit row;
          # it is already in the ledger, and since the xlsx is a human aggregate, this is DRY.
  write_status(phase=idle, ...)

User triggers: mnemos catch-up [--limit N] [--parallel N]
  ↓ foreground
catch_up.run_catch_up()
  ↓ (mine_mode check — error if not skill)
Same _run_skill_pipeline helpers, no cap, parallel ThreadPoolExecutor N workers.
Progress prints to stdout per-source-per-step.
```

## 8. Error handling

| Situation | Behavior |
|---|---|
| Subprocess exit≠0 (refine or mine) | No OK in the ledger → retry on the next fire. Xlsx: `outcome=ERROR`, `notes=exit code`. stderr to the hook log. The hook continues. |
| Subprocess timeout | The current `runner` has no timeout set; not added in 4.3.A (YAGNI). The Claude Code subscription already has internal timeouts. |
| openpyxl write error (disk full, etc.) | Xlsx upsert fail → error to the hook log, the xlsx upsert is skipped. The skill ledgers are canonical anyway. The next upsert may succeed. |
| Filelock timeout (xlsx) | After 5s, warning to the log, that upsert is skipped. The skill ledger is canonical. |
| Refine OK but session path not found in the skill ledger | `SessionOutcome` falls back to the filesystem: the most recently written md file under `Sessions/` (the refine skill probably wrote it there). Pattern borrowed from the pilot orchestrator (pilot.py filesystem fallback). |
| Catch-up Ctrl+C | The active subprocess completes (subprocess is non-preemptive), then exit. Remaining sources stay pending. |
| `mine_mode: script` + `catch-up` | CLI error: "catch-up requires mine_mode: skill in mnemos.yaml". |

## 10.1 Performance notes

- **openpyxl import cold start:** ~300-500 ms. The hook wrapper (`auto_refine_hook.py`) spawns a detached bg subprocess; openpyxl is imported inside `auto_refine_background.py`, so the wrapper's <1s target is not affected. The bg subprocess will already spawn a Claude Code subprocess at cold start (>10s of work) — 500ms import is marginal.
- **Xlsx read-modify-write cycle:** ~50-100ms for a 500-row file (openpyxl benchmark). 3 calls per upsert (PENDING at phase start, OUTCOME at phase end). 5-6 cycles for one JSONL = ~500ms total. Marginal compared to subprocess time (>200s).
