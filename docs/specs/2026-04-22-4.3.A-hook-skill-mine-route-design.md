**Status:** Design — implementation pending

## 1. Problem

v0.4.2-alpha ile skill-mine canlıya geçti (pilot orchestrator + accept). Ama
auto-refine hook şu an sadece **regex mine**'ı tetikliyor
(`mnemos/auto_refine.py:538`). 2026-04-21 commit `a5042a2` acil guard olarak
`mine_mode: skill` için regex mine'ı skip etti — yani hook şu an "yeni gelen
session'ları refine eder, drawer üretmez". Kalıcı çözüm: hook **skill-mine**
çağırsın. Regex mine yolu 4.3.A sonrası tamamen atıl kalır (yalnız
`mine_mode: script` default olan yeni vault'lar için canlıdır, o akış zaten
eski).

Bir yan sorun daha: 2026-04-21 cleanup'ında (535 hybrid drawer temizliği) iki
post-accept session (v042-alpha-skill-mine-accept + gyp-deadline-anchor)
pilot orchestrator ile işlendiğinde **tüm 122 session'ı baştan mine etmeye
başladı**. Teşhis: pilot ledger path normalizasyonu eşleşmedi. Orchestrator
"pilot palace + accept" paradigmasında. Production-path'e tek session atomik
mine için dizayn edilmedi.

## 2. Goal

Auto-refine hook, `mine_mode: skill` vault'larda yeni JSONL transcript'leri
refine ettikten sonra **doğrudan skill-mine-llm skill'ini** çağırsın ve
drawer'ları **production `Mnemos/`'a** yazsın (pilot palace, accept bypass).
İki-fazlı pick/işleme ve persistent audit trail ile görünürlük tam.

Ek olarak: kullanıcıya manuel `mnemos catch-up [--limit N] [--parallel N]`
komutu — fresh-install veya büyük backlog senaryolarında hook'un yanı sıra
bulk işlem imkânı.

## 3. Non-goals

- Skill-mine-llm skill'inin kendisinde değişiklik. Skill zaten
  `<palace-root>` parametresi alıyor (`skills/mnemos-mine-llm/SKILL.md:49`);
  hook sadece production path'i geçer.
- Pilot orchestrator / accept flow değişikliği. Kullanıcı hâlâ
  `mnemos mine --pilot-llm` ile batch compare çalıştırabilir; 4.3.A bu yola
  dokunmaz.
- Yaml knob'lar (`auto_refine_parallel`, `hook_cap`). Şimdi YAGNI, sabit
  değerlerle başlanır. Talep gelirse v0.4.1'de eklenir.
- LLM rerank / recall. Bu 4.3 Skill-Recall'ın konusu, 4.3.A değil.
- Regex mine akışını kaldırmak. `mine_mode: script` default ve geriye
  dönük desteklenir; sadece `skill` modda regex skip.

## 4. Scope constraints (kullanıcı ile kararlaştırılan)

### 4.1 Pick stratejisi — "unprocessed tümü (aktif hariç)"

- Hook iki ayrı picker çağırır (bkz. 4.3 iki-fazlı queue):
  - Phase A picker: `Sessions/*.md` içinde **mine ledger'da OK olmayan** ve refine ledger'da OK olan (refine tamam, mine yarım kalmış) dosyalar.
  - Phase B picker: `~/.claude/projects/**/*.jsonl` içinde **refine ledger'da OK/SKIP olmayan** + `.mnemos-active-sessions/` altında live PID işareti **olmayan** + subagent altyolunda olmayan dosyalar.
- Mevcut `compute_backlog()` fonksiyonu predikat olarak yeniden kullanılır (DRY); picker aynı filter'i `list` olarak döner.
- Mevcut "last 3 by mtime" pick logic'i (`pick_recent_jsonls`) kaldırılır **sadece skill-mine yolunda**; `mine_mode: script` için korunur (regression yok).

### 4.2 Sabit cap: 10 / fire (hook, toplam A+B)

- Cap **fire başına toplam iş sayısı**, Phase A + Phase B birlikte. Örn: 7 unmined Session + 15 unrefined JSONL varsa → 7 Phase A + 3 Phase B = 10. Kalan 12 JSONL bir sonraki fire'a.
- Phase A önce yer doldurur (priority), kalan slot Phase B'ye.
- Rutin akışta backlog 1-2; cap hissedilmez.
- Fresh-install / uzun ara edge case'i: cap devreye girer, kalan backlog bir sonraki SessionStart'ta eline alınır (slow burn).
- Adaptive yok. Sabit 10 = tahmin edilebilir davranış, tek kod yolu.

### 4.3 İki-fazlı queue

- **Phase A (priority):** `Sessions/*.md` içinde mine-skill ledger'ında OK olmayanlar → skill-mine. Zaten refined, sadece mine kaldı.
- **Phase B (main):** unrefined JSONL'ler → worker başına zincir (refine → aynı worker aynı session için mine).

Phase A bitince Phase B başlar. Phase A 0 iterasyon ise doğrudan B'ye geçilir.

### 4.4 Paralellik

- **Hook:** sabit `parallel=1`. Bg subprocess'i kullanıcının görmediği yerde subscription quota'yı sorumluca kullanır.
- **Catch-up:** `--parallel N` flag, default 1. Kullanıcı bilinçli opt-in.

### 4.5 Ledger koordinasyonu

- Refine ledger: `~/.claude/skills/mnemos-refine-transcripts/state/processed.tsv` (mevcut).
- Mine ledger: `~/.claude/skills/mnemos-mine-llm/state/mined.tsv` (mevcut).
- Her iki ledger skill'in kendisi tarafından yazılır. Hook/catch-up koordinasyon için okur (sorgulamak: bu source processed mi?).
- Backlog fonksiyonu iki ledger'ın **kesişimini** alır: "refine OK VE mine OK" = processed. Sadece refine OK sayılmaz (crash mid-chain).

### 4.6 Failure mode

- Subprocess crash / timeout / token-quota: o source ledger'da OK işaretlenmez → bir sonraki fire'da doğal retry.
- Hook durmaz; bir sonraki source'a geçer. Log dosyası (`<vault>/.mnemos-hook.log`) hata detayını tutar.
- `_processing.xlsx`'te ilgili satırın `refine_outcome` veya `mined_outcome` kolonu `ERROR` + `notes` ile.

## 5. UX channels

### 5.1 Statusline (live, ephemeral)

Kısa, compact format — detay `_processing.xlsx`'te.

```
mnemos: refining 3/10 · backlog 42       # Phase B, refine alt-adımı
mnemos: mining 5/10 · backlog 42         # Phase A veya Phase B mine alt-adımı
mnemos: idle · last run 2m ago · backlog 0
```

Mevcut `write_status()` helper'ı `phase` field'ını genişletir: `refining | mining | idle`. `current / total` picked içinde konum.

### 5.2 `_processing.xlsx` (persistent audit trail — yeni)

**Konum:** `<vault>/Mnemos/_processing.xlsx`

- `_` prefix → Obsidian dosya listesinde görünmez.
- Windows Explorer'da çift-tık → Excel açar.
- Native `.xlsx` (CSV değil), openpyxl ile yazılır.
- Hook + catch-up ortak kullanır. `filelock` (`.mnemos-processing-xlsx.lock`) ile concurrent-safe.

**Şema — her kaynak (JSONL veya Session md) için tek satır:**

| Kolon | Tip | Örnek | Anlam |
|---|---|---|---|
| `source_type` | enum | `jsonl` / `md` | Ham transcript mi, refined Session mi? |
| `path` | abs-path | `C:/.../session-abc.jsonl` | Kaynak dosya. Upsert key. |
| `refined_at` | ISO-ts / `-` | `2026-04-22T10:05:30Z` | md için `-` (zaten refined). |
| `refine_outcome` | enum | `OK` / `SKIP` / `ERROR` / `PENDING` / `-` | md için `-`. |
| `mined_at` | ISO-ts / `PENDING` | `2026-04-22T10:14:12Z` | Skill-mine başarı ts'i. |
| `mined_outcome` | enum | `OK` / `SKIP` / `ERROR` / `PENDING` | Final state. |
| `drawer_count` | int / `-` | `7` | Bu source'un ürettiği drawer. |
| `tokens` | int / `-` | `480000` | Skill-mine subprocess `usage` toplamı. |
| `notes` | str | `timeout` / `ledger-skipped; recovered` | Hata/uyarı. |

**Upsert kuralı:**
- `path` kolonu primary key.
- Satır yoksa: yeni satır eklenir, `PENDING`'lerle.
- Satır varsa: ilgili kolonlar güncellenir.
- `_processing.xlsx` **canonical kayıt değildir** — skill ledger'ları canonical. Xlsx human-readable aggregate view; skill ledger'larıyla desync olursa skill'ler doğru kabul edilir, xlsx yeniden generate edilebilir.

**Rebuild komutu (YAGNI ama ucuz):** `mnemos processing-log --rebuild` → iki ledger'dan xlsx'i sıfırdan üretir. İlk sürümde yok, sonra ekleme küçük.

### 5.3 Hook log (mevcut, değişmez)

`<vault>/.mnemos-hook.log` — append-only text log. Subprocess exit code, timestamp, ham stderr. Debugging için.

## 6. Components

### 6.1 `mnemos/auto_refine.py` (değişiklik)

Yeni akış:

```python
def _run_locked(..., runner, ...):
    mine_mode = _read_mine_mode(vault)  # mevcut helper

    if mine_mode == "skill":
        _run_skill_pipeline(vault, projects_dir, ledger_path, runner, cap=10)
    else:
        _run_legacy_regex_pipeline(...)  # mevcut kod, yeniden yazılmaz
```

Yeni helper'lar:

- `_pick_unprocessed_jsonls(projects_dir, refine_ledger, active_sessions_dir, limit)` → list[Path]. Refine ledger'da OK/SKIP olmayan + live PID olmayan + subagent olmayan JSONL'lerin ilk `limit` tanesi (mtime desc).
- `_pick_unmined_sessions(vault, mine_ledger, limit)` → list[Path]. `Sessions/*.md` içinde mine ledger'da OK/SKIP olmayanlar.
- `_run_skill_pipeline(vault, ..., cap=10)`:
  1. Phase A: `unmined = _pick_unmined_sessions(vault, mine_ledger, limit=cap)`. Her biri için `claude --print "/mnemos-mine-llm <path> <palace>"`. Statusline + xlsx update.
  2. Phase B: `remaining = cap - len(unmined)`. Eğer `remaining > 0`: `unrefined = _pick_unprocessed_jsonls(..., limit=remaining)`. Her biri için:
     - Statusline: `refining current/total`
     - `claude --print "/mnemos-refine-transcripts <jsonl>"`
     - Refine OK ise: ilgili Sessions/xx.md path'ini skill ledger sorgusundan çıkar
     - Statusline: `mining current/total`
     - `claude --print "/mnemos-mine-llm <sessions/xx.md> <palace>"`
  3. Xlsx upsert her subprocess öncesi/sonrası.

`_read_mine_mode()` helper'ı zaten var (commit `a5042a2`). Regex skip path artık "mine_mode=skill → _run_skill_pipeline'a route" olarak değişiyor; eski skip davranışı kaldırılır.

### 6.3 `mnemos/catch_up.py` (yeni)

`mnemos catch-up` foreground implementasyonu.

```python
def run_catch_up(
    vault: Path,
    limit: int | None = None,
    parallel: int = 1,
    dry_run: bool = False,
) -> CatchUpResult:
    """Foreground iki-fazlı queue, progress prints."""
```

- `_run_skill_pipeline`'ın aynı helper'ları kullanır (DRY). Fark: cap'siz (veya user-provided `--limit`), parallel N, foreground print.
- Her source için `[i/N]` prefix'li tek satır progress.
- Ctrl+C temiz exit: mevcut subprocess biter, kalan source'lar bir sonraki `catch-up` veya hook fire'ına bırakılır.
- `--dry-run`: plan göster, subprocess çağırma.

## 7. Data flow

```
Hook fire (SessionStart)
  ↓
auto_refine_hook.py (wrapper, <1s)
  ↓ detached subprocess
auto_refine_background → run() → _run_locked()
  ↓ (mine_mode yaml oku)
_run_skill_pipeline(cap=10):
  Phase A: unmined = _pick_unmined_sessions(limit=cap)
    for each session_md:
      processing_log.upsert_row(source_type=md, path=session_md, mined_outcome=PENDING)
      write_status(phase=mining, ...)
      claude --print /mnemos-mine-llm <session_md> <palace>
      read mine ledger → outcome, drawer_count, tokens
      processing_log.upsert_row(path=session_md, mined_at, mined_outcome, drawer_count, tokens)
  Phase B: remaining = cap - len(unmined)
    if remaining > 0:
      unrefined = _pick_unprocessed_jsonls(limit=remaining)
      for each jsonl:
        processing_log.upsert_row(source_type=jsonl, path=jsonl, refine_outcome=PENDING)
        write_status(phase=refining, current=i, total=N)
        claude --print /mnemos-refine-transcripts <jsonl>
        read refine ledger → outcome, session_md_path (if OK)
        processing_log.upsert_row(path=jsonl, refined_at, refine_outcome)
        if refine OK:
          processing_log.upsert_row(path=jsonl, mined_outcome=PENDING)
          write_status(phase=mining, current=i, total=N)
          claude --print /mnemos-mine-llm <session_md_path> <palace>
          read mine ledger → outcome, drawer_count, tokens
          processing_log.upsert_row(path=jsonl, mined_at, mined_outcome, drawer_count, tokens)
          # NOT: JSONL satırı mine bilgisini de tutar (tek-satır-per-JSONL şeması).
          # session_md path'i notes kolonuna veya ayrı bir audit satırına yazılmaz;
          # ledger'da zaten var, xlsx human-aggregate olduğu için DRY.
  write_status(phase=idle, ...)

User triggers: mnemos catch-up [--limit N] [--parallel N]
  ↓ foreground
catch_up.run_catch_up()
  ↓ (mine_mode check — skill değilse error)
Same _run_skill_pipeline helpers, no cap, parallel ThreadPoolExecutor N workers.
Progress prints to stdout per-source-per-step.
```

## 8. Error handling

| Durum | Davranış |
|---|---|
| Subprocess exit≠0 (refine veya mine) | Ledger'da OK yok → bir sonraki fire retry. Xlsx: `outcome=ERROR`, `notes=exit code`. Hook log'a stderr. Hook devam eder. |
| Subprocess timeout | Mevcut `runner` timeout set'i yok; 4.3.A'da eklenmez (YAGNI). Claude Code subscription zaten iç timeout'lu. |
| openpyxl write hatası (disk full vb.) | Xlsx upsert fail → hook log'a error, xlsx skip edilir. Skill ledger'ları zaten canonical. Bir sonraki upsert başarılı olabilir. |
| Filelock timeout (xlsx) | 5s sonra log'a warning, o upsert skip. Skill ledger canonical. |
| Refine OK ama skill ledger'da session path bulunamıyor | `SessionOutcome` için filesystem fallback: `Sessions/` altında son yazılan md dosyası (refine skill muhtemelen oraya yazdı). Pilot orchestrator'dan alıntı pattern (pilot.py filesystem fallback). |
| Catch-up Ctrl+C | Aktif subprocess tamamlanır (subprocess non-preemptive), sonra exit. Kalan source'lar pending. |
| `mine_mode: script` + `catch-up` | CLI error: "catch-up requires mine_mode: skill in mnemos.yaml". |

## 9. Testing strategy

### 9.1 Unit

- `test_processing_log.py` — xlsx oluşturma, upsert (yeni satır + güncelleme), concurrent filelock, rebuild.
- `test_auto_refine.py` (ekleme) — `_pick_unprocessed_jsonls`, `_pick_unmined_sessions`, `_run_skill_pipeline` mock runner ile Phase A+B sırası.
- `test_catch_up.py` — dry-run output, --limit honor, --parallel N worker behavior, Ctrl+C temiz exit (SIGINT mock).

### 9.2 Integration

- Fresh vault + 5 JSONL fixture → hook fire → xlsx oluşur, 5 satır, Phase B çalışır (cap 10 altında), ledger'da 5 OK.
- mine_mode=skill + 3 unmined Session fixture + 2 unrefined JSONL → Phase A 3, sonra Phase B 2. Sırasız garanti.
- mine_mode=script → regex pipeline çalışır (regression).

### 9.3 Manual

- Kasamd'da yeni session aç/kapat → hook fire → `_processing.xlsx` Excel'de aç, yeni satır gör.
- `mnemos catch-up --dry-run` → plan output readable.

Full suite hedef **550+ pass** (mevcut 542 + ~10 yeni test).

## 10. Open questions

1. **Xlsx location alternatifi:** `<vault>/_processing.xlsx` (root, daha görünür ama vault clutter) vs `<vault>/Mnemos/_processing.xlsx` (palace içinde, bizim tercih). Kasamd pilot'unda denenip feedback alınır.
2. **Eski hook status JSON + yeni xlsx sync'i:** `.mnemos-hook-status.json` (live statusline) hâlâ canonical live state. Xlsx batch-aggregate. İki ayrı mekanizma; hybrid değil. Confusion önlemek için statusline docstring'inde belirtilir.
3. **Skill-mine subprocess'in session path'ini nasıl döndürdüğü:** `/mnemos-refine-transcripts` skill'i OK'de Sessions/ altına yazar, ledger'ına append eder `<jsonl>\t<session_md>\t...` formatıyla. Refine ledger okunarak session md path elde edilir. Bu pattern'in implementation sırasında doğrulanması gerekir (refine skill ledger format kontrol).
4. **`mnemos catch-up` pilot orchestrator ile confusion:** v0.4.1'de CLI help'e "For batch compare (script vs skill), use `mnemos mine --pilot-llm`. For routine backlog, use `mnemos catch-up`." eklenir.
