# SessionEnd Persistence + X-Close Coverage — Diagnosis

## Symptom

A user opens a fresh Claude Code session. The injected SessionStart
briefing reflects the **previous session minus one** — i.e. the
content from the session before the most recent /exit is missing.
Subsequent sessions show the same one-cycle lag.

The briefing skill itself works (cwd cache file at
`<vault>/.mnemos-briefings/<cwd-slug>.md` exists, mtime is recent),
but the *injection* used in SessionStart is always one /exit behind
what the cache contains. After a few seconds, the bg catchup spawned
at SessionStart writes a fresher cache — but the injection has
already happened against the older content.

## Reproduction (organic)

1. Session N: do work, type `/exit`. Confirm `Sessions/<date>-<slug>.md`
   appears (refine path is healthy, post-2026-04-28 stale-OK skip fix).
2. Session N+1: open in same cwd. Inspect the SessionStart-injected
   briefing — it does NOT mention session N's content.
3. Inspect `<vault>/.mnemos-briefings/<cwd-slug>.md` mtime — it has
   been updated by the bg catchup, but the cache content lags too
   (it reflects session N-1's content because session N's Session/.md
   was written by SessionEnd worker, which never updated the cache).

## Architecture (v1.1.0 spec, recap)

| Path                | Trigger     | What runs                                                     |
| ------------------- | ----------- | ------------------------------------------------------------- |
| Graceful close      | `/exit`     | SessionEnd hook → detached worker → refine + brief + identity |
| Hard close          | X-button    | SessionEnd skipped → next SessionStart sync fallback catches up |
| Background backlog  | SessionStart | bg auto_refine_hook scans pending JSONLs cross-project       |
| Per-cwd inject + bg | SessionStart | recall_briefing reads cache → injects → fires bg catchup     |

The contract is: **/exit creates the briefing immediately, X-close
defers to the next SessionStart**. Both should result in the next
session opening with current briefing.

## Root cause #1 — `_run_brief_regen` discards skill stdout

`mnemos/session_end_hook.py:_run_brief_regen` (introduced
2d0f077, v1.1 G7) invokes the briefing skill via:

```python
cmd = ["claude", "--print", ..., f"/mnemos-briefing {cwd}"]
subprocess.call(cmd, env=_child_env(),
                stdin=subprocess.DEVNULL,
                stdout=subprocess.DEVNULL,   # ← bug
                stderr=subprocess.DEVNULL)
```

The briefing skill's contract is "emit body to stdout, the wrapper
persists" (skill SKILL.md line 3:
*"Output goes to stdout; wrapper writes it to
<vault>/.mnemos-briefings/<cwd-slug>.md"*). With stdout piped to
`DEVNULL`, the body is **discarded**. No code path inside SessionEnd
ever writes the cwd cache.

The cache only ever updates via the **next** SessionStart's bg
catchup (`recall_briefing._spawn_bg_catchup` →
`brief_and_cache(cwd, vault)` which does capture stdout via a
tempfile). Because that's by-design async, the inject at the next
SessionStart sees the cache produced by the prior bg catchup — i.e.
one cycle stale.

## Root cause #2 — `_run_identity_refresh_if_due` discards skill stdout

Same DEVNULL pattern applied to `/mnemos-identity-refresh`. The
identity-refresh skill emits the refreshed profile to stdout; its
wrapper (the worker, in this case) is supposed to write it to
`<vault>/_identity/L0-identity.md`. With DEVNULL the refreshed body
is discarded; identity layer never advances from SessionEnd.

(There's a separate CLI path `mnemos identity refresh` that uses
`mnemos.identity.refresh()` which calls `_invoke_claude_print` with
`capture_output=True` and writes atomically. That path is healthy.
The bug is exclusively in the SessionEnd worker's parallel skill-
subprocess path.)

## Root cause #3 — sync fallback gate too narrow

`mnemos/recall_briefing.py:handle_session_start` CASE B branches:

```python
if pending and not cache_exists:    # ← gate
    capped = pending[-SUB_B2_PENDING_CAP:]
    for jsonl in capped:
        run_refine_sync(jsonl)
    brief_and_cache(inp.cwd, vault)
    ...
```

Pre-fix, sync fallback fired only when the cache was **completely
absent**. With a stale cache present (the typical case for
returning users), the SessionStart fast-pathed: injected the stale
cache + spawned bg catchup. That left the user one session behind
on every X-close (or any partial-fail SessionEnd) cycle, not just
once.

The original v1.1 sync-fallback design assumed cache_exists implies
cache_fresh. Once a /exit cycle leaves cache stale (root cause #1
above), that assumption breaks.

## Root cause #4 — LLM preamble bleeds into cache

The briefing skill prompt (`skills/mnemos-briefing/prompt.md`)
explicitly says:

> Only markdown body to stdout. The wrapper (recall_briefing.write_cache)
> adds the frontmatter. Start directly with the first bold label in
> the chosen output language (`**Current State:**` for English,
> `**Aktif durum:**` for Turkish).

LLMs occasionally violate this contract and emit conversational
preamble like:

```
I have all the layers I need. Let me synthesize the briefing.

---

**Current State:** ...
```

Reproduced empirically in kasamd's `C--Projeler-mnemos-v1-1.md`
cache (2026-04-29). The preamble bleeds into the cache, the next
SessionStart inject prepends the cross-check directive on top of
this preamble, and the user sees junk text before the actual
briefing.

## Fix

### `_run_brief_regen(cwd, vault)` calls `brief_and_cache`

```python
def _run_brief_regen(cwd: str, vault: Path) -> None:
    from mnemos.recall_briefing import brief_and_cache
    brief_and_cache(cwd, vault)
```

`brief_and_cache` wraps `run_brief_sync` (which captures the skill's
stdout via a tempfile) and writes the resulting body to
`<vault>/.mnemos-briefings/<cwd-slug>.md` via `write_cache`.

### `_run_identity_refresh_if_due` calls `identity.refresh`

```python
def _run_identity_refresh_if_due(vault: Path) -> None:
    from mnemos.identity import refresh, IdentityError
    ...
    # min_days gate (refresh() doesn't check this)
    ...
    try:
        refresh(vault, force=False)
    except IdentityError:
        pass
```

`refresh(vault, force=False)` enforces its own delta + tag-relevance
gates and persists atomically to `_identity/L0-identity.md` plus a
history snapshot. The auto_refresh + min_days gates remain in the
worker because `refresh()` doesn't know about them.

### Sync fallback gate relaxed

```python
if pending:           # was: if pending and not cache_exists
    capped = pending[-SUB_B2_PENDING_CAP:]
    for jsonl in capped:
        run_refine_sync(jsonl)
    brief_and_cache(inp.cwd, vault)
    ...
```

`MIN_USER_TURNS` (default 3) inside `find_unrefined_jsonls_for_cwd`
already filters trivial sessions out of `pending`, so this never
fires for noise — only for real abandoned sessions worth a 1–2
minute SessionStart delay to refine + brief.

### Defensive preamble strip

```python
_BOLD_LABEL_RE = re.compile(r"^\*\*[^*\n]+\*\*", re.MULTILINE)

def _strip_briefing_preamble(body: str) -> str:
    m = _BOLD_LABEL_RE.search(body)
    if m is None:
        return body
    return body[m.start():]
```

Called inside `brief_and_cache` before `write_cache`. Trims any text
before the first `**X:**` bold line. If no bold label exists (empty-
cwd message or unusual output), returns the body unchanged so the
caller decides.

## Out of scope (deferred)

- Bug #2 (`register_active_session` PID source on Windows): unchanged.
  The supersede + this fix together tolerate the PID staleness; durable
  liveness check via JSONL mtime is a future v1.3+ idea.
- LLM preamble at the prompt-engineering layer: this fix strips
  defensively in code. A separate future task could tighten the skill
  prompt's "no preamble" instruction empirically. Not blocking.

## Why this surfaced now

v1.2.0 + v1.2.1 had eliminated every other LLM-output-loss path
(refine race, stale-OK skip, identity isolation). The SessionEnd
persistence gap was the **last remaining** "LLM output never reached
the vault" bug — same class as the others, surfaced by elimination.
