# v0.3.14 — Backend Keşfi, Geçişi ve Kurtarma UX

**Status:** 🔄 design
**Owner:** tugra
**Depends on:** v0.3.0 released, backend parity verified 2026-04-17
**Target PyPI:** `mnemos-dev 0.3.1` (patch bump — v0.4 yolu açık kalsın)

---

## 1. Why

Kod zaten iki backend'i destekliyor (`search_backend: "chromadb" | "sqlite-vec"`)
ve 2026-04-17 parity benchmark'ı gösterdi ki:

- İki backend recall açısından **dördüncü ondalığa kadar aynı**
- Süre ~aynı (~62 dk on LongMemEval 10q)
- Miner pipeline'ı backend'e bakmaksızın 8027 drawer üretiyor

**Ama external user için bu alternatif görünmez:**

| Mevcut durum | Sorun |
|---|---|
| `config.search_backend` default `"chromadb"` | Gizli field, varlığı belirsiz |
| Değiştirmek için `mnemos.yaml`'ı elle açmak gerek | Dokümante değil |
| `mnemos init` backend sormuyor | İlk temas noktasında seçim yok |
| README'de tek parantezli cümle | "ChromaDB (or optional sqlite-vec)" — nasıl? |
| ChromaDB corrupt olunca cryptic hata | MemPalace'te de aynı sorun: issue #239, #832, #965 |
| Backend değiştirdikten sonra rebuild prosedürü yok | Kullanıcı bilmez, index eski data ile kalır |

### Karşılaştırma — MemPalace

MemPalace (42K★) aynı ChromaDB sorunlarını yaşıyor ama **henüz çözmedi**:

- [#239](https://github.com/MemPalace/mempalace/pull/239) repair komutu — open PR
- [#832](https://github.com/MemPalace/mempalace/issues/832) SQLite concurrency — open
- [#965](https://github.com/MemPalace/mempalace/issues/965) HNSW compaction bug — open, kritik
- [#700](https://github.com/MemPalace/mempalace/pull/700) Qdrant backend — open PR
- [#574](https://github.com/MemPalace/mempalace/pull/574) LanceDB migration — open PR

Yani: **alternatif backend + düzgün migrate akışı MemPalace'te bile yok**. Biz
zaten ship'lediğimiz sqlite-vec backend'i kullanıcıya sunarsak gerçek bir
farklılaştırıcı elde ederiz. Argüman artık "experimental" değil,
"production-equal alternative, recommended when ChromaDB misbehaves".

## 2. Scope

Beş iş, tek PyPI release:

1. **`mnemos init` backend prompt** — onboarding akışına seçim kapısı
2. **`mnemos migrate --backend X` komutu** — güvenli geçiş (backup + rebuild)
3. **Anlamlı hata mesajları** — ChromaDB init/load hatasında fallback öneren error
4. **README Troubleshooting bölümü** — iki satırda "index bozuldu → sqlite-vec'e geç"
5. **`mnemos status` backend bilgisi** — kullanıcı hangi backend'de, boyut, drawer sayısı

Out of scope:

- Yeni backend eklemek (Qdrant, LanceDB) — ihtiyaç yok, iki backend yeterli
- Backend-level benchmark UI — parity kanıtlandı, tekrar ölçmek gereksiz
- ChromaDB corruption'ını otomatik tespit ve otomatik switch — risk/fayda yanlış

## 3. Design — iş başına

### 3.14a — `mnemos init` backend prompt

**Akış:** Faz 3 (A/S/L karar) ile Faz 5 (hook) arasına yeni bir mini-prompt.
Sadece ilk-run'da, `.mnemos-pending.json`'da `backend_chosen=true` yoksa.

```
Vector backend — choose how embeddings are indexed:

  [C] ChromaDB (default)
      Mature, widely used. Occasional index corruption has been reported.

  [S] sqlite-vec
      Single-file SQLite + vec0 extension. Same recall as ChromaDB
      (verified 2026-04-17), recommended if you hit ChromaDB issues.

[C]hromaDB (default) / [S]qlite-vec:
```

**Platform vurgusu:** Windows'ta Python 3.14 tespit edilirse (`sys.platform
== 'win32' and sys.version_info >= (3, 14)`) ek bir satır:

```
  Note: on Windows + Python 3.14 sqlite-vec has shown fewer flush issues
  in our own testing — you may prefer [S] here.
```

Diğer platformlarda nötr copy korunur. Platform sniff'i `_resolve_backend_hint()`
helper'ı içinde; test edilebilir.

- i18n: yeni `backend_prompt` + `backend_chose_chromadb` + `backend_chose_sqlite`
  keys (EN + TR)
- Default boş Enter → ChromaDB (status quo, geri uyumlu)
- Seçim `mnemos.yaml`'a `search_backend: <backend>` satırı yazar
- `.mnemos-pending.json`'a `backend_chosen: true, backend: <name>` alanı eklenir
  (re-run idempotent)

**Neden "experimental" demiyoruz:** parity benchmark'ı var, yalan olur.
Copy "same recall as ChromaDB (verified)" diyor — dürüst satış.

### 3.14b — `mnemos migrate --backend <name>` komutu

**Kabuk:**

```bash
mnemos migrate --backend sqlite-vec [--dry-run] [--no-rebuild]
```

**Akış:**

1. Config yükle → mevcut backend tespit et
2. Aynı backend'e geçiş isteği → "already on X" no-op (exit 0)
3. **Pre-flight plan** — mevcut drawer sayısını ve source file sayısını oku:
   - `search.get_stats()["total_drawers"]` → mevcut N
   - `Sessions/`, `Topics/`, `memory/` altındaki `.md` sayısı → M
   - Tahmini rebuild süresi: **N × 0.5 saniye** (LongMemEval 2026-04-17 parity
     run: 8027 drawer / 62 dk ≈ 0.46 s/drawer). Aralık "≈ min–max min" olarak
     kullanıcıya gösterilir (±%30 marj).
   - Çıktı örneği:
     ```
     Migration plan:
       - From chromadb  →  to sqlite-vec
       - Current index: 8027 drawers
       - Source files:  518 md files in Sessions/ + Topics/ + memory/
       - Rebuild will scan these and produce fresh drawers
       - Estimated time: 30 – 55 minutes
       - Backup: <palace>/.chroma → .chroma.bak-2026-04-17
     ```
   - **Uyarı koşulu:** N > 0 ve M == 0 ise "No source files found — rebuild
     will produce an empty index. Continue? [y/N]" prompt. Bu user'ın `Sessions/`
     dizinini silmiş olduğu edge case'i yakalar.
4. Backup:
   - ChromaDB'den göç: `<palace>/.chroma/` → `<palace>/.chroma.bak-YYYY-MM-DD/`
   - sqlite-vec'ten göç: `<palace>/search.sqlite3` → `.bak-YYYY-MM-DD`
5. `mnemos.yaml`'ı güncelle (`search_backend: <new>`)
6. `--no-rebuild` değilse otomatik `mnemos mine Sessions/ --rebuild` tetikle
   - Rebuild başlangıcında: `Rebuilding index... this may take ≈ X minutes.
     Do not Ctrl+C; progress is resumable but interrupted state triggers
     migration-lock recovery on the next run.`
   - Vault'un Sessions/ (+ Topics/ + memory/) dizinleri canonical source
   - Yeni backend sıfırdan index eder
7. Özet rapor: "Migrated X → Y, N drawers indexed in T min, backup at Z"
   - Eğer rebuild sonrası drawer sayısı << önceki sayı (örn. < %80) → uyarı:
     "Index has fewer drawers than before (N_old → N_new). Missing source
     files? Backup preserved at Z."

**Güvenlik / ihtiyat:**

- Backup asla overwrite edilmez (aynı gün ikinci migration'da `.bak-YYYY-MM-DD.2`
  gibi counter; veya hata + kullanıcıya manuel silme talimatı)
- `--dry-run` sadece ne yapılacağını bastırır, hiçbir dosyayı değiştirmez
- Rebuild sırasında hata → yaml'ı geri al, backup'tan dizini geri yükle
  (atomic-ish — perfect atomic değil ama "no partial state" hedefi)
- Kullanıcı migration ortasında `^C` → WIP marker `<palace>/.mnemos-migration.lock`;
  sonraki run "previous migration incomplete" uyarısı + recovery opsiyonu

**Dosyalar:**
- `mnemos/migrate.py` (yeni) — saf mantık, test edilebilir
- `mnemos/cli.py` — `migrate` subparser + handler
- `tests/test_migrate.py` — dry-run, same-backend noop, ChromaDB→sqlite,
  sqlite→ChromaDB, backup overwrite guard, rebuild failure rollback

### 3.14c — Backend init/load hata mesajları

**Mevcut durum:** `ChromaBackend.__init__` bir `RuntimeError` veya
`chromadb.errors.*` atarsa Python traceback'i kullanıcıya doğrudan gidiyor.
Anlamsız ve panic-induce edici.

**Çözüm:** `search.py` factory wrapper — backend init veya ilk search
call'unda catch:

```python
try:
    engine = SearchEngine(config)
except (chromadb.errors.ChromaError, ChromaHNSWLoadError, sqlite3.DatabaseError) as e:
    raise BackendInitError(
        f"Failed to load {config.search_backend} index: {e.__class__.__name__}.\n"
        f"If this is a corruption, switch backends with:\n"
        f"  mnemos migrate --backend "
        f"{'sqlite-vec' if config.search_backend == 'chromadb' else 'chromadb'}\n"
        f"Original error: {e}"
    ) from e
```

- Hata sınıfı: `mnemos.errors.BackendInitError` (mevcut değilse yeni bir modül)
- CLI handler bu exception'ı yakalayıp kullanıcı-dostu mesaj döndürür; stack trace
  sadece `--verbose` ile gösterilir
- MCP server init sırasında da aynı mesaj `mnemos_status`'a iletilir (kullanıcı
  agent'ta da görebilsin)

### 3.14e — `mnemos status` backend bilgisi

**Sorun:** Bugün kullanıcı hangi backend'de çalıştığını öğrenmek için
`mnemos.yaml`'ı açıp `search_backend` field'ına bakmak zorunda. Destek
isteyen biri "hangi backend?" sorusuna cevap veremiyor.

**Çözüm:** `mnemos status` çıktısına tek bir satır ekle:

```
Backend: sqlite-vec (search.sqlite3 · 8027 drawers · 42.3 MB)
```

veya

```
Backend: chromadb (.chroma/ · 8027 drawers · 51.8 MB)
```

**Nereden:**
- Backend adı: `config.search_backend`
- Dosya/dizin: backend sınıfından `storage_path()` (yeni public method — hem
  Chroma hem sqlite-vec implement edecek)
- Drawer sayısı: zaten `get_stats()["total_drawers"]`
- Boyut: `du -sb` equivalent — `pathlib` ile recursive file size toplam

**Dosyalar:**
- `mnemos/search.py` — `SearchBackend` abstract'a `storage_path() -> Path`
  eklenir; iki backend implement eder
- `mnemos/cli.py` — `cmd_status` bu satırı render eder
- `tests/test_search.py` — iki backend için storage_path testi
- `tests/test_cli_status.py` (varsa) — backend satırı smoke testi

### 3.14d — README Troubleshooting

Tek yeni bölüm, README sonunda:

```markdown
## Troubleshooting

### ChromaDB index corruption

If `mnemos search` or `mnemos mine` fails with a ChromaDB error like
"HNSW corrupted", "DatabaseError", or segfaults, you can switch to the
sqlite-vec backend — same recall, single-file storage, more robust on
Windows + Python 3.14:

    mnemos migrate --backend sqlite-vec

This backs up the ChromaDB folder, updates `mnemos.yaml`, and rebuilds
the index from your vault's `Sessions/` + `Topics/` + `memory/` folders.
No memories are lost — your `.md` files are the source of truth.

### Switching back

    mnemos migrate --backend chromadb
```

Ayrıca hero section'da küçük bir ekleme — "Two vector backends (ChromaDB
default, sqlite-vec alternative — same recall, choose what suits your
machine)".

CONTRIBUTING'de de bir not: yeni backend eklemek yerine iki mevcut
backend'i korumak varsayılan (Qdrant/LanceDB gibi 3. backend contributor
PR'sı — "mevcut iki backend'i geçerse değerlendirilir").

## 6. Karar günlüğü

- **"Experimental" etiketi atıldı** — parity benchmark yalan söyletmez. Kullanıcıya
  sunulan copy "same recall, more robust on Windows" (dürüst).
- **Default ChromaDB'de kaldı** — mevcut kullanıcıları kırmamak için. Yeni
  install'larda init-time seçim kapısı açılıyor.
- **Migrate tek yönlü değil, iki yönlü** — kullanıcı sqlite-vec'i deneyip
  vazgeçebilmeli. Geri dönüş cezası yok.
- **Backup rebuild üzerine değil, rebuild yanında** — rebuild başarısız olursa
  yaml'ı eski haline geri alıp backup'ı restore edebilmek için. Perfect atomic
  değil ama "no lost state" hedefine yaklaşır.
- **3. backend eklemiyoruz** — MemPalace'in planladığı Qdrant/LanceDB yolu
  bizim için scope creep. İki iyi çalışan backend yeter; fark UX'te.
- **Migrate süre tahmini parity benchmark'ından** — 2026-04-17 LongMemEval
  parity run'ı 8027 drawer / 62 dk verdi, ≈0.46 s/drawer. Kullanıcıya
  ±%30 marjlı aralık gösteriyoruz ("30–55 min"). Tek sayı göstermek yanıltıcı
  olur, çok geniş aralık da güven verir ama faydasız — %30 marj dengeli.
- **Platform hint ayrı bir helper** — init prompt metni sabit kalıyor;
  sadece Windows+Py3.14 için ek satır eklensin diye `_resolve_backend_hint()`
  helper'ı. Diğer platformda sessiz. Over-engineering değil, çünkü mesaj
  copy'si platformdan bağımsız — hint ek bilgidir, ana copy değil.
- **Status backend satırı tek satır** — multi-line tablo değil. `mnemos status`
  çoktan yoğun; backend info tek bakışta "Backend: X (path · N drawers · Y MB)"
  formatında yeterli.
- **Rebuild sonrası drawer-drop uyarısı** — kullanıcı Sessions/'ta dosya
  silmişse migrate bunu yutmaz, açıkça bildirir ama **backup'ı silmez**.
  Kullanıcı backup'tan restore edebilir veya kabul edebilir — karar onun.
