# v0.3.14 — Backend Discovery, Migration and Recovery UX

**Status:** 🔄 design
**Owner:** tugra
**Depends on:** v0.3.0 released, backend parity verified 2026-04-17
**Target PyPI:** `mnemos-dev 0.3.1` (patch bump — keep the v0.4 path open)

---

## 1. Why

The code already supports two backends (`search_backend: "chromadb" | "sqlite-vec"`)
and the parity benchmark on 2026-04-17 showed that:

- Both backends are **identical down to the fourth decimal** in recall
- Runtime is roughly the same (~62 min on LongMemEval 10q)
- The miner pipeline produces 8027 drawers regardless of the backend

**But for an external user this alternative is invisible:**

| Current state | Problem |
|---|---|
| `config.search_backend` defaults to `"chromadb"` | Hidden field, its existence is unclear |
| Changing it requires editing `mnemos.yaml` by hand | Not documented |
| `mnemos init` does not ask about the backend | No choice at the first touchpoint |
| README has a single parenthetical sentence | "ChromaDB (or optional sqlite-vec)" — how? |
| Cryptic error when ChromaDB is corrupt | Same problem in MemPalace: issue #239, #832, #965 |
| No rebuild procedure after switching backends | The user does not know; the index stays stale |

### Comparison — MemPalace

MemPalace (42K stars) suffers the same ChromaDB issues but **has not solved them yet**:

- [#239](https://github.com/MemPalace/mempalace/pull/239) repair command — open PR
- [#832](https://github.com/MemPalace/mempalace/issues/832) SQLite concurrency — open
- [#965](https://github.com/MemPalace/mempalace/issues/965) HNSW compaction bug — open, critical
- [#700](https://github.com/MemPalace/mempalace/pull/700) Qdrant backend — open PR
- [#574](https://github.com/MemPalace/mempalace/pull/574) LanceDB migration — open PR

So: **even MemPalace has no alternative backend with a proper migration flow**. If we
expose the sqlite-vec backend we have already shipped, we get a real
differentiator. The argument is no longer "experimental" but
"production-equal alternative, recommended when ChromaDB misbehaves".

---

## 2. Scope

Five items, one PyPI release:

1. **`mnemos init` backend prompt** — add a choice gate to the onboarding flow
2. **`mnemos migrate --backend X` command** — safe migration (backup + rebuild)
3. **Meaningful error messages** — fallback-suggesting error on ChromaDB init/load failure
4. **README Troubleshooting section** — two lines: "index broken → switch to sqlite-vec"
5. **`mnemos status` backend info** — which backend the user is on, size, drawer count

Out of scope:

- Adding a new backend (Qdrant, LanceDB) — not needed, two backends are enough
- Backend-level benchmark UI — parity is proven, re-measuring is unnecessary
- Auto-detecting ChromaDB corruption and auto-switching — risk/reward is wrong

## 3. Design — per task

### 3.14a — `mnemos init` backend prompt

**Flow:** A new mini-prompt between Phase 3 (A/S/L decision) and Phase 5 (hook).
Only on first run, when `.mnemos-pending.json` does not have `backend_chosen=true`.

```
Vector backend — choose how embeddings are indexed:

  [C] ChromaDB (default)
      Mature, widely used. Occasional index corruption has been reported.

  [S] sqlite-vec
      Single-file SQLite + vec0 extension. Same recall as ChromaDB
      (verified 2026-04-17), recommended if you hit ChromaDB issues.

[C]hromaDB (default) / [S]qlite-vec:
```

**Platform emphasis:** If Python 3.14 on Windows is detected (`sys.platform
== 'win32' and sys.version_info >= (3, 14)`), an extra line:

```
  Note: on Windows + Python 3.14 sqlite-vec has shown fewer flush issues
  in our own testing — you may prefer [S] here.
```

On other platforms the neutral copy is preserved. The platform sniff lives in
the `_resolve_backend_hint()` helper; testable.

- i18n: new `backend_prompt` + `backend_chose_chromadb` + `backend_chose_sqlite`
  keys (EN + TR)
- Empty default Enter → ChromaDB (status quo, backward compatible)
- The choice writes a `search_backend: <backend>` line to `mnemos.yaml`
- A `backend_chosen: true, backend: <name>` field is added to `.mnemos-pending.json`
  (re-run idempotent)

**Why we are not saying "experimental":** the parity benchmark exists, it would be a lie.
The copy says "same recall as ChromaDB (verified)" — an honest pitch.

### 3.14b — `mnemos migrate --backend <name>` command

**Shell:**

```bash
mnemos migrate --backend sqlite-vec [--dry-run] [--no-rebuild]
```

**Flow:**

1. Load config → detect current backend
2. Migration request to the same backend → "already on X" no-op (exit 0)
3. **Pre-flight plan** — read current drawer count and source-file count:
   - `search.get_stats()["total_drawers"]` → current N
   - count of `.md` files under `Sessions/`, `Topics/`, `memory/` → M
   - Estimated rebuild duration: **N × 0.5 seconds** (LongMemEval 2026-04-17 parity
     run: 8027 drawers / 62 min ≈ 0.46 s/drawer). The user is shown a range
     "≈ min–max min" (±30% margin).
   - Example output:
     ```
     Migration plan:
       - From chromadb  →  to sqlite-vec
       - Current index: 8027 drawers
       - Source files:  518 md files in Sessions/ + Topics/ + memory/
       - Rebuild will scan these and produce fresh drawers
       - Estimated time: 30 – 55 minutes
       - Backup: <palace>/.chroma → .chroma.bak-2026-04-17
     ```
   - **Warning condition:** if N > 0 and M == 0, prompt "No source files found — rebuild
     will produce an empty index. Continue? [y/N]". This catches the edge case where
     the user has deleted their `Sessions/` directory.
4. Backup:
   - Migrating from ChromaDB: `<palace>/.chroma/` → `<palace>/.chroma.bak-YYYY-MM-DD/`
   - Migrating from sqlite-vec: `<palace>/search.sqlite3` → `.bak-YYYY-MM-DD`
5. Update `mnemos.yaml` (`search_backend: <new>`)
6. Unless `--no-rebuild`, automatically trigger `mnemos mine Sessions/ --rebuild`
   - At rebuild start: `Rebuilding index... this may take ≈ X minutes.
     Do not Ctrl+C; progress is resumable but interrupted state triggers
     migration-lock recovery on the next run.`
   - The vault's Sessions/ (+ Topics/ + memory/) directories are the canonical source
   - The new backend indexes from scratch
7. Summary report: "Migrated X → Y, N drawers indexed in T min, backup at Z"
   - If the post-rebuild drawer count is much lower than before (e.g., < 80%) → warning:
     "Index has fewer drawers than before (N_old → N_new). Missing source
     files? Backup preserved at Z."

**Safety / caution:**

- A backup is never overwritten (a second migration on the same day uses a counter
  like `.bak-YYYY-MM-DD.2`; or errors out with a manual-deletion instruction to the user)
- `--dry-run` only prints what would happen; it modifies no files
- On a rebuild error → revert the yaml, restore the directory from backup
  (atomic-ish — not perfect atomic, but the goal is "no partial state")
- If the user hits `^C` mid-migration → WIP marker `<palace>/.mnemos-migration.lock`;
  the next run shows a "previous migration incomplete" warning + recovery option

**Files:**
- `mnemos/migrate.py` (new) — pure logic, testable
- `mnemos/cli.py` — `migrate` subparser + handler
- `tests/test_migrate.py` — dry-run, same-backend noop, ChromaDB→sqlite,
  sqlite→ChromaDB, backup overwrite guard, rebuild failure rollback

### 3.14c — Backend init/load error messages

**Current state:** if `ChromaBackend.__init__` raises a `RuntimeError` or
`chromadb.errors.*`, the Python traceback goes straight to the user.
Meaningless and panic-inducing.

**Solution:** a `search.py` factory wrapper — catches errors at backend init
or first search call:

```python
try:
    engine = SearchEngine(config)
except (chromadb.errors.ChromaError, ChromaHNSWLoadError, sqlite3.DatabaseError) as e:
    raise BackendInitError(
        f"Failed to load {config.search_backend} index: {e.__class__.__name__}.\n"
        f"If this is a corruption, switch backends with:\n"
        f"  mnemos migrate --backend "
        f"{'sqlite-vec' if config.search_backend == 'chromadb' else 'chromadb'}\n"
        f"Original error: {e}"
    ) from e
```

- Error class: `mnemos.errors.BackendInitError` (new module if it does not exist)
- The CLI handler catches this exception and returns a user-friendly message; the stack trace
  is shown only with `--verbose`
- The same message is also forwarded to `mnemos_status` during MCP server init (so the user
  can see it in the agent too)

### 3.14e — `mnemos status` backend info

**Problem:** today, to find out which backend is in use, the user must open
`mnemos.yaml` and look at the `search_backend` field. Anyone seeking support
cannot answer the question "which backend?".

**Solution:** add a single line to the `mnemos status` output:

```
Backend: sqlite-vec (search.sqlite3 · 8027 drawers · 42.3 MB)
```

or

```
Backend: chromadb (.chroma/ · 8027 drawers · 51.8 MB)
```

**Sources:**
- Backend name: `config.search_backend`
- File/directory: from the backend class, `storage_path()` (new public method —
  to be implemented by both Chroma and sqlite-vec)
- Drawer count: already `get_stats()["total_drawers"]`
- Size: `du -sb` equivalent — recursive total file size with `pathlib`

**Files:**
- `mnemos/search.py` — add `storage_path() -> Path` to the `SearchBackend` abstract;
  implemented by both backends
- `mnemos/cli.py` — `cmd_status` renders this line
- `tests/test_search.py` — storage_path test for both backends
- `tests/test_cli_status.py` (if present) — smoke test for the backend line

### 3.14d — README Troubleshooting

A single new section at the end of the README:

```markdown
## Troubleshooting

### ChromaDB index corruption

If `mnemos search` or `mnemos mine` fails with a ChromaDB error like
"HNSW corrupted", "DatabaseError", or segfaults, you can switch to the
sqlite-vec backend — same recall, single-file storage, more robust on
Windows + Python 3.14:

    mnemos migrate --backend sqlite-vec

This backs up the ChromaDB folder, updates `mnemos.yaml`, and rebuilds
the index from your vault's `Sessions/` + `Topics/` + `memory/` folders.
No memories are lost — your `.md` files are the source of truth.

### Switching back

    mnemos migrate --backend chromadb
```

Also a small addition in the hero section — "Two vector backends (ChromaDB
default, sqlite-vec alternative — same recall, choose what suits your
machine)".

A note in CONTRIBUTING as well: the default stance is to maintain the two
existing backends rather than adding new ones (a contributor PR for a 3rd
backend like Qdrant/LanceDB — "considered if it beats both existing backends").

## 4. Acceptance criteria

- [ ] `mnemos init` on a clean vault asks for a backend choice (EN + TR), and the
      choice is written to `mnemos.yaml`
- [ ] `mnemos migrate --backend sqlite-vec` works on a ChromaDB vault,
      backup + rebuild + yaml update all complete
- [ ] `mnemos migrate --backend chromadb` works on a sqlite-vec vault
- [ ] `mnemos migrate --dry-run` modifies no files, only prints the plan
- [ ] Migrate to the same backend → exit 0, no-op message
- [ ] On a corrupt ChromaDB, `mnemos search` → BackendInitError + migrate
      suggestion, traceback hidden
- [ ] README has a Troubleshooting section, single paragraph + two commands
- [ ] New-user pilot: clean vault → init → choose sqlite-vec → mine →
      search works
- [ ] `mnemos status` output contains a `Backend: <name> (<path> · N drawers · X MB)`
      line, rendering correctly for both ChromaDB and sqlite-vec
- [ ] `mnemos migrate --dry-run` shows drawer + source-file counts + duration estimate;
      asks for confirmation with a warning when `Sessions/` is empty

## 6. Decision log

- **The "experimental" label is dropped** — the parity benchmark does not lie. The user-facing
  copy is "same recall, more robust on Windows" (honest).
- **Default stays at ChromaDB** — to avoid breaking existing users. A new install opens the
  init-time choice gate.
- **Migrate is bidirectional, not one-way** — the user must be able to try sqlite-vec
  and back out. No penalty for going back.
- **Backup alongside rebuild, not on top of it** — so that if the rebuild fails we can
  revert the yaml and restore the backup. Not perfect atomic, but it approaches the
  "no lost state" goal.
- **No 3rd backend** — MemPalace's planned Qdrant/LanceDB path is scope creep for us.
  Two well-functioning backends are enough; the differentiator is UX.
- **Migrate duration estimate from the parity benchmark** — the 2026-04-17 LongMemEval
  parity run gave 8027 drawers / 62 min, ≈0.46 s/drawer. We show the user a ±30%
  margin range ("30–55 min"). A single number would be misleading; a very wide
  range looks safe but is useless — 30% margin is balanced.
- **Platform hint is a separate helper** — the init prompt body stays fixed;
  `_resolve_backend_hint()` exists only to add the extra line for Windows+Py3.14.
  Silent on other platforms. Not over-engineering, because the message
  copy is platform-independent — the hint is supplementary information, not the main copy.
- **Status backend line is single-line** — no multi-line table. `mnemos status` is
  already dense; backend info is enough at a glance in the
  format "Backend: X (path · N drawers · Y MB)".
- **Drawer-drop warning after rebuild** — if the user has deleted files in Sessions/,
  migrate does not swallow this; it reports it explicitly but **does not delete the backup**.
  The user can restore from the backup or accept it — their call.
