# Duplicate-Refine Race Condition — Diagnosis

## Symptom

A single Claude Code transcript (one `.jsonl`) gets refined twice and
produces TWO `Sessions/<date>-<slug>.md` files in the vault, each with
a different slug because the LLM picks a non-deterministic title on
each refine run.

## Reproduction (organic)

1. Open Claude Code session in any project cwd (e.g.
   `C:\Users\tugrademirors\OneDrive\Masaüstü\farcry`).
2. Have a normal exchange (≥3 user turns).
3. Type `/exit`.
4. Inspect `<vault>/Sessions/` — TWO new `<date>-<slug>.md` files exist
   for the same JSONL.

Frequency: appears to happen on **every** /exit since the user upgraded
to v1.1.0. Verified on the v1.2.0 implementation session itself
(`3078dfac-...jsonl` → `2026-04-28-mnemos-v1-2-english-only-output-impl.md`
ledger-recorded twice, same slug because the LLM happened to pick the
same title under both runs).

## Evidence (ledger, `~/.claude/skills/mnemos-refine-transcripts/state/processed.tsv`)

```
…\farcry\b11f3a74-…jsonl	OK	2026-04-28-desktop-tavuk-sprite-teknoloji-kararlari.md
…\farcry\b11f3a74-…jsonl	OK	2026-04-28-desktop-tavuk-sprite-teknoloji-kararlari.md
…\farcry\b11f3a74-…jsonl	OK	2026-04-28-desktop-tavuk-tech-stack-sprite-strateji.md
…\farcry\b11f3a74-…jsonl	OK	2026-04-28-desktop-tavuk-tech-stack-sprite-strateji.md
```

Same JSONL, two different slugs, ledger entries duplicated. Some lines
in the TSV are also corrupted — TAB delimiters lost, paths spliced —
which is the fingerprint of two processes appending concurrently
without file locking.

## Root cause (high confidence)

Three independent code paths all call into the refine pipeline, and
none of them coordinate via lock or atomic claim:

1. **SessionStart hook** `mnemos-auto-refine` (`mnemos/auto_refine_hook.py`)
   spawns `mnemos.auto_refine_background` which scans pending JSONLs
   for the cwd and refines them.
2. **SessionStart hook** `mnemos-recall-briefing` (`mnemos/recall_briefing.py`)
   has a `--catchup` path (CASE B sync fallback for missed SessionEnd)
   that also scans pending JSONLs and refines them.
3. **SessionEnd hook** `mnemos-session-end` (`mnemos/session_end_hook.py`)
   spawns `mnemos.session_end_worker` which refines THIS transcript's
   JSONL (and only this one).

When the user opens a session in cwd X, both SessionStart hooks fire
in parallel. They both look at the pending JSONL list — which now
contains the previously-completed JSONL — and both decide to refine
it. Worker A and Worker B run `claude --print` independently, the LLM
chooses different slug words, both write `Sessions/.md`, both append
to the ledger. The ledger never serves as a dedup gate because:

- The check-then-write pattern is not atomic across processes.
- The append uses plain Python `open(…, "a")`, no `fcntl.flock`,
  no `msvcrt.locking`.

The settings.json today has the v1.0 `mnemos-auto-refine` SessionStart
entry alongside the v1.1 `mnemos-session-end` SessionEnd entry. With
v1.1's SessionEnd-driven architecture, the SessionStart auto-refine is
redundant — its only legitimate role is the "missed SessionEnd"
catchup, which `recall_briefing --catchup` (CASE B) already handles.

## Architectural intent (corrected 2026-04-28)

Both hooks coexist by design — they are not a duplicate that one
should remove. The split is:

- **SessionEnd** (`/exit` graceful close): primary refine path.
  Worker spawns with `CREATE_BREAKAWAY_FROM_JOB` and refines THIS
  transcript synchronously.
- **SessionStart** (next session opens): two safety nets.
  - `mnemos-auto-refine` legacy v1.0 entry — proactive backlog refine
    across all projects (still useful for users with deep unrefined
    history).
  - `mnemos-recall-briefing --catchup` — CASE B sync fallback that
    catches transcripts whose SessionEnd never fired (X-close,
    `kill -9`, OS shutdown mid-conversation).

Removing either hook would re-introduce the very gap v1.1 was
designed to close: a forceful X-close leaves a transcript that no
one will ever refine. The fix must make BOTH paths safe to run
without producing duplicate work — not collapse them into one.

## Why the same race manifests differently day-to-day

LLM slug naming is non-deterministic. Two parallel
`claude --print /mnemos-refine-transcripts <jsonl>` calls can:

- **Converge** on the same slug — both write to the same path,
  the second `Write` overwrites the first → user sees one final
  file, but the contents are whichever invocation finished last
  (silent data loss; the other invocation's summary is gone).
- **Diverge** on different slugs — both files persist as siblings
  under `Sessions/` (visible duplicate, observable).

The race itself is identical in both cases; only the artifact
differs. This explains why the user observed "yesterday a duplicate
seemed to auto-delete itself" (converged-slug overwrite) vs "today
two duplicates persist" (divergent-slug). OneDrive sync conflict
resolution may also occasionally rename one writer's output to a
`*-conflict.md` filename, adding another visible mode.

The race is therefore not just a cosmetic dup-file bug; it is a
**data consistency bug** — the user can lose summary content
silently whenever two writers happen to converge on the same slug.

## Fix scope (v1.2.1)

### Required

1. **Per-JSONL file lock + ledger recheck.**
   - New module `mnemos/refine_lock.py` exposing
     `claim_jsonl_for_refine(jsonl, ledger) -> ContextManager | None`.
     Returns a context manager (lock held) on successful claim, or
     `None` if either (a) another worker holds the lock right now,
     or (b) the ledger already has an `OK\t<slug>` entry for this
     JSONL.
   - Lock backend: `filelock.FileLock` (already a project dependency,
     used elsewhere in the worker) keyed on the JSONL filename stem,
     stored under `<ledger-dir>/locks/<stem>.lock`.
   - `timeout=0` — fail fast on busy lock; the caller must skip
     silently rather than queue up.
   - Wire all THREE refine callers through this gate:
     - `session_end_hook._run_refine`
     - `recall_briefing.run_refine_sync` (used by `--catchup`)
     - `auto_refine.run` (per-JSONL loop body)
   - Inside the lock: re-read the ledger. If another worker finished
     between scan and lock acquire, the ledger now has the OK entry
     → release lock and skip. Without this recheck, a worker that
     waited briefly for the lock would invariably do redundant work.

2. **Ledger corruption recovery CLI.**
   - `mnemos refine-ledger --normalize [--vault PATH]` — reads the
     TSV, reconstructs valid 3-column lines (heuristic: split on the
     last two TAB columns; the path is everything before), drops
     duplicate paths keeping the most recent OK entry (SKIP entries
     yield to OK), drops entries whose JSONL no longer exists,
     writes back atomically (tmp + rename).
   - Idempotent. Users run it once after upgrading to v1.2.1 to
     clean the corruption already present.

### Recommended (defense in depth, optional in v1.2.1)

3. **Pre-refine Sessions/.md collision check.** Before `claude --print`,
   the wrapper inspects `<vault>/Sessions/` for files whose stem
   begins with `<date>-` and contains overlapping slug tokens. If a
   recently-modified match exists (mtime within last 60s), skip.
   Layered defense against lock bypass — useful but not required;
   the lock alone is sufficient for the documented scenarios.

### Tests

- `test_claim_jsonl_for_refine_succeeds_when_unprocessed` — happy
  path: lock acquired, ledger has no OK entry → returns context
  manager, releases on exit.
- `test_claim_jsonl_for_refine_skips_when_ledger_has_ok` — ledger
  already records OK for this JSONL → returns None (no lock
  acquired, no work to do).
- `test_claim_jsonl_for_refine_skips_when_concurrent_holder` — second
  caller while first holds lock → returns None.
- `test_concurrent_refine_workers_produce_one_session_file` —
  end-to-end: two `concurrent.futures` workers both call
  `_run_refine_with_lock` on the same JSONL. Assert exactly ONE
  `claude --print` invocation occurred (mock the runner) and
  exactly ONE OK line in the ledger.
- `test_refine_ledger_normalize_repairs_corrupted_lines` — feed a
  TSV with concatenated paths (TAB-stripped), duplicate OK rows for
  the same path, an entry pointing to a deleted JSONL; assert
  post-normalize is valid 3-column TSV with no dups and no dead
  entries.

## Status of v1.2.0 vis-à-vis this bug

This race **predates v1.2.0** — evidence shows it affecting JSONLs
refined this morning during the v1.2.0 implementation itself,
before any of the v1.2.0 prompt changes landed. v1.2.0 is purely a
locale-aware output schema change; it neither introduced nor depends
on the refine-pipeline concurrency model.

**Decision (2026-04-28):** Ship v1.2.0 as-is with this bug documented
as a known issue. Defer the fix to v1.2.1 to keep release scope
crisp and avoid coupling a non-trivial concurrency fix to a
documentation-strings release.

## User-facing workaround (until v1.2.1)

When `<vault>/Sessions/` contains two `<date>-<slug>.md` files for
the same conversation:

1. Compare both files; pick the more complete one.
2. Move the loser to `<vault>/_recycled/` (the watcher will clean
   the index).
3. Optionally append a one-line note to the keeper's `## See Also`
   section explaining the dedup decision.

To stop the race from recurring before v1.2.1 ships, manually edit
`~/.claude/settings.json` and remove the `mnemos-auto-refine`
SessionStart hook entry. Keep only `mnemos-recall-briefing` and
`mnemos-session-end`.
