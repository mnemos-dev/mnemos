# Duplicate-Refine Race Condition — Diagnosis

## Symptom

A single Claude Code transcript (one `.jsonl`) gets refined twice and
produces TWO `Sessions/<date>-<slug>.md` files in the vault, each with
a different slug because the LLM picks a non-deterministic title on
each refine run.

## Reproduction (organic)

1. Open Claude Code session in any project cwd (e.g.
   `C:\Users\tugrademirors\OneDrive\Masaüstü\farcry`).
2. Have a normal exchange (≥3 user turns).
3. Type `/exit`.
4. Inspect `<vault>/Sessions/` — TWO new `<date>-<slug>.md` files exist
   for the same JSONL.

Frequency: appears to happen on **every** /exit since the user upgraded
to v1.1.0. Verified on the v1.2.0 implementation session itself
(`3078dfac-...jsonl` → `2026-04-28-mnemos-v1-2-english-only-output-impl.md`
ledger-recorded twice, same slug because the LLM happened to pick the
same title under both runs).

## Evidence (ledger, `~/.claude/skills/mnemos-refine-transcripts/state/processed.tsv`)

```
…\farcry\b11f3a74-…jsonl	OK	2026-04-28-desktop-tavuk-sprite-teknoloji-kararlari.md
…\farcry\b11f3a74-…jsonl	OK	2026-04-28-desktop-tavuk-sprite-teknoloji-kararlari.md
…\farcry\b11f3a74-…jsonl	OK	2026-04-28-desktop-tavuk-tech-stack-sprite-strateji.md
…\farcry\b11f3a74-…jsonl	OK	2026-04-28-desktop-tavuk-tech-stack-sprite-strateji.md
```

Same JSONL, two different slugs, ledger entries duplicated. Some lines
in the TSV are also corrupted — TAB delimiters lost, paths spliced —
which is the fingerprint of two processes appending concurrently
without file locking.

## Root cause (high confidence)

Three independent code paths all call into the refine pipeline, and
none of them coordinate via lock or atomic claim:

1. **SessionStart hook** `mnemos-auto-refine` (`mnemos/auto_refine_hook.py`)
   spawns `mnemos.auto_refine_background` which scans pending JSONLs
   for the cwd and refines them.
2. **SessionStart hook** `mnemos-recall-briefing` (`mnemos/recall_briefing.py`)
   has a `--catchup` path (CASE B sync fallback for missed SessionEnd)
   that also scans pending JSONLs and refines them.
3. **SessionEnd hook** `mnemos-session-end` (`mnemos/session_end_hook.py`)
   spawns `mnemos.session_end_worker` which refines THIS transcript's
   JSONL (and only this one).

When the user opens a session in cwd X, both SessionStart hooks fire
in parallel. They both look at the pending JSONL list — which now
contains the previously-completed JSONL — and both decide to refine
it. Worker A and Worker B run `claude --print` independently, the LLM
chooses different slug words, both write `Sessions/.md`, both append
to the ledger. The ledger never serves as a dedup gate because:

- The check-then-write pattern is not atomic across processes.
- The append uses plain Python `open(…, "a")`, no `fcntl.flock`,
  no `msvcrt.locking`.

The settings.json today has the v1.0 `mnemos-auto-refine` SessionStart
entry alongside the v1.1 `mnemos-session-end` SessionEnd entry. With
v1.1's SessionEnd-driven architecture, the SessionStart auto-refine is
redundant — its only legitimate role is the "missed SessionEnd"
catchup, which `recall_briefing --catchup` (CASE B) already handles.

## v1.1 install-hook expectation

`mnemos install-hook --v1` was supposed to deactivate or remove the
v1.0 SessionStart auto-refine entry when the user opted into the v1.1
SessionEnd worker pipeline. It did not — the entry survives in
settings.json with `_version: "v1.0"`.

## Fix scope (v1.2.1)

### Required

1. **`mnemos install-end-hook`** (or `install-hook --v1`) must remove
   the legacy `mnemos-auto-refine` SessionStart entry on install. Add
   one-shot migration logic + a unit test that asserts post-install
   only `mnemos-recall-briefing` remains in SessionStart.
2. **`recall_briefing --catchup`** is the ONLY SessionStart path that
   may refine. `auto_refine_hook` should either become a no-op shim
   that points users to `install-end-hook --v1` (and exits 0), or be
   deleted entirely.
3. **Ledger atomic claim:** wrap the "is-this-pending-or-claimed-by-
   another-worker?" decision in a file lock. On Windows use
   `msvcrt.locking`; on POSIX use `fcntl.flock`. The lock guards a
   read-modify-write of the ledger entry: a worker that sees the
   JSONL as unprocessed claims it by writing a sentinel
   `<path>\tCLAIMED\t<pid>` line, then refines, then rewrites
   `OK\t<slug>` over the claim. Concurrent workers see the
   `CLAIMED` line and bail out.
4. **Ledger corruption recovery:** add a one-shot
   `mnemos refine-ledger --normalize` CLI that reads the TSV,
   reconstructs valid lines (heuristics: 3-column TAB split, drop
   lines with non-existent paths, dedup duplicate paths keeping the
   most recent OK entry), and writes back atomically. Run this once
   after install-hook upgrades.

### Recommended (defense in depth)

5. Pre-refine duplicate guard: before invoking `claude --print`,
   re-check the `Sessions/` directory for `<date>-<slug-prefix>*.md`
   files matching the JSONL's date + cwd hint. If any exist, skip
   (idempotent). This protects against future regressions where the
   ledger lock is bypassed.

### Tests

- `test_install_end_hook_removes_legacy_auto_refine_entry` — set up
  settings.json with both v1.0 auto-refine and v1.1 session-end;
  run `install-end-hook`; assert auto-refine is gone.
- `test_concurrent_refine_workers_dedup_via_lock` — spawn two
  threads/processes that both call the refine entry point on the
  same JSONL; assert exactly ONE Session file is written and exactly
  ONE OK line appears in the ledger.
- `test_refine_ledger_normalize_repairs_corrupted_lines` — feed a
  TSV with concatenated paths, missing TABs, duplicate OK lines;
  assert post-normalize is valid 3-column TSV with no duplicates.

## Status of v1.2.0 vis-à-vis this bug

This race **predates v1.2.0** — evidence shows it affecting JSONLs
refined this morning during the v1.2.0 implementation itself,
before any of the v1.2.0 prompt changes landed. v1.2.0 is purely a
locale-aware output schema change; it neither introduced nor depends
on the refine-pipeline concurrency model.

**Decision (2026-04-28):** Ship v1.2.0 as-is with this bug documented
as a known issue. Defer the fix to v1.2.1 to keep release scope
crisp and avoid coupling a non-trivial concurrency fix to a
documentation-strings release.

## User-facing workaround (until v1.2.1)

When `<vault>/Sessions/` contains two `<date>-<slug>.md` files for
the same conversation:

1. Compare both files; pick the more complete one.
2. Move the loser to `<vault>/_recycled/` (the watcher will clean
   the index).
3. Optionally append a one-line note to the keeper's `## See Also`
   section explaining the dedup decision.

To stop the race from recurring before v1.2.1 ships, manually edit
`~/.claude/settings.json` and remove the `mnemos-auto-refine`
SessionStart hook entry. Keep only `mnemos-recall-briefing` and
`mnemos-session-end`.
