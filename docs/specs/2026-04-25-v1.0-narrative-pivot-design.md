**Date:** 2026-04-25
**Status:** Draft (post-brainstorm, pre-plan)
**Previous version:** v0.4 AI Boost (left half-done, not shipping)
**Target version:** v1.0.0-alpha → v1.0.0
**README draft:** [`docs/v1.0-readme-draft.md`](../v1.0-readme-draft.md)

---

## 1. Pivot Summary

The v0.x series of Mnemos was built on the **atomic-fragmentation paradigm** —
conversations were converted into refined Sessions/.md files and additionally
broken down by an LLM into smaller "drawer" fragments. Drawers were stored in a
`<vault>/Mnemos/wings/<wing>/<room>/<hall>/` hierarchy classified by hall
(decisions / problems / preferences / events), and the vector index searched
both raw (Sessions) and mined (drawer) collections via RRF.

Measurements falsified this paradigm:

1. **Vector recall converges on the fragments.** With k=60, the RRF score
   distribution clusters in the 0.014–0.017 band; the "smaller chunk → sharper
   embedding" hypothesis does not hold on conversational data.
2. **LLM synthesis is better fed by narrative.** Briefing quality is sharper
   when full Session notes are provided, and fuzzier when fragment atoms are
   provided.
3. **A 600-node graph is unreadable.** The user does not scan it manually, and
   the AI does not traverse it. The "palace" became a visual poster, not a
   functional tool.
4. **Drawer entity hygiene is weak.** 663 drawers carry roughly ~860 entity
   instances total, ~50% of which come solely from the wing name (`Mnemos`,
   `ProcureTrack`, `GYP`) — redundant. Net valuable unique entities: ~40-60.

v1.0 abandons this paradigm with a **clean break**:

- The mining pipeline is removed entirely (from the codebase)
- Drawer files remain in place inside the user's vault (decision D,
  non-destructive)
- Sessions become the sole memory unit
- Refine prompt v2 injects a tag + wikilink hybrid into every Session
- A new **Identity Layer** (`<vault>/_identity/L0-identity.md`) holds a
  persistent user profile, which the briefing reads as its base layer
- The briefing skill becomes 3-layered (Identity + cwd Sessions + 1-hop
  wikilink expansion)
- The recall skill becomes flat (Sessions search + grep rescue, no graph
  expansion)
- The MCP tool surface is simplified; `mnemos_graph` and `mnemos_timeline` are
  reimplemented over the Obsidian wikilink graph (signature unchanged, backend
  changed)

## 2. Pivot Philosophy

**From the atomic-fragmentation paradigm to a narrative-first paradigm.**

MemPalace (42K-star), mem0, most LangChain memory backends — all share the
hypothesis "split conversations into small pieces, vector-index them, retrieve
top-K." This hypothesis works for transactional memory (user preference lists,
fact extraction from documents). It breaks for conversational session memory:
the chronology of decisions is carried across paragraphs, and once you split it
into fragments the cause-and-effect link is severed.

**Mnemos's counter-bet:** large unit (the whole Session), sparse but meaningful
links (entity wikilinks + tags), and an **Identity Layer** that accumulates
over time. The result: an AI that does not "remember conversations" but
instead "knows the user" — this is exactly what sets Mnemos apart from
MemPalace/mem0.

**Identity Layer rationale.** Each Session is a momentary snapshot; on its own
it cannot answer "who is Tugra?" Once an LLM reads all 78 Sessions in one shot
and produces a structural summary (working style, technical preferences,
anti-patterns, recurring projects, people in orbit, revised decisions), that
summary lives in a permanent markdown file. As new Sessions arrive it is
**incrementally updated** — not rewritten, only the changed fields receive a
delta injection. On every startup the briefing reads this profile as the base
layer and stacks cwd-specific real-time context on top.

**Why a clean break.** We chose to delete the old mining pipeline code rather
than lay down a soft deprecation. Reasons: (1) v1.0 is a major version,
breaking changes are expected; (2) the `legacy/atomic-paradigm` branch is
preserved as a permanent fallback; (3) for a new user, two paradigms living
side-by-side would be confusing to read; (4) narrative consistency — a pivot
should be a pivot, not a "half renovation."

## 5. Identity Layer

the vault holds the persistent user profile. The briefing skill reads it as
its base layer.

next_refresh_at: 88 sessions (or after 7 days, whichever first)
schema_version: 1
---

### 5.2 Bootstrap mechanics

The initial creation is expensive (one-shot). Command:

```bash
mnemos identity bootstrap
```

Flow:
1. Read every Sessions/.md in the vault, sorted date desc
2. Compute the token budget (assuming ~1.5K per Session)
3. **Context window cap (critical):**
   - If total input ≤ **150K tokens**: all Sessions are sent to the LLM
   - If > 150K: take the most recent 100 Sessions + sample from earlier
     Sessions (first 5 + every 10th one) → a "baseline + recent" hybrid
   - This cap leaves a safety margin against Sonnet's 200K context window
     (input + prompt + output total ≤ 180K)
   - If the vault has fewer than 100 Sessions there is no issue; at 100+ the
     hybrid kicks in
4. Forward to the canonical bootstrap prompt via `claude --print --model
   sonnet` or `--model opus` (user choice)
5. The LLM emits structured markdown (matching the template above)
6. Write to `<vault>/_identity/L0-identity.md`
7. Store a snapshot at `<vault>/_identity/_history/<date>-bootstrap.md`

Duration: ~5-10 minutes. Cost: within subscription quota (claude --print).

**Chunked bootstrap (v1.1 target, not in v1.0):**
If the vault reaches 200+ Sessions, even the hybrid cap may be insufficient.
The solution: chunked bootstrap — first 50 Sessions → draft profile, next 50 +
draft → refined, and so on. This is YAGNI in v1.0 because the current kasamd
holds 78 Sessions and most users will be under 100. It will be added in v1.1
if a user with 200+ Sessions emerges.

Canonical bootstrap prompt: a new file `docs/prompts/identity-bootstrap.md`.
Prompt structure:
- Role definition (project historian)
- Input (contents of the Sessions folder or hash list)
- Output structure (the 7 sections above)
- Quality control checklist (is the frontmatter valid, were conflicts
  detected, etc.)
- Size limits (max 8 items per section, max 15 in "Revised decisions")
- Cap note: "if input exceeds 150K, use the baseline+recent hybrid"

### 5.3 Incremental refresh

A low-cost update. Command:

```bash
mnemos identity refresh
```

Flow:
1. Read `last_refreshed` and `session_count_at_refresh` from the existing
   frontmatter of `_identity/L0-identity.md`
2. Build a list of Sessions/.md files written after that date (~5-10 new ones)
3. Write a pre-refresh **backup snapshot** (§5.7)
4. Send the current profile + new Sessions to the LLM
5. The prompt steers the LLM with 4 rules:
   - "If new information conflicts with an existing **(general)** preference,
     decide explicitly whether this is project-specific (a new
     `(proj/<name>)` line) or a **genuine general preference change** (move
     the old line into Revised decisions)"
   - "If new information reinforces an existing general, leave it alone"
   - "If new information introduces a new entity (person, tool, project),
     don't hesitate to add it"
   - "Don't touch what's still consistent — no unnecessary revisions"
6. The LLM returns the updated profile
7. Atomic write (`.tmp` + `os.replace`)
8. Frontmatter is updated (`last_refreshed`, `session_count_at_refresh`,
   `next_refresh_at`)

Duration: ~1-2 minutes. Cost: ~20K tokens.

**Scope distinction is critical.** The statement "I prefer SQLite" is
ambiguous on its own — project-specific or general? The refresh prompt forces
the LLM to make this judgment explicitly; otherwise the risk of identity
contamination is high (just because Mnemos uses sqlite-vec does not mean
"Tugra prefers SQLite in every project"). The `(general)` vs. `(proj/<name>)`
notation makes this distinction permanently explicit on disk.

### 5.4 Trigger strategy

| Version | Trigger |
|---|---|
| v1.0-alpha | **Manual** — the user calls `mnemos identity refresh`. Fires once at install during bootstrap. |
| v1.0 stable | **Manual + optional hook.** Conditionally triggered by the SessionStart hook if `auto_identity_refresh: true` is set in the config (see below). |
| v1.1 | **Automatic + conflict UI.** If a conflict is detected during refresh, the user is asked and manually confirms. |

**Auto-refresh trigger condition (v1.0 stable, with the optional hook on):**

Both conditions must hold together, otherwise the refresh does not fire:

1. **Quantity:** **≥10 new Sessions** have been written since the last refresh
2. **Relevance:** at least one of these new Sessions contains an
   **identity-relevant tag** — an entity like `proj/<new>`, `person/<new>`,
   `tool/<new>` that wasn't in the profile at the last refresh. If only
   recurring tags appear (e.g. `proj/mnemos` in every Session), the refresh
   is skipped.

This way no API call is made on every session, and the same profile is not
regenerated on every Session. It only fires when there is a "new information"
signal.

Trigger evaluation can also be checked manually (dry-run) via `mnemos
identity refresh --check`.

### 5.5 Conflict handling (3-case scenario)

During refresh the LLM can detect three different conflict types:

**Case 1 — Project-specific addition (no conflict):**
A new Session mentions "we're using sqlite-vec for Mnemos." The existing
profile has `(general) prefers Postgres`.
→ The general preference stays; a new line `(proj/mnemos) sqlite-vec backend`
is added. **No** conflict, because the scopes differ.

**Case 2 — Genuine general shift:**
Across the last 5 Sessions Tugra has explicitly said "I'm now writing
TypeScript in every project." The existing profile has `(general) Python >
other languages`.
→ The old general line moves to `Revised decisions` with a rationale, and a
new general line `(general) TypeScript > Python` is added. There **is** a
conflict, and the resolution is clear.

**Case 3 — Project override of general:**
The existing line is `(general) Markdown-native > opaque binary`. A new
Session says "we're moving the ProcureTrack database to Postgres with a JSON
binary blob."
→ The general preference stays; a new line `(proj/procuretrack) JSON binary
OK` is added. The override means "an exception in this project"; it does not
falsify the general.

The LLM must place every conflict into one of these three cases — the refresh
prompt emphasizes this rule explicitly. An "ambiguous" output is not accepted.

In v1.1, when Case 2 (genuine shift) is detected, user confirmation will
become mandatory; in v1.0 the LLM judgment is trusted, but thanks to the
pre-refresh backup snapshot (§5.7) every refresh remains reversible.

### 5.6 File size control

The profile must not bloat. Per-section limits:

| Section | Max items |
|---|---|
| Working style | 8 |
| Technical preferences | 12 |
| Rejected approaches | 10 (oldest + least used dropped first) |
| Active projects | 8 |
| People in orbit | 12 |
| Tools mastered | 15 |
| Revised decisions | 15 (oldest dropped) |

The total file is at most ~3K words. It enters the briefing as ~5K tokens.

### 5.7 Backup mechanism

A snapshot **before** every refresh is mandatory, for rollback:

**Decision logic:**
1. Is the vault git-tracked (check with `git rev-parse --git-dir`)? (i.e. the
   user is syncing kasamd via git rather than Obsidian Sync or OneDrive)
2. If git-tracked: an automatic commit is made before refresh — `git add
   _identity/L0-identity.md && git commit -m "mnemos identity refresh
   checkpoint <date>"`. Rollback is via `git revert HEAD` or `git checkout
   HEAD~1 _identity/L0-identity.md`
3. If not git-tracked: a pre-refresh copy
   `_identity/L0-identity.md.bak-YYYY-MM-DD-HHmm` is created. The most recent
   5 .bak files are kept; older ones are deleted (rolling window). Rollback
   is via manual copy or the `mnemos identity rollback <date>` command.

The `mnemos identity rollback [<date>]` command (new):
- Without arguments: dumps the latest .bak and asks for `mv` confirmation
- With a date: `mnemos identity rollback 2026-04-25-1430` reverts to that
  specific snapshot
- In git-tracked vaults it suggests `git revert` or `git checkout`, but does
  not commit itself (so it does not interfere with the user's git workflow)

This mechanism makes the identity contamination risk no longer "irreversible"
— its impact on the risk matrix (§15) drops.

## 6. Refine Prompt v2

`docs/prompts/refine-transcripts.md` is updated. It injects the tag + wikilink
hybrid ("Path 2").

### 6.1 Frontmatter change

Old:
```yaml
tags: [session-log, mnemos, v0.4, recall, skill]
```

New:
```yaml
tags: [session-log, proj/mnemos, tool/supabase, tool/outlook, person/tugra, file/test_orchestrator, skill/po-olusturma]
```

Tag-prefix category dictionary:
- `proj/<name>` — project (wing equivalent): `proj/mnemos`, `proj/procuretrack`, `proj/gyp`
- `tool/<name>` — tool/service: `tool/supabase`, `tool/outlook`, `tool/chromadb`, `tool/claude-code`
- `person/<name>` — person: `person/tugra`, `person/safa-clutch`
- `file/<name>` — source file (only when recurring, appears in ≥2 Sessions): `file/test_orchestrator`
- `skill/<name>` — skill / CLI command: `skill/po-olusturma`, `skill/mnemos-catch-up`

The `session-log` tag remains on every Session (for type filtering).

### 6.2 Wikilinks in prose

The refine prompt tells the LLM explicitly: wrap entities mentioned in prose
as wikilinks **at the first occurrence**.

**Becomes a wikilink:**
- Project/wing names (Mnemos, ProcureTrack, GYP)
- Consistent tool/service names (Supabase, Outlook, ChromaDB, Claude Code)
- Skill / CLI command names (po-olusturma, mnemos-catch-up)
- Person/company names (Tugra, Safa Clutch, GYP Energy)
- Recurring concept hubs (auto-refine-hook, sqlite-vec)

**Does not become a wikilink:**
- Single-use file names (test_orchestrator.py)
- Generic technology (Python, JSON, Excel)
- Any name inside a code block
- Version numbers (3.12, v0.4.0)
- Dates / numbers

The judgment rule for the LLM: "Does this entity appear in any other
Session?" If yes (recurring hub) → wikilink. If no (single-use) → plain text.

### 6.3 Example

**Current output:**
> Today I fixed a bug in the Supabase RLS policy for ProcureTrack.
> Discussed it with Tugra, and the resolution moved to the Outlook integration.
> Added 3 new tests in test_orchestrator.py.

**v2 output:**
> Today I fixed a bug in the [[Supabase]] RLS policy for [[ProcureTrack]].
> Discussed it with [[Tugra]], and the resolution moved to the [[Outlook]] integration.
> Added 3 new tests in test_orchestrator.py.

Frontmatter:
```yaml
tags: [session-log, proj/procuretrack, tool/supabase, tool/outlook, person/tugra]
```

### 6.4 Refine skill quality

A post-write frontmatter check is added to the skill:
- At least 1 `proj/*` tag must be present
- At least 1 wikilink in the prose
- Tag-prose consistency: if `tool/supabase` is in the tags, `[[Supabase]]`
  must appear in the prose

This "Quality Control checklist" is appended to the end of the refine prompt.

## 7. Briefing Skill v2

`skills/mnemos-briefing/` SKILL.md and prompt.md are refreshed. It produces a
**3-layer** synthesis.

### 7.1 Flow

```
1. Read <vault>/_identity/L0-identity.md → Identity Layer (3K cap)
2. Read <vault>/Sessions/*.md filtered by frontmatter cwd: == input cwd
   sorted by date desc → Cwd Layer (priority 1)
3. Extract wikilinks from Cwd Layer Sessions → list of [[X]] entities
4. For each [[X]]: glob <vault>/Sessions/*.md grep "[[X]]" → backlink Sessions
   (sorted by date desc + Summary keyword overlap with cwd; dedup; exclude
   already in Cwd Layer) → Cross-context Layer (priority 2)
5. Priority-based token assembly (hard cap 15K):
   - Identity Layer: 3K (fixed; the §5.6 section limits already produce this)
   - Cwd Layer: 8K (priority 1, full body of the last N Sessions — N is filled
     up to 8K consumed, max 5)
   - Cross-context Layer: 4K (priority 2, sorted by recency+relevance hybrid;
     pull Sessions until the budget is filled; Summary + Decisions sections only)
6. Synthesize: 3-section briefing (output 400-600 words, ~1K tokens)
```

**Ranking rule (for Cross-context):**

For each wikilink entity, backlink Sessions are collected and scored as:
- Date desc (newest first) — recency weight 0.6
- Summary section keyword overlap with the Cwd Layer's combined Summary —
  relevance weight 0.4
- Score = 0.6 × normalize(date) + 0.4 × overlap_count

The highest-scoring items are picked until the total Cross-context budget
(4K) is filled. In cwd-Session-heavy situations (e.g. 15+ cwd Sessions) the
Cwd Layer fully consumes 8K, and Cross-context can drop to 0.

### 7.2 Output structure

```markdown
**Active status:** <1-2 sentences, cwd-local + identity-aware>

**User profile (important):**
- Working style / technical-preference notes (2-3 selected items from
  identity, the ones most relevant to this cwd)

**Decisions in effect:**
- <date> — <decision> [[wikilink]]
- (cwd-local + at least 1 item from cross-context)

**Revised/cancelled decisions:**
- <old> → <new>. Rationale: <brief>
- (revised decisions in cwd + identity; skip the section if there are no
  conflicts)

**Open issues:**
- <unresolved problem or TODO> [[wikilink]]

**Up next:**
- <likely next task>

**Related (cross-context):**
- [[Project X]] — <where a similar pattern to this cwd appeared, brief>
- (Sessions from entity hubs, max 3 items)
```

### 7.3 Token budget (priority-based, hard cap 15K)

Fixed total budget — priority-based sharing across layers:

| Layer | Budget | Priority | Behavior |
|---|---|---|---|
| Identity Layer | 3K (fixed) | — | Always full read; the §5.6 limits already guarantee this |
| Cwd Layer | 8K | 1 | Full body of the last N Sessions until 8K is filled; max N=5 |
| Cross-context Layer | 4K | 2 | Sorted by recency+relevance hybrid; Summary+Decisions sections until budget is filled |
| **Total input** | **15K hard cap** | — | Output ~1K (400-600 words) |

**Edge case — cwd-Session-rich environment:**
If the cwd has 10+ Sessions, the Cwd Layer can fully fill 8K. In that case
the Cross-context Layer budget drops to 0 and the "Related (cross-context)"
section is dropped from the briefing. This is **acceptable** because the cwd
context is already dense and expansion adds marginal value. The briefing is
still useful — it just has 2 headers instead of 3.

**Edge case — cwd-Session-poor environment:**
If the cwd has 1-2 Sessions, the Cwd Layer uses 2-3K, leaving the remaining
5-6K to Cross-context. In a new cwd, building a bridge between Identity and
cross-context is emphasized.

**Expected output size:**
The 3-layer briefing is 400-600 words, ~1K tokens. It's larger than the
v0.x's 200-400-word briefing, but because input drops 35K → 15K, **net token
consumption is lower**.

### 7.4 Subprocess invocation

Called from the hook:
```bash
claude --print --dangerously-skip-permissions --model sonnet \
  /mnemos-briefing "<absolute cwd path>"
```

Output to stdout; the hook wrapper writes it to
`<vault>/.mnemos-briefings/<cwd-slug>.md`.

## 8. Recall Skill v2

`skills/mnemos-recall/` SKILL.md is updated. **Flat Sessions search + grep
rescue.** No graph expansion (v1.1 target).

### 8.1 Flow

The only change from the current flow: no drawer search, only Sessions search.

```
1. mnemos_search(query, limit=8, collection="raw")  # mined collection removed
2. Top-3 score evaluate against threshold 0.015 (k=60 RRF)
3. If pass: read top 5 Sessions, synthesize narrative answer
4. If fail: Sessions grep rescue (Step 7 was added in v0.4.1; default path in v1.0)
   - Extract 2-4 keywords from query
   - Glob <vault>/Sessions/ + Grep keywords
   - Top 3 hit Sessions read, synthesize
5. Soft fallback: list top 3 Sessions with scores + snippets, no synthesis
```

### 8.2 No subprocess

Recall runs in-session inline (the current design). MCP `mnemos_search` →
read → synthesize. There is no `claude --print` subprocess call.

### 8.3 Coverage drop

In v0.x, recall searched both the mined collection (drawers) and the raw
collection (Sessions). In v1.0 it only searches raw. **Is this a quality
regression?** No, because:
- Smoke measurements (4.3.1) already showed drawer search was working poorly
  (RRF cluster 0.014-0.017)
- Sessions grep rescue is an already-working path; in v1.0 it's the default
- Sessions content-richness is higher than drawers (narrative integrity)

### 8.4 Threshold calibration note

The `0.015` threshold was calibrated on the kasamd vault (sqlite-vec backend,
~78 Sessions) with k=60 RRF scoring. **It may behave differently in a new
user's vault** because:
- ChromaDB vs. sqlite-vec score distributions can differ
- If the vault has few Sessions (e.g. 10), RRF rank-based scoring spreads over
  a wider range
- An embedding-model change (in the future) shifts the absolute values of
  scores

In v1.0 the threshold is **hardcoded** (`0.015`) — exposing calibration to the
user is YAGNI. Two safety nets exist in practice:
1. The threshold is already a **soft check** — if the score is low, the
   Sessions grep rescue path takes over and can produce a valuable answer
2. The soft fallback (Step 8) returns a scored list if no path works

In v1.1, if a real user vault produces a smoke complaint of "every query
falls through to rescue," the threshold will be added to `mnemos.yaml` as a
`recall_threshold` field and become configurable. So far the need has not
been observed.

#### `mnemos reindex` behavior

```bash
mnemos reindex [--backend chromadb|sqlite-vec] [--no-backup]
```

Flow:
1. Read the `search_backend:` field from `mnemos.yaml` (if present, the `--backend` flag overrides it and writes back to yaml)
2. Back up the current index storage to `<vault>/Mnemos/<storage>.bak-YYYY-MM-DD-HHmm` (skippable with `--no-backup`)
3. Initialize the new backend (empty)
4. List `<vault>/Sessions/*.md`, bulk-add each Session to the index with a deduped id
5. Verify: does the index document count match the Sessions count?
6. Delete the old backend's storage (the backup is already in `.bak`)

Typical flow for a backend switch:
```bash
# 1. mnemos.yaml: search_backend: chromadb → sqlite-vec
# 2. mnemos reindex
# To skip manual yaml editing:
mnemos reindex --backend sqlite-vec  # yaml is updated automatically
```

For index-corruption recovery, the same command (no `--backend` flag):
```bash
mnemos reindex
```

Duration: 78 Sessions × ~500ms = ~40 seconds (kasamd measurement).

### 12.2 Idempotent behavior of `mnemos install-hook --v1`

**Critical:** Removal and writing are a **single atomic operation**. While
old entries are removed and new entries are written, the file must never even
momentarily be in a state where "two hooks are registered side-by-side"
(double-fire risk under race conditions).

```python
def install_hook_v1():
    settings_path = user_settings_path()  # ~/.claude/settings.json
    settings = read_user_settings(settings_path)
    hooks = settings.get("hooks", {}).get("SessionStart", [])

    # In-memory modify — settings.json is still in its old state, file untouched
    hooks = [
        h for h in hooks
        if h.get("_managed_by") not in
            ("mnemos-auto-refine", "mnemos-recall-briefing")
    ]

    hooks.append({
        "matcher": "*",
        "_managed_by": "mnemos-auto-refine",
        "_version": "v1.0",
        "hooks": [{
            "type": "command",
            "command": "python -m mnemos.auto_refine_hook \"<vault>\"",
            "timeout": 30000
        }]
    })
    hooks.append({
        "matcher": "*",
        "_managed_by": "mnemos-recall-briefing",
        "_version": "v1.0",
        "hooks": [{
            "type": "command",
            "command": "python -m mnemos.recall_briefing \"<vault>\"",
            "timeout": 600000
        }]
    })

    settings["hooks"]["SessionStart"] = hooks

    # Atomic write: tmp + os.replace
    tmp = settings_path.with_suffix(".tmp")
    tmp.write_text(json.dumps(settings, indent=2))
    os.replace(tmp, settings_path)  # POSIX/NTFS atomic rename
```

Idempotent: re-running it removes the old v1.0 entries and writes the same
new ones. The net effect doesn't change. A **single file write operation** —
the period between old entries being removed and new entries being added is
invisible on disk.

### 12.3 Stale hook graceful failure

**Scenario:** the user upgraded to v1.0 with `pip install
mnemos-dev==1.0.0a1` but did not run `mnemos install-hook --v1`. The old
hook entry is still in `~/.claude/settings.json`, the old command `python -m
mnemos.auto_refine_hook` imports the v1.0 package. But `auto_refine.py` no
longer contains a `_run_skill_pipeline` function.

**v0.x behavior:** ImportError → SessionStart hook crash → "hook failed"
stderr at Claude Code session start → frightening UX.

**v1.0 graceful failure:**

Add a shim at the entry point of `mnemos/auto_refine_hook.py` and
`mnemos/recall_briefing.py`:

```python
# mnemos/auto_refine_hook.py top-level
import sys

def _detect_stale_hook_signature():
    """Returns True if old semantics not present in the v1.0+ package are required."""
    # What was the v0.x hook calling? The old entry signature: 1 argument (vault).
    # The new v1.0 hook signature: same (1 argument vault). So the signature
    # is not broken — but there is no "_managed_by" version check inside the
    # hook command body, so a stale entry still calls the v1.0 binary.
    # To detect stale: if the _version of the hook entry in settings.json is
    # not "v1.0" (either empty, missing, or v0.x), it is stale.
    settings = read_user_settings()
    for entry in settings.get("hooks", {}).get("SessionStart", []):
        if entry.get("_managed_by") == "mnemos-auto-refine":
            if entry.get("_version") != "v1.0":
                return True
    return False

def main():
    if _detect_stale_hook_signature():
        print(
            "Mnemos v1.0 detected an outdated SessionStart hook entry.\n"
            "Run `mnemos install-hook --v1` to update.\n"
            "Skipping this run to avoid errors.",
            file=sys.stderr,
        )
        sys.exit(0)
    # ... normal v1.0 flow
```

The same shim is also added to `recall_briefing.py`.

**User experience:** instead of an ImportError explosion, a clear message.
The session start is not broken (exit 0); it just states that the hook did
not run. The user sees the message → runs the command → in the next session
the hook runs with v1.0 behavior.

- Atomic-fragmentation paradigm ("drawer" mining) removed. Mnemos no longer
  produces or indexes drawer fragments.
- CLI: `mnemos mine`, `mnemos catch-up`, `mnemos migrate`, `mnemos import
  claude-code`, `mnemos processing-log` removed.
- MCP: `mnemos_mine` tool removed. `mnemos_search` no longer accepts
  `collection="mined"` (warns and falls back to `raw`).
- Wing/Hall/Room hierarchy retired. `mnemos_recall(level="L1"|"L2")` no
  longer returns useful data.

### Added
- **Identity Layer** (`<vault>/_identity/L0-identity.md`) — persistent user
  profile, bootstrapped from existing Sessions, evolves via incremental
  refresh.
- `mnemos identity bootstrap`, `refresh`, `show` CLI commands.
- Briefing skill now produces 3-layer narrative (Identity + cwd + cross-
  context wikilink expansion).
- Refine prompt v2 produces tag + wikilink hybrid (`tags: [proj/x,
  tool/y, person/z]` + prose `[[X]]`).

### Refactored
- `mnemos_graph(entity)` and `mnemos_timeline(entity)` now query Obsidian
  wikilink graph instead of SQLite triple store. Tool signatures unchanged;
  return dicts unchanged.

### Removed code
- Mining pipeline (8 modules ~3K LOC).
- ~200 drawer-related tests.
- `mnemos-mine-llm` and `mnemos-compare-palaces` skills.

### Migration
- Existing drawer files in `<vault>/Mnemos/wings/` are preserved untouched.
  Mnemos no longer reads from them. Users may delete manually if desired.
- Run `mnemos identity bootstrap` after upgrade to populate the new
  Identity Layer (~5–10 minutes).
- Existing `mnemos.yaml` `mine_mode` field is ignored; can be removed.
- Run `mnemos install-hook --v1` to update SessionStart hooks.

### Pre-v1.0 paradigm preservation
- The full v0.x atomic-paradigm codebase is preserved at branch
  `legacy/atomic-paradigm` and tag `v0.4.0-archived`. Users wishing to
  continue with the drawer model can pin `pip install mnemos-dev==0.3.3`.

Blog post main structure:
1. MemPalace inspiration + 4 months of testing

4. v1.0 demo (kasamd examples)

**Broad-contribution framing (important):** the blog post should not just
introduce Mnemos — the finding "why the atomic-fragmentation paradigm does
not work for conversational memory" is a valid insight for the **AI memory
field** as a whole, including MemPalace, mem0, and LangChain memory. The
blog title should emphasize this broader insight (something like "Why
fragmentation hurts conversational AI memory"), and present Mnemos's pivot
as evidence for the insight. This positioning:
- Lifts Mnemos out of being "MemPalace's competitor" and creates **its own
  category** (Identity-aware narrative memory)
- Contributes to the industry conversation rather than just announcing one
  tool
- Increases distribution potential on platforms like HN / Reddit /
  r/LocalLLaMA

1. **`mnemos import chatgpt|slack|markdown|memory`** — do these commands
   stay in v1.0? They could be integrated with the refine pipeline but they
   were tied to mining. Test coverage will be inspected during planning and
   decided.
2. **LongMemEval baseline** — does the existing benchmark still run before
   the v1.0a1 ship? If there are drawer-dependent parts, the harness will
   be updated.
3. **`raw` collection backend compatibility** — do both ChromaDB and
   sqlite-vec stay in v1.0, or does the Sessions-only paradigm consolidate
   on one? YAGNI: both stay.
4. **Model choice for identity bootstrap** — sonnet vs. opus default? A
   token/cost comparison will be done in the plan.
5. **`mnemos init` Phase 5 "hook activation" flow** — should
   install-recall-hook in v1.0 prompt before or after identity bootstrap?

These questions are answered during plan writing; they do not affect the
core of the design spec.
