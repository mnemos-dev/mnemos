# Mnemos v1.0 — Narrative-First Pivot · Design Spec

**Tarih:** 2026-04-25
**Durum:** Draft (brainstorming sonrası, plan yazımı öncesi)
**Önceki sürüm:** v0.4 AI Boost (yarım kaldı, ship edilmiyor)
**Hedef sürüm:** v1.0.0-alpha → v1.0.0
**README draft:** [`docs/v1.0-readme-draft.md`](../v1.0-readme-draft.md)

## 1. Pivot Özeti

Mnemos'un v0.x serisi **atomic-fragmentation paradigm**'ı üzerine kuruluydu —
sohbetler refined Sessions/.md'lere dönüştürülüp ayrıca LLM ile daha küçük
"drawer" fragmanlarına ayrıştırılıyor, drawer'lar hall (decisions / problems /
preferences / events) sınıflandırmasıyla `<vault>/Mnemos/wings/<wing>/<room>/<hall>/`
hiyerarşisinde saklanıyor, vector index hem raw (Sessions) hem mined (drawer)
collection'ları üzerinde RRF ile arıyor.

Ölçümler bu paradigmayı çürüttü:

1. **Vector recall fragmanlar üzerinde converge ediyor.** RRF skor dağılımı
   k=60 ile 0.014–0.017 bandına çakılıyor; "küçük chunk → keskin embedding"
   hipotezi conversational data üzerinde gerçekleşmiyor.
2. **LLM sentezi narrative'den iyi besleniyor.** Briefing kalitesi tam Session
   notlarını verince daha keskin, fragment atomlarını verince daha bulanık.
3. **600-node graph okunaksız.** Kullanıcı manuel taramıyor, AI traversal
   yapmıyor. "Palace" görsel poster oldu, fonksiyonel araç değil.
4. **Drawer entity hijyeni zayıf.** 663 drawer toplam ~860 entity instance
   taşıyor, ~%50'si yalnızca wing adından (`Mnemos`, `ProcureTrack`, `GYP`) —
   redundant. Net değerli unique entity ~40-60.

v1.0 bu paradigmayı **clean break** ile bırakır:

- Mining pipeline tamamen silinir (kod tabanından)
- Drawer dosyaları kullanıcı vault'unda yerinde kalır (D kararı, non-destructive)
- Sessions tek memory unit olur
- Refine prompt v2 her Session'a tag + wikilink hibridi enjekte eder
- Yeni **Identity Layer** (`<vault>/_identity/L0-identity.md`) kalıcı user
  profile tutar, briefing onu base layer olarak okur
- Briefing skill 3-katmanlı (Identity + cwd Sessions + 1-hop wikilink genişleme)
- Recall skill düz (Sessions search + grep rescue, graph genişleme yok)
- MCP tool yüzeyi sadeleşir; `mnemos_graph` ve `mnemos_timeline` Obsidian
  wikilink ağı üzerine yeniden implement (signature aynı, backend değişti)

## 2. Pivot Felsefesi

**Atomic-fragmentation paradigm'dan narrative-first paradigm'a.**

MemPalace (42K-star), mem0, çoğu LangChain memory backend — hepsi
"konuşmaları küçük parçacıklara böl, vector index'le, top-K çek" hipotezini
paylaşır. Bu hipotez transactional memory için çalışır (kullanıcı tercihi
listesi, dokümanlardan fact extraction). Conversational session memory için
çürür: kararların kronolojisi paragraflar arasında taşınır, fragment'a
ayırınca neden-sonuç bağı kopar.

**Mnemos'un karşı bahsi:** büyük birim (whole Session), seyrek ama anlamlı
linkler (entity wikilink + tag), zaman üstüne biriken **Identity Layer**.
Sonuç: AI "konuşmaları hatırlayan" değil, "kullanıcıyı tanıyan" sistem
olur — Mnemos'un MemPalace/mem0'dan ayrılan asıl noktası budur.

**Identity Layer mantığı.** Her Session anlık bir snapshot; tek başına
"Tugra kim" sorusunu cevaplamaz. 78 Session'ı LLM'e bir kez okutup yapısal
özet çıkarınca (working style, technical preferences, anti-patterns,
recurring projects, people in orbit, revised decisions) bu özet kalıcı bir
markdown dosyasında yaşar. Yeni Session geldikçe **incremental update**
edilir — yeniden yazılmaz, sadece değişen alanlara delta inject edilir.
Briefing her açılışta bu profile'ı base layer olarak okur ve üstüne
cwd-spesifik anlık bağlam katmanlar.

**Clean break tercihi.** Soft deprecation döşemek yerine eski mining
pipeline kodunu silmek tercih edildi. Sebepler: (1) v1.0 majör sürüm,
breaking change beklenir; (2) `legacy/atomic-paradigm` branch'i geri
dönüş için kalıcı korunuyor; (3) yeni kullanıcı için iki paradigma yan
yana okuyabilir kafa karışıklığı; (4) anlatım tutarlılığı — pivot pivot
olur, "yarım yenileme" olmaz.

## 5. Identity Layer

v1.0'ın yeni asıl bileşeni. Vault'ta `_identity/L0-identity.md` dosyası
kalıcı user profile tutar. Briefing skill bunu base layer olarak okur.

### 5.1 Dosya yapısı

`<vault>/_identity/L0-identity.md`:

```markdown
---
generated_from: 78 sessions across 6 projects
last_refreshed: 2026-04-25
session_count_at_refresh: 78
next_refresh_at: 88 sessions (or after 7 days, whichever first)
schema_version: 1
---

# User Identity

## Çalışma stili
- Tip B: vault'u AI hafıza deposu olarak kullanır, manuel tarama yapmaz
- Türkçe konuşur, teknik terimler İngilizce
- Yapısal sorgu sorar, soyut cevap istemez
- "Devam edelim mi?" gibi onay isteyince hızlı karar verir

## Teknik tercihler (yürürlükte)
- Python > diğer diller (oyun yazılımı dahil)
- Minimal abstraction, üç-benzer-satır > erken refactor
- Backward compat hack'lerden kaçınır
- Markdown-native > opaque binary

## Reddedilen yaklaşımlar (anti-pattern)
- Atomic memory fragmentation (drawer paradigm) — 2026-04-25 pivot
- Mock testler — gerçek vault entegre testleri tercih
- "Memory palace" terminolojisi — narrative-first yaklaşımı seçti

## Aktif projeler
- [[Mnemos]] (memory system, primary focus)
- [[ProcureTrack]] (Supabase satın alma otomasyonu)
- [[GYP]] (Petrol Ofisi bayi ekosistemi)
- [[Farcry]] (kişisel oyun brainstorm)

## Yörüngedeki insanlar
- [[Tugra]] — projenin sahibi
- [[Safa Clutch]] — müşteri
- [[Enver]] — meslektaş

## Ustalaşmış araçlar
- [[Claude Code]], [[Obsidian]], [[Supabase]], [[Outlook]], [[ChromaDB]]

## Revize edilen kararlar (zaman ekseni)
- 2026-04-19 "skill-mine + drawer atomization" → 2026-04-25 "Sessions-only narrative". Gerekçe: drawer entity kalitesi düşük, recall sentezi narrative'den iyi besleniyor.
- 2026-04-13 "Mnemos LLM çağırmaz" → 2026-04-15 "claude --print kullanılır". Subscription fallback skill için zorunlu.
```

### 5.2 Bootstrap mekaniği

İlk yaratım pahalı (one-shot). Komut:

```bash
mnemos identity bootstrap
```

Akış:
1. Vault'taki tüm Sessions/.md'leri okur (bytes ya da chunked)
2. Token bütçesi hesaplar (78 Session × ~1.5K = ~120K input)
3. `claude --print --model sonnet` veya `--model opus` (kullanıcı seçimi) ile
   canonical bootstrap prompt'a iletir
4. LLM yapısal markdown çıkarır (yukarıdaki şablona uyar)
5. `<vault>/_identity/L0-identity.md`'ye yazar
6. `<vault>/_identity/_history/2026-04-25-bootstrap.md` snapshot saklar

Süre: ~5-10 dakika. Maliyet: subscription quota dahilinde (claude --print).

Canonical bootstrap prompt: yeni dosya `docs/prompts/identity-bootstrap.md`.
Prompt yapısı:
- Rol tanımı (project historian)
- Girdi (Sessions klasörünün içeriği veya hash listesi)
- Çıktı yapısı (yukarıdaki 7 section)
- Kalite kontrol checklist (frontmatter geçerli mi, çelişkiler tespit edildi mi, vd.)
- Boyut limiti (her section max 8 madde, "Revize edilen kararlar" max 15)

### 5.3 Incremental refresh

Düşük maliyetli güncelleme. Komut:

```bash
mnemos identity refresh
```

Akış:
1. `_identity/L0-identity.md` mevcut frontmatter'ından `last_refreshed`,
   `session_count_at_refresh` okunur
2. O tarihten sonra yazılan Sessions/.md'ler listesi (~5-10 yeni)
3. Mevcut profile + yeni Sessions LLM'e iletilir
4. Prompt: "şu Sessions'a göre profile'ı incremental güncelle. Yeni
   tercihler/projeler ekle. Çelişki varsa eski tercihi 'Revize edilen
   kararlar' bölümüne taşı. Tutarlı olanlara dokunma."
5. LLM güncellenmiş profile döner
6. Atomic write (`.tmp` + `os.replace`)
7. `_history/<date>-refresh.md` snapshot eklenir
8. Frontmatter güncellenir (`last_refreshed`, `session_count_at_refresh`,
   `next_refresh_at`)

Süre: ~1-2 dakika. Maliyet: ~20K token.

### 5.4 Tetikleme stratejisi

| Sürüm | Tetikleme |
|---|---|
| v1.0-alpha | **Manuel** — kullanıcı `mnemos identity refresh` çağırır. Bootstrap install sırasında bir kez fire eder. |
| v1.0 stable | **Manuel + opsiyonel hook.** SessionStart hook'u config'te `auto_identity_refresh: true` ise her N session'da bir background tetiklenir (default N=10). |
| v1.1 | **Otomatik + çelişki UI.** Refresh sırasında çelişki tespit edilirse kullanıcıya sorar, manuel onay alır. |

### 5.5 Çelişki yönetimi

Refresh sırasında LLM iki durum tespit edebilir:

1. **Net revize:** "Önceden Python, son 5 Session'da Rust'a geçti." → eski
   tercih `Reddedilen yaklaşımlar` veya `Revize edilen kararlar` bölümüne
   taşınır, yeni tercih `Teknik tercihler (yürürlükte)`'e eklenir.
2. **Belirsiz değişim:** "Belirli bir bağlamda Python, başka bir bağlamda Rust
   tercih ediliyor." → her iki tercih `Teknik tercihler` altında bağlamla
   beraber listelenir.

LLM yargısına bırakılır; v1.1'de otomatik tetik için kullanıcı onayı eklenir.

### 5.6 Dosya boyutu kontrolü

Profile şişmemeli. Section başına limitler:

| Section | Max madde |
|---|---|
| Çalışma stili | 8 |
| Teknik tercihler | 12 |
| Reddedilen yaklaşımlar | 10 (en eski + en az kullanılan ilk silinir) |
| Aktif projeler | 8 |
| Yörüngedeki insanlar | 12 |
| Ustalaşmış araçlar | 15 |
| Revize edilen kararlar | 15 (en eski silinir) |

Toplam dosya max ~3K kelime. Briefing'e ~5K token olarak girer.

## 6. Refine Prompt v2

`docs/prompts/refine-transcripts.md` güncellenir. Tag + wikilink hibrit
("Yol 2") enjekte eder.

### 6.1 Frontmatter değişikliği

Eski:
```yaml
tags: [session-log, mnemos, v0.4, recall, skill]
```

Yeni:
```yaml
tags: [session-log, proj/mnemos, tool/supabase, tool/outlook, person/tugra, file/test_orchestrator, skill/po-olusturma]
```

Tag prefix kategori sözlüğü:
- `proj/<name>` — proje (wing eşdeğeri): `proj/mnemos`, `proj/procuretrack`, `proj/gyp`
- `tool/<name>` — araç/servis: `tool/supabase`, `tool/outlook`, `tool/chromadb`, `tool/claude-code`
- `person/<name>` — kişi: `person/tugra`, `person/safa-clutch`
- `file/<name>` — kaynak dosya (sadece tekrar eden, ≥2 Session'da geçen): `file/test_orchestrator`
- `skill/<name>` — skill/CLI komut: `skill/po-olusturma`, `skill/mnemos-catch-up`

`session-log` etiketi her Session'da kalır (tip filter için).

### 6.2 Prose içinde wikilink

Refine prompt LLM'e açıkça söyler: prose'da geçen entity'leri **ilk geçtiği
yerde** wikilink olarak sar.

**Wikilink olur:**
- Proje/wing adları (Mnemos, ProcureTrack, GYP)
- Tutarlı araç/servis isimleri (Supabase, Outlook, ChromaDB, Claude Code)
- Skill/CLI komut isimleri (po-olusturma, mnemos-catch-up)
- İnsan/şirket isimleri (Tugra, Safa Clutch, GYP Energy)
- Tekrar eden konsept hub'lar (auto-refine-hook, sqlite-vec)

**Wikilink olmaz:**
- Tek-kullanımlık dosya isimleri (test_orchestrator.py)
- Generic teknoloji (Python, JSON, Excel)
- Kod blok içindeki tüm isimler
- Versiyon numaraları (3.12, v0.4.0)
- Tarih/sayı

LLM yargısı için kural: "Bu entity başka bir Session'da da geçer mi?" Eğer
evet (tekrar eden hub) → wikilink. Eğer hayır (tek-kullanım) → düz metin.

### 6.3 Örnek

**Mevcut çıktı:**
> Bugün ProcureTrack için Supabase RLS policy'sinde bir bug düzelttim.
> Tugra'yla görüştük, çözüm Outlook entegrasyonuna geçti.
> test_orchestrator.py dosyasında 3 yeni test eklendi.

**v2 çıktı:**
> Bugün [[ProcureTrack]] için [[Supabase]] RLS policy'sinde bir bug düzelttim.
> [[Tugra]]'yla görüştük, çözüm [[Outlook]] entegrasyonuna geçti.
> test_orchestrator.py dosyasında 3 yeni test eklendi.

Frontmatter:
```yaml
tags: [session-log, proj/procuretrack, tool/supabase, tool/outlook, person/tugra]
```

### 6.4 Refine skill kalitesi

Frontmatter post-write kontrolü skill'e eklenir:
- En az 1 `proj/*` tag'i olmalı
- En az 1 wikilink prose'da
- Tag-prose tutarlılığı: tag'de `tool/supabase` varsa prose'da `[[Supabase]]` olmalı

Bu "Quality Control checklist" refine prompt'unun sonuna eklenir.

## 7. Briefing Skill v2

`skills/mnemos-briefing/` SKILL.md ve prompt.md yenilenir. **3-katmanlı**
sentez yapar.

### 7.1 Akış

```
1. Read <vault>/_identity/L0-identity.md → Identity Layer
2. Read <vault>/Sessions/*.md filtered by frontmatter cwd: == input cwd
   → Cwd Layer (anlık, son 10 Session)
3. Extract wikilinks from Cwd Layer Sessions → list of [[X]] entities
4. For each [[X]]: glob <vault>/Sessions/*.md grep "[[X]]" → backlink Sessions
   (limit: 5 per entity, dedup, exclude already in Cwd Layer)
   → Cross-context Layer
5. Token budget assembly:
   - Identity Layer: full read (~5K)
   - Cwd Layer: last 10 Sessions full body (~20K)
   - Cross-context Layer: Özet + Alınan Kararlar sections only (~10K)
   - Budget total: ≤40K input
6. Synthesize: 3-section briefing
```

### 7.2 Çıktı yapısı

```markdown
**Aktif durum:** <1-2 cümle, cwd local + identity-aware>

**Kullanıcı profili (önemli):**
- Çalışma stili / teknik tercih notları (identity'den seçili 2-3 madde,
  bu cwd'ye en alakalı olanlar)

**Geçerli kararlar:**
- <tarih> — <karar> [[wikilink]]
- (cwd-local + cross-context'ten en az 1 madde)

**Revize/iptal edilen kararlar:**
- <eski> → <yeni>. Gerekçe: <kısa>
- (cwd + identity'deki revize kararlar; eğer çelişki yoksa bölümü atla)

**Açık meseleler:**
- <çözülmemiş sorun veya TODO> [[wikilink]]

**Sırada:**
- <olası sonraki iş>

**İlgili (cross-context):**
- [[Project X]] — <bu cwd'ye benzer pattern'in geçtiği yer, kısa>
- (entity hub'larından gelen Sessions, max 3 madde)
```

### 7.3 Token bütçesi tablosu

| Katman | Hedef token | Strateji eğer aşılırsa |
|---|---|---|
| Identity Layer | 5K | Section başına madde limiti uygulanır (zaten yapılıyor) |
| Cwd Layer | 20K | Son 10 Session yerine son 7; Özet+Kararlar+Sorunlar tut, Yapılanlar drop |
| Cross-context Layer | 10K | Entity başına 5 yerine 3 backlink Session |
| **Toplam** | **35K** | Çıktı 800-1200 token (200-400 kelime) |

Çıktı boyutu artar (3 katman + cross-context). 200-400 kelimeden 400-600
kelimeye yakın çıkar. Hâlâ kabul edilebilir.

### 7.4 Subprocess invocation

Hook'tan çağrılır:
```bash
claude --print --dangerously-skip-permissions --model sonnet \
  /mnemos-briefing "<absolute cwd path>"
```

Output stdout, hook wrapper `<vault>/.mnemos-briefings/<cwd-slug>.md`'ye yazar.

## 8. Recall Skill v2

`skills/mnemos-recall/` SKILL.md güncellenir. **Düz Sessions search +
grep rescue.** Graph genişleme yok (v1.1 hedefi).

### 8.1 Akış

Mevcut akıştan tek değişiklik: drawer search yok, sadece Sessions search.

```
1. mnemos_search(query, limit=8, collection="raw")  # mined collection silindi
2. Top-3 score evaluate against threshold 0.015 (k=60 RRF)
3. If pass: read top 5 Sessions, synthesize narrative answer
4. If fail: Sessions grep rescue (Step 7 v0.4.1'de eklenmişti, v1.0'da default path)
   - Extract 2-4 keywords from query
   - Glob <vault>/Sessions/ + Grep keywords
   - Top 3 hit Sessions read, synthesize
5. Soft fallback: list top 3 Sessions with scores + snippets, no synthesis
```

### 8.2 Subprocess yok

Recall in-session inline çalışır (mevcut tasarım). MCP `mnemos_search` →
oku → sentezle. Subprocess `claude --print` çağrısı yok.

### 8.3 Söylenmedeki düşüş

v0.x'te recall hem mined collection (drawer) hem raw collection (Sessions)
arardı. v1.0'da sadece raw. **Bu kalite düşüşü mü?** Hayır, çünkü:
- Smoke ölçümleri (4.3.1) drawer search'in zaten zayıf çalıştığını
  gösterdi (RRF cluster 0.014-0.017)
- Sessions grep rescue mevcut çalışan path; v1.0'da default
- Sessions content-richness drawer'lardan yüksek (anlatı bütünlüğü)

## 12. Hook Geçişi

### 12.1 v1.0 hook'larının davranışı

**Auto-refine hook (`auto_refine_hook.py`):**

| Adım | v0.x | v1.0 |
|---|---|---|
| 1. SessionStart event geldi | Aynı | Aynı |
| 2. Source filter (subagent, source==compact, vd.) | Aynı | Aynı |
| 3. PID marker yaz | Aynı | Aynı |
| 4. Pick unprocessed JSONLs | Aynı | Aynı |
| 5. Subprocess spawn | `_run_skill_pipeline` (Phase A/B mining) veya `mnemos mine` | Sadece `claude --print /mnemos-refine-transcripts <jsonl>` |
| 6. Background mine | `mnemos mine` | **YOK** |
| 7. Status JSON güncelle | Mining + refine field'ları | Sadece refine field'ları |

`auto_refine.py` dosyasında `_run_skill_pipeline`, `_pick_unmined_sessions`,
Phase A/B mantığı silinir. Geriye basit refine-only akışı kalır
(~50 satır kod, mevcut 800+ satırın %5'i).

**Recall-briefing hook (`recall_briefing.py`):**

| Adım | v0.x | v1.0 |
|---|---|---|
| 1. Path checker | Aynı | Aynı |
| 2. Cwd state file | Aynı | Aynı |
| 3. CASE A first-visit | Silent exit | Silent exit |
| 4. SUB-B1 cache fresh | Inject cache | Inject cache (yeni 3-katman çıktı formatı) |
| 5. SUB-B2 unrefined JSONLs | Sync refine + sync mine + sync brief | Sync refine + sync brief (mine yok) |
| 6. Subprocess `mnemos-briefing` | 2-katman briefing | **3-katman briefing (Identity + cwd + cross-context)** |

`recall_briefing.py`'da SUB-B2 mining kısmı silinir. Subprocess invocation
aynı (`claude --print --model sonnet /mnemos-briefing <cwd>`), sadece
skill'in iç davranışı değişti.

### 12.2 `mnemos install-hook --v1` idempotent davranışı

```python
def install_hook_v1():
    settings = read_user_settings()
    hooks = settings.get("hooks", {}).get("SessionStart", [])

    # Söküm: eski v0.x entry'leri (matcher tracking via _managed_by)
    hooks = [h for h in hooks if h.get("_managed_by") != "mnemos-auto-refine"]
    hooks = [h for h in hooks if h.get("_managed_by") != "mnemos-recall-briefing"]

    # Yeni v1.0 entry'leri
    hooks.append({
        "matcher": "*",
        "_managed_by": "mnemos-auto-refine",
        "_version": "v1.0",
        "hooks": [{
            "type": "command",
            "command": "python -m mnemos.auto_refine_hook \"<vault>\"",
            "timeout": 30000
        }]
    })
    hooks.append({
        "matcher": "*",
        "_managed_by": "mnemos-recall-briefing",
        "_version": "v1.0",
        "hooks": [{
            "type": "command",
            "command": "python -m mnemos.recall_briefing \"<vault>\"",
            "timeout": 600000
        }]
    })

    write_user_settings(settings)
```

İdempotent: tekrar çalıştırılırsa eski v1.0 entry'leri sökülür ve aynı
yenisi yazılır. Net effect değişmez.
