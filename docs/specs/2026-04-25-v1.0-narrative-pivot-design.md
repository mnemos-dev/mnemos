## 1. Pivot Özeti

Mnemos'un v0.x serisi **atomic-fragmentation paradigm**'ı üzerine kuruluydu —
sohbetler refined Sessions/.md'lere dönüştürülüp ayrıca LLM ile daha küçük
"drawer" fragmanlarına ayrıştırılıyor, drawer'lar hall (decisions / problems /
preferences / events) sınıflandırmasıyla `<vault>/Mnemos/wings/<wing>/<room>/<hall>/`
hiyerarşisinde saklanıyor, vector index hem raw (Sessions) hem mined (drawer)
collection'ları üzerinde RRF ile arıyor.

Ölçümler bu paradigmayı çürüttü:

1. **Vector recall fragmanlar üzerinde converge ediyor.** RRF skor dağılımı
   k=60 ile 0.014–0.017 bandına çakılıyor; "küçük chunk → keskin embedding"
   hipotezi conversational data üzerinde gerçekleşmiyor.
2. **LLM sentezi narrative'den iyi besleniyor.** Briefing kalitesi tam Session
   notlarını verince daha keskin, fragment atomlarını verince daha bulanık.
3. **600-node graph okunaksız.** Kullanıcı manuel taramıyor, AI traversal
   yapmıyor. "Palace" görsel poster oldu, fonksiyonel araç değil.
4. **Drawer entity hijyeni zayıf.** 663 drawer toplam ~860 entity instance
   taşıyor, ~%50'si yalnızca wing adından (`Mnemos`, `ProcureTrack`, `GYP`) —
   redundant. Net değerli unique entity ~40-60.

v1.0 bu paradigmayı **clean break** ile bırakır:

- Mining pipeline tamamen silinir (kod tabanından)
- Drawer dosyaları kullanıcı vault'unda yerinde kalır (D kararı, non-destructive)
- Sessions tek memory unit olur
- Refine prompt v2 her Session'a tag + wikilink hibridi enjekte eder
- Yeni **Identity Layer** (`<vault>/_identity/L0-identity.md`) kalıcı user
  profile tutar, briefing onu base layer olarak okur
- Briefing skill 3-katmanlı (Identity + cwd Sessions + 1-hop wikilink genişleme)
- Recall skill düz (Sessions search + grep rescue, graph genişleme yok)
- MCP tool yüzeyi sadeleşir; `mnemos_graph` ve `mnemos_timeline` Obsidian
  wikilink ağı üzerine yeniden implement (signature aynı, backend değişti)

## 2. Pivot Felsefesi

**Atomic-fragmentation paradigm'dan narrative-first paradigm'a.**

MemPalace (42K-star), mem0, çoğu LangChain memory backend — hepsi
"konuşmaları küçük parçacıklara böl, vector index'le, top-K çek" hipotezini
paylaşır. Bu hipotez transactional memory için çalışır (kullanıcı tercihi
listesi, dokümanlardan fact extraction). Conversational session memory için
çürür: kararların kronolojisi paragraflar arasında taşınır, fragment'a
ayırınca neden-sonuç bağı kopar.

**Mnemos'un karşı bahsi:** büyük birim (whole Session), seyrek ama anlamlı
linkler (entity wikilink + tag), zaman üstüne biriken **Identity Layer**.
Sonuç: AI "konuşmaları hatırlayan" değil, "kullanıcıyı tanıyan" sistem
olur — Mnemos'un MemPalace/mem0'dan ayrılan asıl noktası budur.

**Identity Layer mantığı.** Her Session anlık bir snapshot; tek başına
"Tugra kim" sorusunu cevaplamaz. 78 Session'ı LLM'e bir kez okutup yapısal
özet çıkarınca (working style, technical preferences, anti-patterns,
recurring projects, people in orbit, revised decisions) bu özet kalıcı bir
markdown dosyasında yaşar. Yeni Session geldikçe **incremental update**
edilir — yeniden yazılmaz, sadece değişen alanlara delta inject edilir.
Briefing her açılışta bu profile'ı base layer olarak okur ve üstüne
cwd-spesifik anlık bağlam katmanlar.

**Clean break tercihi.** Soft deprecation döşemek yerine eski mining
pipeline kodunu silmek tercih edildi. Sebepler: (1) v1.0 majör sürüm,
breaking change beklenir; (2) `legacy/atomic-paradigm` branch'i geri
dönüş için kalıcı korunuyor; (3) yeni kullanıcı için iki paradigma yan
yana okuyabilir kafa karışıklığı; (4) anlatım tutarlılığı — pivot pivot
olur, "yarım yenileme" olmaz.

### 5.2 Bootstrap mekaniği

İlk yaratım pahalı (one-shot). Komut:

```bash
mnemos identity bootstrap
```

Akış:
1. Vault'taki tüm Sessions/.md'leri okur, date desc sıralar
2. Token bütçesi hesaplar (Session başına ~1.5K varsayımıyla)
3. **Context window cap (kritik):**
   - Eğer toplam input ≤ **150K token** ise: tüm Sessions LLM'e gönderilir
   - Eğer > 150K ise: en son 100 Session alınır + önceki Session'lardan
     örnekleme (ilk 5 + her 10'da bir) yapılır → "baseline + recent" hibrit
   - Bu cap Sonnet'in 200K context window'una emniyet payı bırakır
     (input + prompt + output toplamı ≤ 180K)
   - Vault 100 Session'dan az ise sorun yok; 100+ ise hibrit devreye girer
4. `claude --print --model sonnet` veya `--model opus` (kullanıcı seçimi) ile
   canonical bootstrap prompt'a iletir
5. LLM yapısal markdown çıkarır (yukarıdaki şablona uyar)
6. `<vault>/_identity/L0-identity.md`'ye yazar
7. `<vault>/_identity/_history/<date>-bootstrap.md` snapshot saklar

Süre: ~5-10 dakika. Maliyet: subscription quota dahilinde (claude --print).

**Chunked bootstrap (v1.1 hedefi, v1.0'da yok):**
Vault 200+ Session'a ulaşırsa hibrit cap bile yetmeyebilir. Çözüm: chunked
bootstrap — ilk 50 Session → draft profile, sonraki 50 + draft → refined,
böyle devam. v1.0'da YAGNI çünkü mevcut kasamd 78 Session, çoğu kullanıcı
100 altında olur. v1.1'de 200+ Session sahibi kullanıcı çıkarsa eklenir.

Canonical bootstrap prompt: yeni dosya `docs/prompts/identity-bootstrap.md`.
Prompt yapısı:
- Rol tanımı (project historian)
- Girdi (Sessions klasörünün içeriği veya hash listesi)
- Çıktı yapısı (yukarıdaki 7 section)
- Kalite kontrol checklist (frontmatter geçerli mi, çelişkiler tespit edildi mi, vd.)
- Boyut limiti (her section max 8 madde, "Revize edilen kararlar" max 15)
- Cap notası: "girdi 150K aşarsa baseline+recent hibrit kullan"

### 5.3 Incremental refresh

Düşük maliyetli güncelleme. Komut:

```bash
mnemos identity refresh
```

Akış:
1. `_identity/L0-identity.md` mevcut frontmatter'ından `last_refreshed`,
   `session_count_at_refresh` okunur
2. O tarihten sonra yazılan Sessions/.md'ler listesi (~5-10 yeni)
3. Pre-refresh **backup snapshot** (§5.7) yazılır
4. Mevcut profile + yeni Sessions LLM'e iletilir
5. Prompt 4 kuralla yönlendirir:
   - "Yeni bilgi mevcut bir **(general)** preference ile çelişiyorsa, bunun
     proje-spesifik mi (yeni `(proj/<name>)` satırı) yoksa **gerçek genel
     tercih değişikliği** mi (Revize edilen kararlar bölümüne taşı)
     olduğunu açıkça belirle"
   - "Yeni bilgi mevcut general'i pekiştiriyorsa dokunma"
   - "Yeni bilgi yeni bir entity (kişi, araç, proje) tanıtıyorsa eklemekten
     çekinme"
   - "Tutarlı olanlara dokunma — gereksiz revizyon yok"
6. LLM güncellenmiş profile döner
7. Atomic write (`.tmp` + `os.replace`)
8. Frontmatter güncellenir (`last_refreshed`, `session_count_at_refresh`,
   `next_refresh_at`)

Süre: ~1-2 dakika. Maliyet: ~20K token.

**Scope ayrımı kritik.** "SQLite tercih ediyorum" ifadesi tek başına
ambiguous — proje-spesifik mi genel mi? Refresh prompt LLM'i bu yargıyı
açıkça yapmaya zorlar; aksi halde identity contamination riski yüksek
(Mnemos'un sqlite-vec kullanması Tugra'nın "her projede SQLite tercih
eder" anlamına gelmez). `(general)` ile `(proj/<name>)` notation bu ayrımı
kalıcı yazılı tutar.

### 5.4 Tetikleme stratejisi

| Sürüm | Tetikleme |
|---|---|
| v1.0-alpha | **Manuel** — kullanıcı `mnemos identity refresh` çağırır. Bootstrap install sırasında bir kez fire eder. |
| v1.0 stable | **Manuel + opsiyonel hook.** SessionStart hook'u config'te `auto_identity_refresh: true` ise koşullu tetiklenir (aşağıda). |
| v1.1 | **Otomatik + çelişki UI.** Refresh sırasında çelişki tespit edilirse kullanıcıya sorar, manuel onay alır. |

**Auto-refresh trigger condition (v1.0 stable, opsiyonel hook açıkken):**

İki koşul birlikte sağlanmalı, aksi halde fire etmez:

1. **Quantity:** Son refresh'ten beri **≥10 yeni Session** yazılmış
2. **Relevance:** Bu yeni Sessions içinde en az biri **identity-relevant
   tag** içeriyor — `proj/<new>`, `person/<new>`, `tool/<new>` gibi
   son refresh'te profile'da olmayan entity. Sadece tekrarlayan tag'ler
   varsa (ör. her Session `proj/mnemos`) refresh atlanır.

Bu sayede her session'da API call yapılmaz; her Session'da aynı profile
tekrar üretilmez. "Yeni bilgi var" sinyali olduğunda fire eder.

Trigger evaluation `mnemos identity refresh --check` komutuyla manuel da
denetlenebilir (dry-run).

### 5.5 Çelişki yönetimi (3 case scenario)

Refresh sırasında LLM üç farklı çelişki tipi tespit edebilir:

**Case 1 — Project-specific addition (no conflict):**
Yeni Session'da "Mnemos için sqlite-vec kullanıyoruz" geçti. Mevcut
profile'da `(general) Postgres tercih eder` var.
→ Genel tercih kalır, yeni satır olarak `(proj/mnemos) sqlite-vec backend`
eklenir. Çelişki **yok** çünkü scope farklı.

**Case 2 — Genuine general shift:**
Son 5 Session'da Tugra "artık her projede TypeScript yazıyorum" diye
açıkça dile getirdi. Mevcut profile'da `(general) Python > diğer diller`.
→ Eski general satır `Revize edilen kararlar`'a taşınır gerekçe ile, yeni
general satır `(general) TypeScript > Python` eklenir. Çelişki **var**,
çözüm net.

**Case 3 — Project override of general:**
Mevcut `(general) Markdown-native > opaque binary`. Yeni Session'da
"ProcureTrack veritabanını Postgres'e taşıyoruz, JSON binary blob ile"
çıktı.
→ Genel tercih kalır, yeni satır `(proj/procuretrack) JSON binary OK`
eklenir. Override "bu projede istisna" anlamı taşır, general'i çürütmez.

LLM bu üç case'ten birine her çelişkiyi yerleştirmek zorunda — refresh
prompt'u bu kuralı açıkça vurgular. "Belirsiz" çıktı kabul edilmez.

v1.1'de Case 2 (genuine shift) tespit edildiğinde kullanıcı onayı zorunlu
hâle gelir; v1.0'da LLM yargısına güvenilir, ama her refresh öncesi
backup snapshot (§5.7) sayesinde geri alınabilir.

### 5.6 Dosya boyutu kontrolü

Profile şişmemeli. Section başına limitler:

| Section | Max madde |
|---|---|
| Çalışma stili | 8 |
| Teknik tercihler | 12 |
| Reddedilen yaklaşımlar | 10 (en eski + en az kullanılan ilk silinir) |
| Aktif projeler | 8 |
| Yörüngedeki insanlar | 12 |
| Ustalaşmış araçlar | 15 |
| Revize edilen kararlar | 15 (en eski silinir) |

Toplam dosya max ~3K kelime. Briefing'e ~5K token olarak girer.

### 5.7 Backup mechanism

Her refresh **öncesi** snapshot zorunlu, geri dönüş için:

**Karar mantığı:**
1. Vault `git rev-parse --git-dir` ile git-tracked mı? (kullanıcının
   kasamd'i Obsidian Sync veya OneDrive ile değil, git ile sync ediyorsa)
2. Git-tracked ise: refresh öncesi `git add _identity/L0-identity.md && git
   commit -m "mnemos identity refresh checkpoint <date>"` otomatik commit
   atılır. Geri dönüş `git revert HEAD` veya `git checkout HEAD~1
   _identity/L0-identity.md`
3. Git-tracked değilse: `_identity/L0-identity.md.bak-YYYY-MM-DD-HHmm`
   pre-refresh kopyası oluşturulur. Son 5 .bak dosyası tutulur, daha
   eskiler silinir (rolling window). Geri dönüş manuel kopyalama veya
   `mnemos identity rollback <date>` komutu.

`mnemos identity rollback [<date>]` komutu (yeni):
- Argümansız çağrı: en son .bak'ı dump eder ve `mv` onayı ister
- Tarihli: `mnemos identity rollback 2026-04-25-1430` o spesifik snapshot'a döner
- Git-tracked vault'larda `git revert` veya `git checkout` önerir, kendisi
  commit atmaz (kullanıcı git workflow'una müdahale etmez)

Bu mekanizma Identity contamination riskini "geri alınamaz" olmaktan
çıkarır — risk matrisinde (§15) etkisi düşer.

## 6. Refine Prompt v2

`docs/prompts/refine-transcripts.md` güncellenir. Tag + wikilink hibrit
("Yol 2") enjekte eder.

### 6.1 Frontmatter değişikliği

Eski:
```yaml
tags: [session-log, mnemos, v0.4, recall, skill]
```

Yeni:
```yaml
tags: [session-log, proj/mnemos, tool/supabase, tool/outlook, person/tugra, file/test_orchestrator, skill/po-olusturma]
```

Tag prefix kategori sözlüğü:
- `proj/<name>` — proje (wing eşdeğeri): `proj/mnemos`, `proj/procuretrack`, `proj/gyp`
- `tool/<name>` — araç/servis: `tool/supabase`, `tool/outlook`, `tool/chromadb`, `tool/claude-code`
- `person/<name>` — kişi: `person/tugra`, `person/safa-clutch`
- `file/<name>` — kaynak dosya (sadece tekrar eden, ≥2 Session'da geçen): `file/test_orchestrator`
- `skill/<name>` — skill/CLI komut: `skill/po-olusturma`, `skill/mnemos-catch-up`

`session-log` etiketi her Session'da kalır (tip filter için).

### 6.2 Prose içinde wikilink

Refine prompt LLM'e açıkça söyler: prose'da geçen entity'leri **ilk geçtiği
yerde** wikilink olarak sar.

**Wikilink olur:**
- Proje/wing adları (Mnemos, ProcureTrack, GYP)
- Tutarlı araç/servis isimleri (Supabase, Outlook, ChromaDB, Claude Code)
- Skill/CLI komut isimleri (po-olusturma, mnemos-catch-up)
- İnsan/şirket isimleri (Tugra, Safa Clutch, GYP Energy)
- Tekrar eden konsept hub'lar (auto-refine-hook, sqlite-vec)

**Wikilink olmaz:**
- Tek-kullanımlık dosya isimleri (test_orchestrator.py)
- Generic teknoloji (Python, JSON, Excel)
- Kod blok içindeki tüm isimler
- Versiyon numaraları (3.12, v0.4.0)
- Tarih/sayı

LLM yargısı için kural: "Bu entity başka bir Session'da da geçer mi?" Eğer
evet (tekrar eden hub) → wikilink. Eğer hayır (tek-kullanım) → düz metin.

### 6.3 Örnek

**Mevcut çıktı:**

## 7. Briefing Skill v2

`skills/mnemos-briefing/` SKILL.md ve prompt.md yenilenir. **3-katmanlı**
sentez yapar.

### 7.1 Akış

```
1. Read <vault>/_identity/L0-identity.md → Identity Layer (3K cap)
2. Read <vault>/Sessions/*.md filtered by frontmatter cwd: == input cwd
   sorted by date desc → Cwd Layer (priority 1)
3. Extract wikilinks from Cwd Layer Sessions → list of [[X]] entities
4. For each [[X]]: glob <vault>/Sessions/*.md grep "[[X]]" → backlink Sessions
   (sorted by date desc + Özet keyword overlap with cwd; dedup; exclude
   already in Cwd Layer) → Cross-context Layer (priority 2)
5. Priority-based token assembly (hard cap 15K):
   - Identity Layer: 3K (sabit, section limits §5.6 zaten bunu yaratıyor)
   - Cwd Layer: 8K (priority 1, son N Session full body — N şu ana kadar
     8K tüketilene değin doldurulur, max 5)
   - Cross-context Layer: 4K (priority 2, recency+relevance hybrid sıralı,
     budget dolana kadar Sessions çek; Özet + Alınan Kararlar sections only)
6. Synthesize: 3-section briefing (output 400-600 kelime, ~1K token)
```

**Sıralama kuralı (Cross-context için):**

Her wikilink entity'si için backlink Sessions toplanır, şu skorlama:
- Date desc (newest first) — recency ağırlığı 0.6
- Özet section keyword overlap with cwd Layer's combined Özet — relevance
  ağırlığı 0.4
- Skor = 0.6 × normalize(date) + 0.4 × overlap_count

Toplam Cross-context budget'ı (4K) dolana kadar en yüksek skorlulardan
seçilir. Çok cwd-Session'lı durumlarda (örn. 15+ cwd Session) Cwd Layer
8K'yı tamamen tüketir, Cross-context 0'a düşebilir.

### 7.2 Çıktı yapısı

```markdown
**Aktif durum:** <1-2 cümle, cwd local + identity-aware>

**Kullanıcı profili (önemli):**
- Çalışma stili / teknik tercih notları (identity'den seçili 2-3 madde,
  bu cwd'ye en alakalı olanlar)

**Geçerli kararlar:**
- <tarih> — <karar> [[wikilink]]
- (cwd-local + cross-context'ten en az 1 madde)

**Revize/iptal edilen kararlar:**
- <eski> → <yeni>. Gerekçe: <kısa>
- (cwd + identity'deki revize kararlar; eğer çelişki yoksa bölümü atla)

**Açık meseleler:**
- <çözülmemiş sorun veya TODO> [[wikilink]]

**Sırada:**
- <olası sonraki iş>

**İlgili (cross-context):**
- [[Project X]] — <bu cwd'ye benzer pattern'in geçtiği yer, kısa>
- (entity hub'larından gelen Sessions, max 3 madde)
```

### 7.3 Token bütçesi (priority-based, hard cap 15K)

Sabit total budget — katmanlar arasında priority-based paylaşım:

| Katman | Budget | Priority | Davranış |
|---|---|---|---|
| Identity Layer | 3K (sabit) | — | Always full read; §5.6 limits zaten bunu garantiler |
| Cwd Layer | 8K | 1 | Son N Session full body, 8K dolana kadar; max N=5 |
| Cross-context Layer | 4K | 2 | Recency+relevance hybrid sıralı; budget dolana kadar Özet+Alınan Kararlar |
| **Toplam input** | **15K hard cap** | — | Output ~1K (400-600 kelime) |

**Edge case — Cwd-Session-zengin ortam:**
Eğer cwd'de 10+ Session varsa, Cwd Layer 8K'yı tamamen doldurabilir.
Bu durumda Cross-context Layer budget'ı 0'a düşer, "İlgili (cross-context)"
bölümü briefing'den atılır. Bu **kabul edilebilir** çünkü cwd context
zaten yoğun, expansion marjinal değer ekler. Briefing yine yararlı
çıkar, sadece header'lar 3 yerine 2 olur.

**Edge case — Cwd-Session-fakir ortam:**
Eğer cwd'de 1-2 Session varsa, Cwd Layer 2-3K kullanır, kalan 5-6K
Cross-context'e devredilir. Yeni cwd'de Identity + cross-context bağ
kurma vurgulanır.

**Çıktı boyut beklentisi:**
3-katman briefing 400-600 kelime, ~1K token. v0.x'in 200-400 kelime
brifinginden büyük ama input 35K → 15K düştüğü için **net token tüketimi
azalmıştır**.

Hook'tan çağrılır:
```bash

  /mnemos-briefing "<absolute cwd path>"
```

Output stdout, hook wrapper `<vault>/.mnemos-briefings/<cwd-slug>.md`'ye yazar.

## 8. Recall Skill v2

`skills/mnemos-recall/` SKILL.md güncellenir. **Düz Sessions search +
grep rescue.** Graph genişleme yok (v1.1 hedefi).

### 8.1 Akış

Mevcut akıştan tek değişiklik: drawer search yok, sadece Sessions search.

```
1. mnemos_search(query, limit=8, collection="raw")  # mined collection silindi
2. Top-3 score evaluate against threshold 0.015 (k=60 RRF)
3. If pass: read top 5 Sessions, synthesize narrative answer
4. If fail: Sessions grep rescue (Step 7 v0.4.1'de eklenmişti, v1.0'da default path)
   - Extract 2-4 keywords from query
   - Glob <vault>/Sessions/ + Grep keywords
   - Top 3 hit Sessions read, synthesize
5. Soft fallback: list top 3 Sessions with scores + snippets, no synthesis
```

### 8.2 Subprocess yok

Recall in-session inline çalışır (mevcut tasarım). MCP `mnemos_search` →
oku → sentezle. Subprocess `claude --print` çağrısı yok.

### 8.3 Söylenmedeki düşüş

v0.x'te recall hem mined collection (drawer) hem raw collection (Sessions)
arardı. v1.0'da sadece raw. **Bu kalite düşüşü mü?** Hayır, çünkü:
- Smoke ölçümleri (4.3.1) drawer search'in zaten zayıf çalıştığını
  gösterdi (RRF cluster 0.014-0.017)
- Sessions grep rescue mevcut çalışan path; v1.0'da default
- Sessions content-richness drawer'lardan yüksek (anlatı bütünlüğü)

### 8.4 Threshold kalibrasyon notu

`0.015` threshold kasamd vault'unda (sqlite-vec backend, ~78 Session) k=60
RRF skorlamasıyla kalibre edildi. **Yeni kullanıcının vault'unda farklı
davranabilir** çünkü:
- ChromaDB vs sqlite-vec score distribution'u farklı olabilir
- Vault Session sayısı az ise (örn. 10 Session) RRF rank-based scoring
  daha geniş aralıkta dağılır
- Embedding model değişikliği (ileride) skorların absolute değerini değiştirir

v1.0'da threshold **hardcode** edilir (`0.015`) — calibration sürecini
kullanıcıya bırakmak YAGNI. Pratikte iki güvenlik var:
1. Threshold zaten **soft check** — düşük skor olursa Sessions grep rescue
   path'i devreye girer ve değerli cevap üretebilir
2. Soft fallback (Step 8) hiçbir path işe yaramazsa skorlu liste döner

v1.1'de gerçek bir kullanıcı vault'unda smoke "her query rescue'ya düşüyor"
şikayeti gelirse threshold `mnemos.yaml`'a `recall_threshold` field olarak
eklenir, configurable hale gelir. Şimdilik bu ihtiyaç gözlemlenmedi.

### 12.2 `mnemos install-hook --v1` idempotent davranışı

**Kritik:** Söküm ve yazım **tek atomik operasyon**. Eski entry'ler
silinip yeni entry'ler yazılırken aralarda bir an bile "iki hook yan yana
kayıtlı" durum oluşmamalı (race condition'da double-fire riski).

```python
def install_hook_v1():
    settings_path = user_settings_path()  # ~/.claude/settings.json
    settings = read_user_settings(settings_path)
    hooks = settings.get("hooks", {}).get("SessionStart", [])

    # In-memory modify — settings.json hâlâ eski hâlinde, dosya dokunulmamış
    hooks = [
        h for h in hooks
        if h.get("_managed_by") not in
            ("mnemos-auto-refine", "mnemos-recall-briefing")
    ]

    hooks.append({
        "matcher": "*",
        "_managed_by": "mnemos-auto-refine",
        "_version": "v1.0",
        "hooks": [{
            "type": "command",
            "command": "python -m mnemos.auto_refine_hook \"<vault>\"",
            "timeout": 30000
        }]
    })
    hooks.append({
        "matcher": "*",
        "_managed_by": "mnemos-recall-briefing",
        "_version": "v1.0",
        "hooks": [{
            "type": "command",
            "command": "python -m mnemos.recall_briefing \"<vault>\"",
            "timeout": 600000
        }]
    })

    settings["hooks"]["SessionStart"] = hooks

    # Atomic write: tmp + os.replace
    tmp = settings_path.with_suffix(".tmp")
    tmp.write_text(json.dumps(settings, indent=2))
    os.replace(tmp, settings_path)  # POSIX/NTFS atomic rename
```

İdempotent: tekrar çalıştırılırsa eski v1.0 entry'leri sökülür ve aynı
yenisi yazılır. Net effect değişmez. **Tek dosya yazma operasyonu** — eski
entry'lerin silindiği ile yeni entry'lerin eklendiği arası dosyada görünmez.

### 12.3 Stale hook graceful failure

**Senaryo:** Kullanıcı `pip install mnemos-dev==1.0.0a1` ile v1.0'a geçti
ama `mnemos install-hook --v1` çalıştırmadı. Eski hook entry hâlâ
`~/.claude/settings.json`'da, eski command `python -m mnemos.auto_refine_hook`
v1.0 paketini import ediyor. Ama `auto_refine.py` artık `_run_skill_pipeline`
fonksiyonunu içermiyor.

**v0.x davranışı:** ImportError → SessionStart hook crash → Claude Code
session başlangıcında "hook failed" stderr → ürkütücü deneyim.

**v1.0 graceful failure:**

`mnemos/auto_refine_hook.py` ve `mnemos/recall_briefing.py` modüllerinin
giriş noktasına shim ekle:

```python
# mnemos/auto_refine_hook.py top-level
import sys

def _detect_stale_hook_signature():
    """v1.0+ paketinde olmayan eski semantik gerektiriyorsa True döner."""
    # v0.x hook'u ne çağırıyordu? Eski entry signature: 1 argüman (vault).
    # Yeni v1.0 hook signature: aynı (1 argüman vault). Bu yüzden signature
    # bozulmuyor — fakat hook command body'sinde "_managed_by" version
    # kontrolü yok, dolayısıyla stale entry hâlâ v1.0 binary'sini çağırır.
    # Stale tespit için: settings.json'daki hook entry'sinin _version'ı
    # "v1.0" değilse (ya boş ya yok ya v0.x) bu stale demektir.
    settings = read_user_settings()
    for entry in settings.get("hooks", {}).get("SessionStart", []):
        if entry.get("_managed_by") == "mnemos-auto-refine":
            if entry.get("_version") != "v1.0":
                return True
    return False

def main():
    if _detect_stale_hook_signature():
        print(
            "Mnemos v1.0 detected an outdated SessionStart hook entry.\n"
            "Run `mnemos install-hook --v1` to update.\n"
            "Skipping this run to avoid errors.",
            file=sys.stderr,
        )
        sys.exit(0)
    # ... normal v1.0 flow
```

Aynı shim `recall_briefing.py`'da da olur.

**Kullanıcı deneyimi:** ImportError patlaması yerine net mesaj. Session
başlangıcı kırılmaz (exit 0), sadece hook çalışmadığını söyler. Kullanıcı
mesajı görür → komutu çalıştırır → bir sonraki session'da hook v1.0 davranışıyla
çalışır.

9. **Geri dönüş çalışıyor** — `git checkout main` + `pip install -e .` +

## 15. Risk Matrisi

| Risk | Etki | Olasılık | Mitigasyon |
|---|---|---|---|
| Identity contamination (refresh yanlış inject) | **Orta** (yanlış profil → yanlış briefing, ama rollback mümkün) | Orta | (1) Scope'lu preferences `(general)` vs `(proj/<name>)` ayrımı (§5.1); (2) Refresh prompt 4-kural sistemi (§5.3) — proje-spesifik mi gerçek shift mi açıkça yargılatır; (3) 3-case scenario çelişki yönetimi (§5.5); (4) Pre-refresh backup snapshot zorunlu — git commit veya .bak (§5.7); (5) `mnemos identity rollback` komutu |
| Hook geçişinde double-fire (eski + yeni entry yan yana) | Orta (statusline confused, performance hit) | Düşük | `_managed_by` filter + atomik tek-write install (§12.2); aralarda iki hook yan yana hâli olmaz |
| Stale hook (v1.0 paket yüklü, eski entry hâlâ kayıtlı) | Orta (eski entry v1.0 binary'sini çağırır → ImportError) | Yüksek (kullanıcının `install-hook --v1` çağırmasını unutması olağan) | Graceful failure shim (§12.3) — ImportError yerine "Run `mnemos install-hook --v1`" stderr mesajı + exit 0 |
| Alpha smoke gap (kasamd dışı vault'larda untested) | Orta (yeni kullanıcı şikayeti) | Orta | Smoke #1 fresh install testi zorunlu, throwaway vault validation |
| 1-hop wikilink token explosion (briefing budget aşımı) | Düşük (briefing kalitesi düşer) | Orta | Token bütçesi tablo (§7.3) tier'lı drop strategy |
| Drawer dosyaları (kullanıcı vault'unda) sistemi karıştırır | Düşük (D kararı netleştirildi, yok sayılır) | Düşük | Sessions-only filter path explicit; test #7 wikilink üretimi sadece Sessions üzerinde doğrular |
| Knowledge graph SQLite'tan veri kaybı | Yok (D kararıyla zaten dokunulmuyor) | — | Vault'ta yerinde kalır, sistem ona yazmıyor |
| `legacy/atomic-paradigm` branch'i kaybolur | Düşük (geri dönüş yolu kapanır) | Çok düşük | Pivot başında push edilir, branch protection rule (silinemez) |
| MCP `mnemos_graph` refactor backward incompat | Orta (eski client kırılır) | Orta | Signature aynı (entity, as_of), backend wikilink, dönen dict yapısı aynı |
| LongMemEval baseline regresyonu | Yüksek (recall kalitesi düşer) | Orta | Madde 6 alpha kabul + benchmark v1.0a1'da çalıştırılır, baseline ≥%85 R@5 zorunlu |
| `mnemos init` v2 onboarding bozuluyor | Yüksek (yeni kullanıcı kaybı) | Düşük | Smoke #1 fresh install testi zorunlu |

```markdown
### Breaking changes
- Atomic-fragmentation paradigm ("drawer" mining) removed. Mnemos no longer
  produces or indexes drawer fragments.

  claude-code`, `mnemos processing-log` removed.

  profile, bootstrapped from existing Sessions, evolves via incremental
  refresh.
- `mnemos identity bootstrap`, `refresh`, `show` CLI commands.
- Briefing skill now produces 3-layer narrative (Identity + cwd + cross-
  context wikilink expansion).
- Refine prompt v2 produces tag + wikilink hybrid (`tags: [proj/x,
  tool/y, person/z]` + prose `[[X]]`).

### Refactored
- `mnemos_graph(entity)` and `mnemos_timeline(entity)` now query Obsidian
  wikilink graph instead of SQLite triple store. Tool signatures unchanged;
  return dicts unchanged.

### Removed code
- Mining pipeline (8 modules ~3K LOC).
- ~200 drawer-related tests.
- `mnemos-mine-llm` and `mnemos-compare-palaces` skills.

### Migration
- Existing drawer files in `<vault>/Mnemos/wings/` are preserved untouched.
  Mnemos no longer reads from them. Users may delete manually if desired.
- Run `mnemos identity bootstrap` after upgrade to populate the new
  Identity Layer (~5–10 minutes).
- Existing `mnemos.yaml` `mine_mode` field is ignored; can be removed.
- Run `mnemos install-hook --v1` to update SessionStart hooks.

### Pre-v1.0 paradigm preservation
- The full v0.x atomic-paradigm codebase is preserved at branch
  `legacy/atomic-paradigm` and tag `v0.4.0-archived`. Users wishing to
  continue with the drawer model can pin `pip install mnemos-dev==0.3.3`.
```
