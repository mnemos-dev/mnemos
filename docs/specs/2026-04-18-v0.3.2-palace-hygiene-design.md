# v0.3.2 — Palace Hygiene (design spec)

**Date:** 2026-04-18
**Status:** Design approved, implementation pending
**Previous version:** v0.3.1 Backend UX (shipped to PyPI on 2026-04-17)
**Next version:** v0.4.0 AI Boost / Phase 1

---

## 1. Problem summary

When the palace structure in the author's real vault (`kasamd`) was inspected,
10 systemic problems were identified. 9 will be solved in this version; 1
(hall imbalance) is deferred to v0.4 LLM mining. The problems fall into three
categories:

**Structural defects (drawer not computed correctly):**

1. **Wing splitting** — `Satin` (15 drawers) and `Satin-Alma-Otomasyonu` (28
   drawers) are the same project but split across two wings.
   [`palace.canonical_wing`](../../mnemos/palace.py) only does
   case-insensitive matching; a Turkish diacritic (ı/ş/ü/ğ/ç/ö) or
   hyphen/underscore difference produces a split.
2. **Dead wing** — The `MEMORY` wing has 0 drawers but the folder and
   `_wing.md` summary exist. Leftover from an old mining run; never cleaned up.
3. **Phantom room** — `Mnemos/dev-log/` has 5 hall directories created but no
   drawers at all. [`palace.create_room`](../../mnemos/palace.py) pre-creates
   the halls; if no drawers are written afterward, an empty skeleton remains.
4. **Redundant nesting** — `Mnemos/wings/Mnemos/mnemos/` → wing=Mnemos,
   room=mnemos. The same name at two levels. In
   [`miner.mine_file`](../../mnemos/miner.py), frontmatter `tags[0]` is
   assigned directly to the room name; there is no pattern validation.
5. **"General" garbage wing** — 106 drawers, 14 rooms. Room names like
   `mnemos-—-obsidian-native-ai-memory-palace` and
   `durum-tamamlandi-(2026-03-31)` are derived from README headings. Again
   the result of the `tags[0]→room` rule.

**Quality and UX issues:**

6. **Hall imbalance** — The `preferences` hall is nearly empty (0/156 in the
   Mnemos wing); `facts` is bloated (80/156). The regex patterns do not pick
   up preference signals. → **v0.4.2 LLM mining scope; not touched in this
   version**
7. **Double-date filename** —
   format like `2026-04-14-2026-04-13-phase-0-foundation-implementa.md`:
   mining date + source date + truncated slug. Chronological browsing is
   impossible; the file list is unreadable.
8. **Entity garbage** — The drawer frontmatter `entities` list contains values
   like `mnemos, mempalace, session-log, mcp`. Room names and lowercase
   normalizations have been listed as entities; no wikilink can be produced.
9. **Body = bare chunk** — [`palace.add_drawer`](../../mnemos/palace.py)
   writes the chunk text directly into the body; no title, source link, or
   context. The drawer node looks meaningless in Obsidian graph view.
10. **Slug-only title** — The drawer's first line is truncated as
    `# YYYY-MM-DD — <slug>`; the reader cannot tell what the content is
    about. (Resolved by combining 7 + 9.)

Additional problem — **atomic rebuild bug:** `mnemos mine --rebuild` only
clears the `_mine_log` dict ([cli.py:614-616](../../mnemos/cli.py));
drawers and the index accumulate, producing duplicates. The current behavior
is inconsistent with the "atomic rebuild" claim and is also broken for
external users.

## 3. Pipeline fixes (9 problems → code)

| # | Problem | Location | Behavior change |
|---|---|---|---|
| 1 | Wing splitting | [`palace.canonical_wing`](../../mnemos/palace.py) | New `_normalize_for_match(name)` helper: NFKD + TR diacritic strip (ı→i, ş→s, ü→u, ğ→g, ç→c, ö→o) + hyphen/underscore/space strip + lowercase. Match runs against this normalization; the first-created casing is preserved during `wings_dir` iteration. "Satın Alma", "satin-alma", "Satin_Alma" all write to the same wing. |
| 2 | Dead wing | [`palace.create_room`](../../mnemos/palace.py) | Remove pre-create halls (current lines 125-126). Halls already get created lazily inside `add_drawer`. `_wing.md` and `_room.md` summary writes are lazy on first drawer added — empty skeletons impossible. |
| 3 | Phantom room | Same (single fix with 2) | Lazy hall create + lazy summary = a 0-drawer room is impossible. |
| 4 | Redundant nesting | [`miner.mine_file`](../../mnemos/miner.py) room resolution | The `tags[0]→room` rule is removed. `detect_room()` is always used. Additional invariant: `if sanitize(room).lower() == canonical_wing(wing).lower(): room = "general"` — wing==room is forbidden. |
| 5 | "General" garbage wing | Same (4 fixes it) + rooms.yaml whitelist | `detect_room` already returns one of 13 categories; once the `tags[0]` rule is gone, title-derived room names cannot be produced. **Whitelist test-only, runtime soft:** the existing [`room_detector:108`](../../mnemos/room_detector.py) already falls back to `"general"` on no-match — runtime keeps that behavior. The whitelist check is only an **invariant in tests**: `test_miner_room_hygiene.py::test_room_always_in_whitelist` — every room the miner produces on production input must be a `rooms.yaml` key or `"general"`. That way runtime doesn't break when a new room category is added; only the new key is added to `rooms.yaml`. |
| 7 | Double-date filename | [`palace._unique_filename`](../../mnemos/palace.py) | **Decision:** the filename uses **only the source date** (not the mining date). The mining date is already recorded in the drawer frontmatter as `mined_at` and stays there for audit — the filename's purpose is chronological browse, and a mining timestamp is noise there. Instead of `today = date.today()`, a three-step source-date fallback: (a) source filename prefix (`YYYY-MM-DD-...`) — most Sessions/ files are in this format, (b) frontmatter `date`/`created` field, (c) `file.stat().st_mtime` as last resort. Slug from the source title; the leading `YYYY-MM-DD[-—]\s*` regex strips the date first (without truncating dates inside the title). Word-boundary truncation (max 40 chars, last word complete). Example: `2026-04-14-2026-04-13-phase-0-foundation-implementa.md` → `2026-04-13-phase-0-foundation-implementation.md`. |
| 8 | Entity garbage | [`miner.mine_file`](../../mnemos/miner.py) entity merge | **What is removed:** the line `for tag in tags: entities.append(str(tag))` is deleted — tags aren't entities; they were just pushing room names like `session-log` into the list. `meta.get("project")` continues to be added to entities (project ≠ tag). **What is kept:** the output of `EntityDetector.detect()` (person + project lists, e.g. lowercase entities like `mempalace, mcp, chromadb`) — these are legitimate entities; casing stays as the detector returned it. **Dedup:** case-insensitive via `dict[key.lower() → original_key]` — if `Mnemos` and `mnemos` both appear, the first-seen casing (`Mnemos`) is kept. Path-derived CamelCase entities are unchanged. **Out of scope (v0.4 Phase 1 work):** acronym uppercase normalization (mcp → MCP), clarifying person/project distinction with LLM, entity stopword list. This version trusts the entity detector output and only cleans up garbage tags. |
| 9 | Body = bare chunk | [`palace.add_drawer`](../../mnemos/palace.py) | Drawer body template:<br>`# {smart_title}\n\n> From [[{source_stem}]] · {hall} · {source_date}\n\n{text}`<br>`smart_title`: the chunk's first meaningful sentence (max 80 chars, sentence-end `. ? !`); if no sentence, Title Case of the slug. Wikilink to source produces a meaningful node in Obsidian graph view. |
| 10 | Slug-only title | Resolved by 7 + 9 | Clean source-date + semantic H1 = a readable drawer. |

### Test strategy (pipeline)

Minimum two unit tests per fix (success + edge case):

- `tests/test_palace_canonical.py` — canonicalize TR characters,
  hyphen/underscore, casing preserve, first-create-wins property
- `tests/test_miner_room_hygiene.py` — `tags[0]→room` removed,
  wing==room flatten, room whitelist invariant (every room name the miner
  produces on production input must be a `rooms.yaml` key or `"general"`
  — not a runtime assertion, test-time)
- `tests/test_palace_drawer_format.py` — smart_title extraction, source-date
  resolution, filename date strip, body template
- `tests/test_miner_entity_hygiene.py` — tags are not entities,
  case-insensitive dedup, wikilink preserve

The existing suite must not break: `test_auto_refine`, `test_miner`,
`test_palace`, `test_search`, `test_install_hook` all need to pass.

## 4. Atomic rebuild (`mnemos mine --rebuild` redesign)

### Command

```
mnemos mine --rebuild [path] [--dry-run] [--yes] [--no-backup]
```

| Flag | Meaning |
|---|---|
| `path` | Optional. Auto-resolved if not provided. |
| `--dry-run` | Print the plan, make no changes. |
| `--yes` | Skip the "Proceed?" prompt. |
| `--no-backup` | Skip the wings/ backup (dangerous, prints a warning). |

### Source resolution order (when path is not provided)

**Decision:** `mnemos.yaml` is the single source of truth. Rebuild **does
not** use a hardcoded path; it supports custom vault layouts. The fallback
only kicks in when yaml has not been configured at all:

1. **If `cfg.mining_sources` is set in yaml → only those paths are used.**
   When the user has set up a custom directory structure
   (`MyVault/ClaudeSessions/`, `MyVault/DailyNotes/`, etc.), as long as it's
   declared in yaml, rebuild honors it.
2. Otherwise auto-discover: `<vault>/Sessions/`, `<vault>/Topics/` —
   whichever exist (the default layout after a fresh `mnemos init`).
3. If neither, error out: "No mining sources configured; configure
   `mining_sources` in `mnemos.yaml` or pass a path argument"

Archive/ is never included by default. Raw JSONL is never included (sources
outside the refinement pipeline are not mined directly). Claude memory is
not included by default; if the user opts in via `mnemos import memory`, it
gets added to Sessions/ and rebuild includes it.

### 9 phases

1. **Discover** — Resolve the source path list; for each path, collect the
   `.md` files and learn their counts.
2. **Pre-flight plan** — If not `--yes`, print:
   ```
   Rebuild plan:
     Sources: Sessions/ (66 files), Topics/ (15 files) = 81 files
     Current drawers: 670 (642 indexed, 28 orphan)
     Estimated time: ~15-25 min (sqlite-vec embed rate)
     Backup: Mnemos/wings → Mnemos/_recycled/wings-2026-04-18T15-30-00/
     Index backup: search.sqlite3 → search.sqlite3.bak-2026-04-18
     Graph backup: graph.json → graph.json.bak-2026-04-18
   Proceed? [y/N]
   ```
   On `--dry-run`, this plan is printed and phase 3 is not entered.
3. **Acquire lock** — `<vault>/Mnemos/.rebuild.lock` (filelock). Concurrent
   rebuilds are blocked. If the lock sees `phase=refining` or `mining` in
   `.mnemos-hook-status.json`, abort (avoid colliding with the auto-refine
   hook).
4. **Backup**
   - `shutil.move(wings_dir, recycled_dir / f"wings-{ts}")` — atomic on the
     same filesystem.
   - Index: sqlite-vec → `shutil.copy(search.sqlite3, .bak-ts)`; ChromaDB →
     snapshot of the `.chroma` directory.
   - `graph.json` → `.bak-ts`
5. **Drop + reinit index** — Backend-specific: drop + recreate sqlite-vec
   tables; drop + recreate ChromaDB collection. A new `drop_and_reinit()`
   method is added to the `SearchBackend` abstract.
6. **Clear state** — `_mine_log = {}`, `graph.reset()` (new helper, add it
   if missing).
7. **Re-mine** — Loop `handle_mine()` for each source path. Progress
   output: `Mining 23/81: 2026-04-12-mnemos.md (312 drawers)`.
8. **Verify**
   - New drawer count = 0 → **ROLLBACK**, "rebuild produced no drawers"
   - New drawers < old × 0.5 → **WARN**, "drawer count dropped
     significantly, backup preserved"
   - The backup is untouched in either case.
9. **Release lock + report** — "Rebuild complete: X drawers (was Y), Z min.
   Backup preserved at `_recycled/wings-{ts}/` — delete manually when
   satisfied."

### Rollback on failure

Phases 4-7 are inside try/except. If there's an exception or phase 8 fails:

- `shutil.move(recycled/wings-ts, wings_dir)` — restore from backup
- Restore index backup (`.bak-ts` → active path)
- `graph.json.bak-ts` → `graph.json`
- Release lock + clear error message

On success the backup is not deleted; the user deletes it manually or wipes
it manually before the next rebuild. (This is consistent with the v0.3.1
`mnemos migrate` pattern.)

### Edge cases

- **Crashed mid-rebuild** (SIGKILL, power loss) — wings/ gone, index empty,
  backup intact. The user runs again; if wings/ is empty it just mines.
  The backup is untouched.
- **Auto-refine hook fires during rebuild** — if it sees `.rebuild.lock`,
  the hook wrapper immediately `exit 0`s (like the subagent filter
  pattern). A short check is added to
  [`mnemos/auto_refine_hook.py`](../../mnemos/auto_refine_hook.py) (it was
  moved from `scripts/` into the `mnemos/` package in v0.3.0 task 3.10a).
- **Source path missing** (Sessions/ was deleted) — detected in the plan
  phase, "source X not found, skip or abort?" prompt. Under `--yes`, skip.

### File changes (atomic rebuild)

| File | Work |
|---|---|
| `mnemos/rebuild.py` (new) | `rebuild_vault(cfg, path, dry_run, yes, backup)` — orchestrates the 9 phases. |
| `mnemos/cli.py` | `cmd_mine --rebuild` → call `rebuild_vault()`; remove the inline `_mine_log = {}` line. |
| `mnemos/search.py` | `SearchBackend.drop_and_reinit()` abstract + override in both backends. |
| `mnemos/palace.py` | `backup_wings(dest_path) -> Path` atomic-move helper. |
| `mnemos/graph.py` | `reset()` method (if missing) — clears triples. |
| `mnemos/auto_refine_hook.py` | Early-exit when `.rebuild.lock` exists. |
| `tests/test_rebuild.py` (new) | ~8 tests: dry-run plan, full rebuild, no-sources error, lock prevents concurrent, rollback on drop-index fail, rollback on verify-fail, no-backup flag, auto-refine hook early-exit. |

## 5. Pilot + release protocol

### Phase A — Light pilot (3 cheap safety nets)

**Why a light pilot instead of an isolated-vault pilot:** the vault has a
total of 1061 vector rows (670 drawers + 391 raw) — the "8027 drawers"
mentioned in the initial analysis was from the Mnemos benchmark corpus,
not the author's vault. A full rebuild takes ~15-20 min, and an atomic
backup rollback takes ~30 seconds. The extra assurance from a 10-file
isolated pilot is not worth the setup time (~30 min) it costs. Instead,
3 cheap checks:

**1. Unit tests pass** — Non-negotiable. The 4 new test files listed in §3
plus the existing suite must be green before moving to Phase B. This
proves the pipeline fix's logical correctness on a regex basis.

```bash
cd C:/Projeler/mnemos
pytest tests/ -v
```

**2. Dry-run plan check** — Show the plan without triggering a rebuild;
catch unexpected source picking or path errors. Changes nothing, ~5
seconds:

```bash
python -m mnemos.cli --vault "C:/Users/tugrademirors/OneDrive/Masaüstü/kasamd" \
    mine --rebuild --dry-run
```

Expected output: Sessions/ (66 files) + Topics/ (15 files) = 81 files,
670 existing drawers, correct backup path, reasonable estimated time. If
the source list is empty, wrong directories are included, or the drawer
count is unexpected — catch it in dry-run.

**3. Single-file smoke (optional, 30 seconds)** — Without triggering a
rebuild, mine a single varied source file with the new pipeline in a
scratch folder. Verify the drawer format by eye:

```bash
mkdir -p C:/Temp/mnemos-smoke/Sessions
cp "kasamd/Sessions/2026-04-13-mnemos-phase-0-foundation-implementasyonu.md" \
   C:/Temp/mnemos-smoke/Sessions/
python -m mnemos.cli --vault C:/Temp/mnemos-smoke mine Sessions/
```

Open `C:/Temp/mnemos-smoke/Mnemos/wings/` in Obsidian and **check 3
things**:

- [ ] Filename: `2026-04-13-phase-0-foundation-...md` (single date, not double)
- [ ] First line in the drawer is `# <meaningful title>` (not the slug); second line is a `> From [[...]]` blockquote
- [ ] No room name (`session-log`) or lowercase normalization in the
      frontmatter `entities` list (`mnemos` appears once, as `Mnemos` only)

If wrong → fix → re-smoke. If smoke passes → Phase B.

**Unit test fail or smoke fail → don't move to Phase B; go back to the
pipeline fix.**

**Unit tests + dry-run + (optional) smoke green → Phase B.**

### Phase B — Real vault rebuild

**Timing constraint (managing the mixed-format risk):** from the moment
the pipeline fix is committed and activated via `pip install -e` until
the Phase B rebuild completes, the author's vault may contain a **mix of
old-format and new-format drawers** (if the auto-refine hook or a manual
`mnemos mine` runs in between). During that window, `mnemos_search`
results become stylistically inconsistent — but **search correctness is
not affected** (the frontmatter wing/hall fields are indexed the same way;
only the drawer body title/format differs). Mitigation:

1. **Deploy + rebuild in the same work session** — Phase B rebuild within
   30 minutes of the code commit and pilot approval. A mixed-state window
   on the order of minutes.
2. **Disable the auto-refine hook for the duration of Phase B**
   (`.rebuild.lock` already routes the hook wrapper to early-exit, §4).
3. No external users — only the author's own vault is exposed, and they
   know it.

```bash
cd C:/Projeler/mnemos
pip install -e ".[dev]"

# Dry-run plan
python -m mnemos.cli --vault "C:/Users/tugrademirors/OneDrive/Masaüstü/kasamd" \
    mine --rebuild --dry-run

# Real rebuild (interactive confirm)
python -m mnemos.cli --vault "C:/Users/tugrademirors/OneDrive/Masaüstü/kasamd" \
    mine --rebuild
```

**Acceptance criteria (real vault):**

- [ ] Backup sits at `_recycled/wings-2026-04-18T<ts>/`
- [ ] New drawer count is close to the old one (no >50% drop)
- [ ] All 9 hygiene problems resolved (same as pilot checklist)
- [ ] `mnemos search "kanban"` results are meaningful (no regression)
- [ ] `mnemos_wake_up` output is clean (no MEMORY, Satin merged)
- [ ] Benchmark smoke: `mnemos benchmark longmemeval --limit 10` Recall@5
  ≥ 0.85 (no serious drop from v0.3.1's 0.90; ideally the same)

If any acceptance criterion fails: roll `_recycled/wings-<ts>/` back to
`wings/`, restore the index + graph backup, revert the rebuild commit.

## 7. Risk register

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Rebuild drawer count drops by 50%+ | Low | High | Verify phase warns + backup preserved, easy rollback |
| Atomic move cross-drive failure (Windows) | Low | Medium | `_recycled/` is in the same vault = same filesystem → atomicity guaranteed |
| sqlite-vec drop+recreate corruption | Very low | High | Restore `.bak-ts` backup |
| Edge case missed by the light pilot (no isolated 10-file pilot) | Medium | Low-medium | Backup preserved (atomic rollback ~30 seconds); the post-rebuild Phase B acceptance checklist eyeballs drawers; if a problem surfaces, rollback + fix + rebuild ~25 min lost |
| Windows path encoding (TR characters) | Medium | Medium | The existing `_sanitize_name` handles TR characters; the canonicalize change is protected by a regression test |
| `canonical_wing` normalization is too aggressive → different projects merged wrongly | Low | Medium | Property test: first-create-wins + a manual whitelist override option (the `wing_override` flag already exists) |

## 9. Notes for the future

- This version doesn't yet make Mnemos noticeably fairer for external users;
  the goal is hygiene of the author's vault + future users never seeing
  these bugs (pipeline prevention).
- v0.4 AI Boost LLM mining will fill the gap in the `preferences` hall.
- v0.4 LLM rerank and contradiction detection will lift search precision —
  the clean palace this version provides will sit underneath them.
- If the external user base grows in the future and a migration tool is
  needed for old, dirty vaults, this version's `rebuild_vault()`
  infrastructure (backup + rollback + source discovery) becomes the
  foundation; a separate `mnemos cleanup` command would be easier to build.
