## 4 infrastructure commits (in order)

### Commit 1: Skill prompt multi-format update

**File:** `docs/prompts/mine-llm.md`

**Change:** Add an "Input format detection" section to the canonical prompt.
The skill currently only expects refined sessions (Özet / Alınan Kararlar / Yapılanlar
/ Sonraki / Sorunlar). Extend it:

```markdown
## INPUT FORMAT DETECTION

As soon as you Read the file, inspect the frontmatter + first few lines:

### Type A: Refined session (mnemos Sessions/)
Signals:
- Path contains `/Sessions/`
- Frontmatter has `date`, `project`, `tags`, `duration`
- Body has `## Özet` / `## Summary` section

Strategy: existing rules (Özet → skip, Alınan Kararlar → decisions drawers,
Yapılanlar → events, Sonraki Adımlar → skip, Sorunlar → problems).

### Type B: Curated topic note (mnemos Topics/)
Signals:
- Path contains `/Topics/`
- Frontmatter (if present): `type: project/topic`
- Body: free prose + headings (H1 title + H2 subsections)

Strategy: each H2 subsection is a potential drawer. Each drawer's title
comes from the H2 heading; hall inference from content (decision sentence → decisions,
problem narrative → problems, "X shipped/deployed" → events, preference
sentence → preferences). Wing = frontmatter project; if absent, from the Topic filename
or path parent.

### Type C: Claude Code memory file (~/.claude/projects/*/memory/)
Signals:
- Path contains `/memory/`
- Frontmatter has `name`, `description`, `type: user/feedback/project/reference`

Strategy: file = single drawer (already atomic). Frontmatter mapping:
- `type: user` → hall=preferences (facts about the user)
- `type: feedback` → hall=preferences
- `type: project` → hall=events
- `type: reference` → hall=events (known source)
Wing = from the project folder name in the path (e.g.
`C--Projeler-Sat-n-Alma-procuretrack` → "Satin-Alma-Otomasyonu" wing).
Room = slug from filename stem. Title = frontmatter `name`.
Preserve body as prose.

### Type D: MEMORY.md index files
Signals:
- Filename `MEMORY.md`
- Body = bullet list of `[name](file.md) — description` lines

Strategy: **SKIP** — every linked file is already processed individually as Type C.
Duplicate prevention.
```

**Test:** For each type, sample file + run skill manually, eyeball output.
At least 1 Sessions, 1 Topics, 1 memory file.

**Estimated time:** 45 min + test.

### Commit 2: Orchestrator multi-source plan

**File:** `mnemos/pilot.py`

**Change:** `build_plan` should now use the `rebuild._resolve_sources` pattern:

```python
def build_plan(vault, limit=DEFAULT_LIMIT, ...):
    cfg = load_config(vault)  # required
    # Sessions + Topics auto-discover + cfg.mining_sources union
    all_sources = _resolve_source_dirs(cfg)  # copied from rebuild.py
    all_files = []
    for src_dir in all_sources:
        for md in src_dir.glob("*.md"):
            if md.name == "MEMORY.md":  # Type D skip
                continue
            all_files.append(md)
    # mtime-desc sort
    all_files.sort(key=lambda p: p.stat().st_mtime, reverse=True)
    picked = all_files[:limit] if limit > 0 else all_files
    return PilotPlan(sources=picked, ...)  # .sessions → .sources rename?
```

**API rename consideration:** `plan.sessions` → `plan.sources` (breaking change
for this module but no external users). Tests should be updated.

**New tests:**
- `test_build_plan_picks_from_all_source_dirs` — in a tmp vault, create Sessions/ + Topics/
  + a configured mining_source directory, verify files are discovered from all of them
- `test_build_plan_skips_MEMORY_md` — in memory directory, MEMORY.md + regular.md →
  regular.md should be picked up, not MEMORY.md

### Commit 4: Parallel-3 execution

**File:** `mnemos/pilot.py` (`run_pilot` rewrite)

**Design:**
- `concurrent.futures.ThreadPoolExecutor(max_workers=3)`
- Each future = `_run_one_session` for one source file (unchanged)
- Ledger append serialization: `.mine-llm.lock.flock` (advisory, filelock).
  Since the skill writes its own ledger, a process-level lock is not in the pilot
  but between skill invocations — because the skill is a fresh claude --print, the
  process is separate. For the orchestrator: skill subprocesses can already run in
  parallel, the lock is only for `sessions_needing_run` → `_run_one_session`
  ordering + ledger read consistency.
- Progress with `as_completed`: "refining N/M · backlog X"
- `--no-parallel` flag for sequential fallback

**Subprocess env clean pattern (from auto_refine.py):**
`ANTHROPIC_API_KEY` stripped → subscription auth. Already in `default_capture_runner`.
Apply to thread pool runner as well.

**Tests:**
- `test_run_pilot_parallel_spawns_three_at_most` (fake runner + mocked
  threading, verify semaphore behavior)
- `test_run_pilot_parallel_preserves_order_in_outcomes` (futures complete
  out-of-order, final outcomes list in plan order)
- `test_run_pilot_no_parallel_flag_sequential` (fallback path)

**Estimated time:** 1.5 hours.
