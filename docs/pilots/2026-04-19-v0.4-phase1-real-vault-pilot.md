## Operational findings → v0.4.1 backlog

### Finding 1 — Latency 10x spec estimate

Spec §4.2.2 "**~25s per session**" yazmıştı. Gerçek ~264s/session.
Skill'in (a) canonical prompt + session'ı okuması, (b) bir-veya-birkaç
on drawer dosyası yazması, (c) Claude Code reasoning katmanının her
toolcall arası thinking → cumulative 4 dk. 100-session pilot:
sequential ~7h, paralel-3 ile ~2h. Kullanıcıya doğru beklenti vermek
için:

- Spec estimate güncellensin (`25s` → `3-5 dk per session`)
- Orchestrator `Estimated time` satırı aynı şekilde update (CLI'da
  `plan.session_count * 25` → `* 240`)
- Parallel-3 (spec §4.2.2 orijinal tasarımda vardı ama MVP'de sequential
  shipped) v0.4.1'de eklenmeli — sequential bazı session'ı 5+ dk tutarken
  hepsi bekleyecek

### Finding 2 — Ledger reliability bug

Skill 3 session'da da drawer yazdı (filesystem'de 37 dosya var), ama
`state/mined.tsv` ledger'ına sadece **1 OK satırı** düştü. Diğer 2
session'da skill ya ledger append adımına gelmeden bitti ya da sessiz
skipetti.

Sonuç: orchestrator "Per-session outcomes" tablosu `ERROR` gösterdi
(drawer_count=0) — halbuki gerçek drawer sayıları 13 ve 9. **Yalan
rapor.**

Çözüm (v0.4.1):

- **SKILL.md hardening:** Ledger append'i skill'in her drawer write
  döngüsünde refleks kıl ("her Write'dan sonra ledger update ZORUNLU"
  açık kural; pasif "append it at the end" yerine aktif refleks).
  `state/mined.tsv` örnek satırlarını SKILL.md'ye eklentiyle zenginleştir.
- **Orchestrator fallback:** ledger'da satır yoksa `<palace>/wings/**/*.md`
  altında `source: <session-path>` içeren drawer'ları say. Ledger boş =
  error etiketi YERİNE filesystem truth'u kaynak say. Tek-session'da
  hem ledger hem filesystem boş = gerçek error.

### Finding 3 — Report apples-to-oranges

`format_pilot_report`'un "Quantitative summary > Total drawers" sütunu:

- Script: `_count_drawers(Mnemos/)` → **tüm palace** (kasamd: 718)
- Skill: `_count_drawers(Mnemos-pilot/)` → **pilot-only** (37)

Bu rakamlar yan yana olmamalı. Çözüm: `_count_drawers` bir `sessions:
list[Path]` argümanı alsın, drawer'ların frontmatter `source:` field'ını
okuyup sadece pilot session'larına ait drawer'ları saysın. Aynı metod
hall dağılımı için de kullanılsın.

### Finding 4 — Skill-mined palace indexing eksik (known)

`mnemos pilot --accept skill` dosya taşıma yapıyor + yaml update +
WARNING bastırıyor ama **index stale kalıyor**. Skill-mined drawer'lar
ChromaDB/sqlite-vec'e yazılmamış. Arama bu drawer'ları yakalamaz.

Çözüm (v0.4.1): `mnemos/palace_indexer.py` + `mnemos mine --from-palace
<path>` CLI. Frontmatter-authoritative mode — classification'ı tekrar
koşturmaz, sadece embedding + metadata index.

## Recommendation

Kasamd'ın mevcut `Mnemos/` script-mine state'i **v0.3.2 hygiene fix'lerinin
tam devreye girmediği** bir an — çift-tarih, section-başlığı filename
kaçışı hâlâ var. Adil karşılaştırma için script-mine'ı `mnemos mine
--rebuild` ile tazele, sonra pilot tekrar et. Ama drawer kalitesi
konusundaki gap muhtemelen gap olmaya devam edecek — skill-mine'ın
H1-smart-title, 100-200 kelimelik bodytemiz frontmatter çıktısı regex'ten
çıkan her şey için upper bound oluşturuyor.

**Karar kullanıcıda.** v0.4.0'ı ship'lemeden önce v0.4.1 fix'lerini
(finding 1-4) batch halinde kapatmak pilot experience'i düzeltir;
yoksa external user ilk pilot'ta "ERROR=2" + "1.63M tokens" görür ve
panikler.
