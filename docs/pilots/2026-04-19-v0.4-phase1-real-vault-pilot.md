## Operational findings → v0.4.1 backlog

### Finding 1 — Latency 10x spec estimate

Spec §4.2.2 said "**~25s per session**". Reality is ~264s/session.
The skill (a) reads the canonical prompt + session, (b) writes
one-or-several drawer files, and (c) the Claude Code reasoning layer
thinks between each toolcall → cumulative 4 min. 100-session pilot:
sequential ~7h, ~2h with parallel-3. To set proper user expectations:

- Update the spec estimate (`25s` → `3-5 min per session`)
- Update the orchestrator `Estimated time` line in the same way (in
  the CLI, `plan.session_count * 25` → `* 240`)
- Parallel-3 (which existed in the original spec §4.2.2 design but
  shipped sequentially in MVP) should be added in v0.4.1 — when
  sequential mode keeps a session running 5+ min, everything else
  has to wait.

### Finding 2 — Ledger reliability bug

The skill wrote drawers in all 3 sessions (37 files exist in the
filesystem), but only **1 OK line** landed in the `state/mined.tsv`
ledger. In the other 2 sessions, the skill either finished before
reaching the ledger append step or silently skipped it.

Result: the orchestrator's "Per-session outcomes" table showed `ERROR`
(drawer_count=0) — even though the real drawer counts were 13 and 9.
**A false report.**

Fix (v0.4.1):

- **SKILL.md hardening:** Make the ledger append a reflex in every
  drawer-write loop of the skill ("a ledger update after every Write
  is REQUIRED" as an explicit rule; an active reflex instead of the
  passive "append it at the end"). Enrich SKILL.md with sample lines
  from `state/mined.tsv`.
- **Orchestrator fallback:** when there are no lines in the ledger,
  count the drawers under `<palace>/wings/**/*.md` whose frontmatter
  contains `source: <session-path>`. INSTEAD OF an empty-ledger error
  label, treat filesystem truth as the source. In single-session mode,
  both an empty ledger and an empty filesystem = a real error.

### Finding 3 — Report apples-to-oranges

The "Quantitative summary > Total drawers" column of
`format_pilot_report`:

- Script: `_count_drawers(Mnemos/)` → **the entire palace** (kasamd: 718)
- Skill: `_count_drawers(Mnemos-pilot/)` → **pilot-only** (37)

These numbers should not sit side by side. Fix: have `_count_drawers`
take a `sessions: list[Path]` argument, read the drawers' frontmatter
`source:` field, and count only the drawers belonging to the pilot
sessions. The same method should be used for the hall distribution as
well.

### Finding 4 — Skill-mined palace indexing missing (known)

`mnemos pilot --accept skill` does file moves + yaml update + WARNING
suppression, but **the index stays stale**. Skill-mined drawers were
not written to ChromaDB/sqlite-vec. Search does not catch these drawers.

Fix (v0.4.1): `mnemos/palace_indexer.py` + `mnemos mine --from-palace
<path>` CLI. Frontmatter-authoritative mode — does not rerun
classification, only embedding + metadata index.

## Recommendation

Kasamd's current `Mnemos/` script-mine state is a moment **where the
v0.3.2 hygiene fixes haven't fully kicked in** — double-date and
section-heading filename leakage are still present. For a fair
comparison, refresh the script-mine with `mnemos mine --rebuild`, then
rerun the pilot. But the gap in drawer quality is likely to remain a
gap — skill-mine's H1-smart-title, 100-200 word body, and clean
frontmatter output sets an upper bound on anything that comes out of
regex.

**The decision is up to the user.** Closing the v0.4.1 fixes (findings
1-4) in a batch before shipping v0.4.0 will improve the pilot
experience; otherwise an external user will see "ERROR=2" + "1.63M
tokens" on their first pilot and panic.
